# ⚡ 深度研究 ｜ DiffusionGemma：谷歌把「扩散语言模型」推进开放生态——4 倍速度背后的架构豪赌

> **一句话定性**：这是头部实验室**第一次以开放权重 + 宽松许可（Apache 2.0）+ 完整工具链**的形式发布扩散语言模型——自 GPT 以来统治文本生成的「自回归逐 token」范式，第一次在开放生态里迎来一个有大厂背书、能实际部署的替代路线。它今天的质量还打不过自家 Gemma 4，但它瞄准的是**推理经济学**这个行业总成本的分母。

**来源**：Google《DiffusionGemma: 4x faster text generation》（blog.google 官方博客），2026 年 6 月 10 日发布；权重已上 Hugging Face（google/diffusiongemma-26B-A4B-it）。
🔗 原文：https://blog.google/innovation-and-ai/technology/developers-tools/diffusion-gemma-faster-text-generation/

## 一、为什么这一篇值得单独深读

过去三年，所有主流 LLM——GPT、Claude、Gemini、Llama、Qwen——都是**自回归（autoregressive）**模型：一次生成一个 token，下一个 token 必须等上一个。这个范式决定了今天整个行业的延迟曲线、serving 架构和推理成本结构。

「文本扩散」是少数有实质进展的替代路线：像图像扩散模型一样，从一片噪声出发、**并行地**把整块文本迭代「去噪」出来。此前它一直停在边缘——2025 年 2 月 Inception Labs 的 Mercury 是首个商用扩散 LLM，2025 年 5 月谷歌 I/O 上的 Gemini Diffusion 只是个排队体验的研究演示，学术界有 LLaDA、Dream 等小模型。**这次不同**：谷歌把一个 26B、多模态、256K 上下文的扩散模型，以 Apache 2.0 开放权重放出，并且日 1 配齐 Hugging Face Transformers / vLLM / MLX 支持（llama.cpp 在路上），NVIDIA、Unsloth 当周就跟进了量化版。这是扩散路线第一次拿到「进入主流开发者工具链」的门票。

## 二、核心事实（硬数据分层）

**A. 模型规格**
- **26B MoE 架构**：总参数 25.2B，推理时仅激活 **3.8B**（128 个专家激活 8 个）；基于 Gemma 4 26B A4B 改造，编码器-解码器设计、**双向注意力**。
- **多模态输入**：文本 + 图像 + 视频输入，文本输出；上下文窗口最高 **256K**；支持 140+ 语言、原生 function calling；训练数据截止 2025 年 1 月。
- **Apache 2.0** 开放权重；量化后 **18GB 显存**即可在高端消费级 GPU 上运行。

**B. 速度（卖点）**
- 官方口径：GPU 上**最高 4 倍**于标准自回归模型的生成速度。
- 实测：H100（FP8、batch size 1）约 **1,008 tokens/s**（模型卡称低并发可超 1,100）；H200 上 **1,288 tokens/s**——据 vLLM 基准约为自回归基线的 **6 倍**；消费级 RTX 5090 上 **700+ tokens/s**。
- 机制指标：以 **256-token 的「画布」块**为单位生成，每次前向传播可敲定 **15–20 个 token**（自回归模型是 1 个）。

**C. 质量（短板，谷歌自己摆上台面）**
- 对比同尺寸 Gemma 4：**MMLU Pro 77.6 vs 82.6**；**AIME 2026 数学竞赛 69.1 vs 88.3**（差近 20 分）；视觉理解 **MMMU Pro 54.3 vs 73.8**。
- 官方博客原话直白：**「对质量要求最高的应用，我们建议部署标准 Gemma 4。」**——定位是 experimental，不是旗舰替代。

## 三、机制拆解：它为什么快，又为什么（暂时）笨

**自回归的瓶颈在内存带宽**：逐 token 生成时，每产出 1 个 token 都要把全部激活权重从显存搬一遍，GPU 算力大量闲置——低并发场景下 GPU 实际是「等数据」而非「算数据」。

**扩散的打法是「把活儿凑成一大块」**：从 256 个随机占位 token 的画布开始，模型做多轮去噪——每一轮**并行地**评估整块文本，把「有把握」的 token 锁定，不确定的留给下一轮迭代修正。官方的说法是「一次给处理器一大块活儿，把硬件潜力榨满」。额外的结构性优势：**双向注意力**让每个 token 能同时看到前文和后文（自回归只能看前文），并且生成过程**天然带自我修正**——前几轮写错的 token 可以在后几轮被改掉，而自回归一旦吐出就覆水难收。

**代价同样来自机制**：多步去噪的误差会跨轮累积；按固定 256-token 块生成对「长链条推理」不友好——这正是 AIME 差 19 分的合理解释（数学推理高度依赖严格的先后依赖，而并行去噪在这上面先天吃亏）。另外它与现有自回归 serving 栈的核心优化（KV cache、连续批处理）不完全兼容，生态要重新长。

## 四、关键引述

- 谷歌官方博客：「**By giving the computer's processor a larger chunk of work at once, DiffusionGemma utilizes your hardware to its full potential.**」（一次给处理器一大块活儿，把硬件潜力榨满）
- 同一篇博客的诚实警告：「**For applications that demand maximum quality, we recommend deploying standard Gemma 4.**」

## 五、正反观点

**支持方（架构信号论）**
- 这是自回归范式垄断以来，第一次有大厂把替代路线以「可部署、可商用、可微调」的形态交到社区手里——**架构多样化本身就是战略事件**，类比图像领域 GAN→Diffusion 的范式更替也曾从「质量更差但更快/更稳」起步。
- 在**低并发、单用户、本地推理**场景（恰是开源模型主战场）它是真赢家：消费级显卡 700+ tokens/s、18GB 显存、3.8B 激活参数，把「本地秒回」从口号变成可演示的现实。
- 双向注意力 + 自我修正在**结构化输出、代码 infilling、文档填空**类任务上有自回归给不了的天然优势。

**质疑方（工程现实论）**
- 质量差距不是小数点：AIME 差 19 分、MMMU Pro 差 19.5 分，说明**推理与多模态理解这两个当下最值钱的能力恰恰是它最弱的**。「快但答错」在多数生产场景一文不值。
- 速度优势依赖「GPU 有闲置算力」这个前提：**高并发云端 serving 里自回归模型早已通过大批处理把算力吃满**，扩散的并行红利会急剧递减——也就是说它的甜区在边缘和本地，撼动不了数据中心的主流打法。
- Mercury 商用已一年多、Gemini Diffusion 演示了一年，谷歌至今没把扩散用进 Gemini 主线旗舰——**连发明者自己都还没敢押注**，「experimental」三个字就是态度。

## 六、深度解读：对行业 + 对做多 agent 系统的我们

**对行业格局：**
1. **这是一次针对「推理经济学」的卡位**。当前行业叙事正从「谁的模型最聪明」转向「谁的 token 最便宜最快」（DeepSeek 5 月永久降价 75%、xAI 拿 100 tokens/s 当卖点营销）。扩散路线如果在两三代内把质量差距收窄到可接受，**单位算力的 token 产出直接翻几倍**——这是改写毛利率的分母变量，而非又一个 benchmark 故事。
2. **开放模型竞争多了一个维度**。Gemma 对 Qwen/Llama 的竞争此前是「同架构比分数」，现在谷歌单方面开了一条新赛道——社区微调、蒸馏、混合架构（扩散起草 + 自回归精修的 speculative decoding 变体）都会围绕它长出来。值得注意 NVIDIA 第一时间发布了 NVFP4 量化版：硬件厂商对「能把 GPU 算力吃满」的架构有天然好感。
3. **观察谷歌的「演示→开放→主线」节奏**。Gemini Diffusion（2025/5 演示）→ DiffusionGemma（2026/6 开放），下一步若扩散进入 Gemini 主线某个 Flash/Lite 档位，证明该路线真正闭环；若长期停在 Gemma 实验线，则说明质量收敛不及预期。

**对做多 agent 系统的我们（可执行判断）：**
1. **Agent 循环的延迟预算可能被重写**。多 agent 系统的体感速度 = 单步生成延迟 × 串行步数，串行步数我们很难砍，单步延迟是最大杠杆。一个 700–1,000 tokens/s 的本地模型，意味着「计划拆解、工具参数填充、格式转换、摘要压缩」这类**对智商要求低、对延迟敏感**的 agent 子任务有了新选项——值得在我们的编排层里实测一档「扩散快模型路由」。
2. **它在 agent 工程的甜区恰好避开了它的短板**：function calling 参数填充、JSON/结构化输出、模板填空是「填空型」而非「长链推理型」任务，双向注意力 + 自我修正反而占优；把 AIME 型的难推理继续留给旗舰模型即可——**这就是能力路由该做的事**。
3. **本地化 + 隐私敏感场景的成本曲线变了**：18GB 量化 + Apache 2.0 意味着一台 Mac Studio / 单卡工作站就能跑一个多模态 256K 上下文的「快手」，对需要离线/数据不出域的客户场景是新筹码。
4. **节奏判断**：现在不必迁移任何生产负载（质量差距摆在那），但应该**把「扩散模型适配」当作编排层的一个抽象来预留**——如果 12 个月内质量收敛（参照图像扩散的迭代速度，不是没可能），先适配者吃到的是数倍的延迟/成本红利。

## 七、接下来盯什么（信号）

- **llama.cpp 支持落地速度**与社区微调生态是否起量（决定它是玩具还是基础设施）。
- **vLLM 高并发下的真实吞吐对比**——第三方独立基准会很快出现，验证「甜区只在低并发」这个判断是否成立。
- **谷歌是否把扩散引入 Gemini 主线**（尤其 Flash/Live 这类延迟敏感档位），以及 **Inception Labs（Mercury）的应对**。
- 下一代 DiffusionGemma 的**质量收敛斜率**：MMLU Pro / AIME 差距若在下个版本收窄过半，行业会立刻重新定价这条路线。
- OpenAI / Anthropic 是否跟进任何非自回归路线的公开信号。
