# Anthropic 把「无人值守 agent」做成平台：Managed Agents 补上定时调度与凭证保管库

**一句话定性**：6 月 9 日，Anthropic 给 Claude Managed Agents 补上了生产化的最后两块拼图——cron 定时调度和保管库注入凭证——这标志着模型厂商正式向上吞掉「agent 编排 / 托管基建」这一层；叠加 6 月 15 日生效的 Agent SDK 计费拆分，Anthropic 正在同时重写 agent 的**基础设施层**和**经济层**。

- **来源**：Anthropic 官方博客《New in Claude Managed Agents: run agents on a schedule and store environment variables in vaults》
- 🔗 **原始链接**：https://claude.com/blog/whats-new-in-claude-managed-agents
- **日期**：2026-06-09（官方 Claude Platform release notes 同日收录）

## 为什么值得深读 / 背景

企业 agent 项目大量卡死在 pilot 阶段，卡点往往不是模型能力，而是两个「不性感」的工程问题：**没人在场时谁来触发 agent？agent 怎么登录第三方服务而不泄露凭证？** 这次更新正是冲着这两个问题去的。

更重要的是时间线。把官方 release notes 拼起来看，Anthropic 过去两个月在以惊人节奏把一整套 agent 基建产品化：

- **4/8**：Managed Agents 公测（统一 beta header `managed-agents-2026-04-01`）+ `ant` CLI 发布
- **4/23**：Agent 记忆（memory）公测
- **5/6**：多 agent 会话（multi-agent sessions）、Outcomes、webhooks、保管库 OAuth 凭证自动刷新
- **5/11**：Claude Platform on AWS 上线（AWS 计费 + IAM 鉴权）
- **5/19**：自托管沙箱公测 + MCP 隧道 research preview；超 100K token 的工具输出自动落盘（spill-to-file）
- **5/29**：webhooks / 多 agent / 自托管沙箱在 AWS 平台对齐
- **6/9**：定时调度（scheduled deployments）+ 保管库环境变量凭证 ← 本次

也就是说：调度器、凭证管理、沙箱、记忆、编排、webhook——一个 agent 基建团队通常要自建的所有东西，Anthropic 在 8 周内全部做成了平台功能。

## 核心事实

**1）定时调度（scheduled deployments，公测）**
- 给 agent 配一个 cron 表达式；每次触发自动开一个全新 session 完成任务，无需自建/托管调度器。
- 支持暂停、恢复、归档、手动按需触发。
- 典型场景：夜间数据同步、每周合规扫描、每日摘要。
- **不另收费**：按现有 Claude Platform 用量计费，无平台附加费。

**2）保管库环境变量凭证（vaults，公测）**
- API key 存在保管库中，**在网络边界注入**：agent（及其沙箱内的模型上下文）从头到尾见不到明文密钥。
- 密钥只会发送到**白名单（allowlist）域名**——即使 agent 被提示注入劫持，也无法把密钥外传到任意地址。
- 已支持 Browserbase、KERNEL、Notion、Ramp、Sentry 等厂商的 CLI；Browserbase/KERNEL 给 agent 加上浏览器操作能力。
- 凭证变更在运行中 session 的下一次 API 调用即生效。

**3）平台架构（来自官方 docs）**
- 四个核心概念：**Agent**（模型+系统提示+工具+MCP+技能）/ **Environment**(运行环境)/ **Session**（运行实例）/ **Events**（事件流，SSE 推送、服务端持久化）。
- 自托管沙箱：工具执行可放到 **Cloudflare / Daytona / Modal / Vercel 或自有基建**，企业可控 CPU、内存、运行镜像；但**agent loop（编排、上下文管理、错误恢复）始终留在 Anthropic 服务器上**。
- MCP 隧道（research preview）：网关从内网发起单条出站加密连接，agent 即可访问私网 MCP server（内部数据库、私有 API、工单系统），无需开任何防火墙入站规则。
- 合规红线：Managed Agents 有状态、服务端存储会话与沙箱状态，**目前不适用 Zero Data Retention，也不在 HIPAA BAA 覆盖范围内**。

**4）经济层：6/15 计费拆分（5/14 官宣，4 天后生效）**
- Agent SDK、`claude -p`（无头模式）、Claude Code GitHub Actions、所有经 Agent SDK 鉴权的第三方应用（OpenClaw、Conductor、Zed、Jean 等）**迁出订阅用量池**，改用独立月度信用池：Pro $20 / Max 5x $100 / Max 20x $200（Team Standard $20、Team Premium $100、Enterprise Standard 为 $0）。
- 信用**按 API 牌价计费、当月清零不滚存、溢出计费默认关闭**——用完即停。
- 此前订阅对 agent 用量的补贴率估算：典型 OpenClaw 用户约 **12:1**（$20 订阅抽取约 $236 API 等值），Max 20x 重度用户最高约 **175:1**。
- 交互式使用（Claude.ai、终端里的 Claude Code、Cowork）不受影响。

**5）竞争对照**
- OpenAI 的 workspace agents 将于 **7 月 6 日**转向 token 信用计价（GPT-5.5 典型单次运行约 5–25 credits）。两巨头几乎同步把「无人值守 agent」变成计量收费的云服务。

## 关键引述

- Rakuten AI for Business 总经理 Yusuke Kaji：各团队用定时调度「分析表格数据并按周/月节奏自动产出报告」。
- Notion 公共 API 负责人 Quan Nguyen：保管库让 Notion CLI 的推广「不向模型暴露 API token」，满足安全团队的严格要求。
- Actively AI 联创 Mihir Garimella：「我们用 scheduled deployments **替换了自建的调度基础设施**」——这句话就是本次更新的战略含义本身。
- Anthropic 的 Boris Cherny 解释计费拆分动机：订阅「weren't built for the usage patterns of these third-party tools」。
- The Decoder 的批评：「A fully on-premise deployment of the agents isn't possible. Companies that want to control model execution themselves won't find a solution here.」

## 正反观点

**支持方**：
- 调度 + 凭证是企业 agent 从 pilot 到生产的真实卡点，平台化后「几天而非几周/几月」即可部署（VentureBeat 转述 Anthropic 说法）；Rakuten、Notion、Actively AI 等已在生产使用。
- **网络边界注入 + 域名白名单**是目前对「agent 被提示注入后窃取凭证」这一头号安全威胁最干净的工程答案——比把密钥塞进环境变量或上下文里好一代。

**质疑方 / 风险**：
- **厂商锁定**：session 格式、事件协议、容器规格全是 Anthropic 专有，VentureBeat 直言这是「one-stop shop 换 lock-in 风险」；HN 上有人说「锁进一个框架对想保持竞争力的人是输家命题」，反方则认为「从零到一的团队拿易用性换锁定是划算的」。
- **不能真正私有化**：自托管沙箱只下放了工具执行，agent loop 永远在 Anthropic 手里；加上不支持 ZDR / HIPAA BAA，强合规行业暂时进不来。
- **经济组合拳的另一面**：先用平台把你自建基建的理由抽走，再用计费拆分把 agent 用量按 API 全价计——补贴时代（12:1～175:1）结束，无人值守 agent 第一次有了真实价格标签。无人值守运行的产出质量仍需下游人工审核环节兜底。

## 深度解读：对行业 + 对我们的具体含义

**对行业**：
1. **模型厂商正在向上吞掉 agent infra 层**。LangChain/Temporal 式的编排、自建调度器、凭证管理中间件——这一层的独立价值被快速商品化。Anthropic（6/15）与 OpenAI（7/6）同步把 agent 运行变成计量云服务，意味着「agent ops」从工程问题升格为成本/毛利问题。
2. **安全模式定调**：凭证不进上下文、在网络边界注入、出站隧道替代防火墙开洞——这套设计大概率成为行业默认范式，自建系统会被拿来对标。
3. **合规是下一个战场**：谁先让托管 agent 拿到 ZDR/HIPAA/数据驻留资格，谁就打开金融医疗政府市场。

**对我们（做多 agent 系统的团队）——可执行判断**：
1. **6/15 前必须盘点（最紧急，4 天）**：所有 cron/无头跑的自动化任务（包括本 research-monitor 这类 `claude -p` / Agent SDK 驱动的流程、OpenClaw 类工具）——确认各自的鉴权路径，估算月 API 等值消耗，对照所在套餐的信用池（$20/$100/$200）决定：留在订阅信用池内、开启溢出计费、还是迁移到 API key。**溢出默认关闭意味着信用耗尽当天自动化静默停摆**——要加用量监控告警。
2. **自建基建价值上移**：调度器、凭证库、沙箱不再是护城河。自建投入应收敛到模型厂商做不了的层：领域工作流、私有评测、数据资产、跨厂商可移植的编排抽象。
3. **抄安全设计**：我们自己的 agent 凭证管理应对齐「明文不进 agent 上下文 + 注入点在网络边界 + 目的域白名单」三原则。
4. **用而不锁**：Managed Agents 可以当「执行层」用（沙箱、调度、凭证白拿），但编排逻辑和 session 状态语义留在自己侧，保持随时可换到自托管/他家。
5. **成本进入产品视野**：per-run cost 要进 dashboard——agent 运行成本即将像云账单一样成为产品毛利的一等公民。

## 接下来盯什么

- **6/15**：计费拆分生效——实际账单冲击、OpenClaw/Conductor 生态的迁移与反弹。
- **7/6**：OpenAI workspace agents 信用计价生效，两家定价体系正面可比。
