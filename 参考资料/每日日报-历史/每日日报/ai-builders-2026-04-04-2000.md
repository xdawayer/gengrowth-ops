**Andrej Karpathy**（前 Tesla AI 总监，OpenAI 创始团队，斯坦福 PhD）

Karpathy 分享了他最近使用 LLM 构建个人知识库的完整工作流，思路非常清晰：

他将源文档（文章、论文、代码库、数据集、图片）收入 `raw/` 目录，然后用 LLM 将其"编译"成一个 Markdown Wiki，包含摘要、反向链接、概念分类和互相链接的文章。使用 Obsidian 作为前端查看原始数据和 Wiki，LLM 负责写入和维护 Wiki 的所有数据，他自己几乎不直接编辑。为了实现问答，他在 Wiki 足够大（最近的研究项目约有 100 篇文章、40 万词）时，直接让 LLM Agent 对 Wiki 进行复杂查询，效果出乎意料地好——原本以为需要 RAG 才能搞定，实际上 LLM 自动维护索引文件就能搞定。他还将输出渲染成 Markdown、Marp 幻灯片或 matplotlib 图片，进一步丰富 Wiki，形成一个自我积累的知识库。

他还展望了后续方向：当 Wiki 规模继续扩大，自然会想到用合成数据 + 微调把知识"蒸馏"进模型权重，而不只是依赖上下文窗口。

https://x.com/karpathy/status/2039805659525644595

Karpathy 提出了一个更大胆的愿景：未来向前沿 LLM 提问，可以想象成召唤一组 LLM 子团队，自动完成整个研究流程——迭代构建临时 Wiki、lint、循环几轮，最终输出一份完整报告。远超简单的 `.decode()`。

https://x.com/karpathy/status/2039808711452246261

**Ryo Lu**（Cursor 设计负责人，前 Notion / Stripe / Linear）

Ryo Lu 发布了 Cursor 3，同日发表了一篇产品哲学长文阐述"玻璃 vs 黑箱"的核心理念：

他观察到，AI 继承了 Terminal 的黑箱特性并让它变得更上瘾——你输入一个愿望，扳动杠杆，出来一个东西，接受或拒绝，你看到的东西越来越少，思考也越来越少，因为模型越来越强，你反而变成了模型的产物。

Cursor 选择做"玻璃"：Terminal 是黑箱，而 AI 保持黑箱只会让人更依赖。玻璃打破了这个盒子，让用户真正参与进来——PM 写出计划并看着它变成现实，设计师画出草图并在代码中实时迭代，工程师在写第一行代码之前就分解系统架构，初级程序员阅读每一条 diff、提问、学习、建立直觉。

核心观点：AI 越强大，玻璃越重要。不是因为需要盯着每一步，而是因为最好的工作来自你知道你可以随时介入——你补充只有你会想到的那一笔，你发现模型遗漏的那个点，你把它塑造成你真正思考和工作方式的样子。给你的工具学习你的方式，而不是反过来。

https://x.com/ryolu_/status/2039895634313187619

Cursor 3 正式发布：跨本地和云端所有项目运行，界面大幅简化，删除了原来过多的按钮和开关。power meets simplicity。

https://x.com/ryolu_/status/2039780768847958359

**Aaron Levie**（Box CEO）

Levie 分享了构建 AI Agent 过程中最深刻的一条教训——你必须对自己的架构"毫不留情"：

核心循环是：先用一堆系统辅助 LLM 来很好地解决特定任务 → 模型能力大幅提升，那些系统变得多余甚至有害 → 移除旧的脚手架，拿到新的性能收益 → 模型新能力涌现，去解决更困难的问题 → 回到第 1 步。

Box Agent 从最初设计架构到最终发布，中间多次被迫演进多个 Agent harness 组件，因为有些部分在模型变强后反而给 Agent 施加了不必要的约束。原来为处理上下文窗口限制而做的 chunking 策略，一旦模型上下文窗口和准确率提升，反而导致了更低质量的结果。

他总结的 main lesson：始终确保你在充分利用前沿能力，不要对你已经建好的技术产生留恋。模型进步的速度意味着你上个月搭的系统可能下个月就成了瓶颈。

https://x.com/levie/status/2039931799414194621
