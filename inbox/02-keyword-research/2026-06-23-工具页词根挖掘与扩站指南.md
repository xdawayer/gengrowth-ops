# 一、方法论参考：哥飞需求挖掘框架（来自 new.web.cafe 两篇文章）

## 1.1 核心原则
- **关键词＝需求**。新手策略是"新词新站"：优先做过去12个月内才出现显著搜索量的新词，竞争环境接近空白。
- **单域名、单站、单关键词**：把全部资源集中打一个核心关键词，而不是一个站塞很多不相关需求。
- **搜索量大 ≠ 值得做**。决策要看综合性价比，不能只看 Volume。

## 1.2 51个词根（起点池，非必须照搬）
Generator / Converter / Calculator / Editor / Maker / Checker / Detector / Builder / Template / Downloader / Analyzer / Optimizer / Tracker / Planner / Assistant / Simulator / Scraper / Comparator 等。

## 1.3 三个核心公式

**KGR（Keyword Golden Ratio，竞争供给比）**
```
KGR = allintitle:"关键词" 的谷歌结果数 ÷ 月搜索量
```
- 判断标准：KGR < 0.25 → 蓝海词，可以做
- 操作：谷歌搜索框输入 `allintitle:"关键词"`（带引号）→ 记录结果数 → 除以 Ahrefs/Semrush 查到的月搜索量

**KDRoi（性价比排序公式）**
```
KDRoi = Volume × CPC ÷ KD
```
- 用于在多个候选词之间排序，避免被"大词"的虚高搜索量迷惑
- 示例：词A（5000搜索量/$2 CPC/KD25）→ KDRoi=400；词B（50000/$0.1/KD80）→ KDRoi=62.5。A性价比是B的6倍多

**新手筛选区间参考**（两个版本，按风险偏好选）
| 维度 | 保守版（哥飞2025初） | 激进版（2026更新） |
|---|---|---|
| Volume | ≥ 30,000/月 | 3万-10万 |
| KD | ≤ 60 | 0-29 |
| CPC | > $0.01 | ≥ $0.1 |

## 1.4 验证四步法（找到候选词之后，建站/上线之前必走）
1. **Semrush/Ahrefs**：查 Volume + KD + CPC，KD>50新手直接放弃
2. **Google Trends**（近12个月）：曲线上升或新近冒头→优先；下降→放弃；注意季节性词（如圣诞节相关）全年仅1-2月有量
3. **谷歌前10名构成检查**：
   - 是否有 DA 80+ 老站/巨头垄断 → 有则谨慎
   - 内页排名多 还是 首页排名多 → 内页多说明用首页打有以强胜弱的机会
4. **KGR + KDRoi 双公式确认** → 两者没有正相关关系，必须都看（KGR看竞争者数量，KD看竞争者质量）

## 1.5 关键陷阱
- 自嗨选需求：必须用数据，不能凭直觉
- 追大词：搜索量大但KD90+的词（如 Password Generator），新站基本打不过
- 忽略CPC：搜索量大但CPC<$0.1，流量难变现
- 不验证就动手：验证四步法只需30分钟，能省下数周无效劳动

# 二、应用到 astrologywiki.com 的结论与计划

## 2.1 战略判断：51词根不能直接照搬上 astrologywiki

哥飞模型本身的前提是"单域名单站单关键词"——即每个无关词根对应**一个全新独立域名**，而不是塞进一个已有主题积累的老站。

Astrologywiki 是强垂直占星站，51词根里至少一半（Password Generator、Mortgage Calculator、PDF Editor、Currency Converter、YouTube Downloader、Web Scraper 等）跟占星毫无主题关联。直接塞入：
- 触发 Google **Site Reputation Abuse / Scaled Content Abuse** 政策风险，可能反向拖累现有占星词排名权重
- 跟目标用户（想测算星盘的人）完全不重合，转化路径断裂

**结论：词根只作为"搜索意图命名公式"参考，不当现成清单抄。** 分两条路径处理：

| 路径 | 内容 | 处理方式 |
|---|---|---|
| **A：强化 astrologywiki** | 占星适配版词根（见2.2） | 直接在现有域名加落地页，吃自身权重 |
| **B：独立新站** | 无关但数据好的词根（Password Generator类） | 哥飞模式——单独注册新域名打，与GenGrowth占星主线分开记账。**当前未决定是否启动，待资源评估** |

### ⚠️ 重要修正：FAQ结构化数据不是"缺失"，是"已实现但内容不匹配"——这是更严重的问题

最初判断"无FAQ结构化数据"是错的（WebFetch工具摘要看不到`<script>`里的JSON-LD，漏判）。用curl核实原始HTML后发现：**16个工具页面全部已经植入了FAQPage schema**，但抽查16/16后发现：

- **15/16 页面**：FAQPage schema里的问题文本，跟页面可见的`<h2>`标题**文字不一致**（即使主题相关，逐字对比也不匹配），其中多数schema问题在页面上**完全找不到对应的可见文字**
- **唯一正确实现的页面：saturn-return-calculator**——它的4个FAQ schema问题与前4个可见H2标题逐字一致，可以作为修复其余15个页面的参照模板

示例（birth-chart-calculator）：
- Schema问题："Is the birth chart calculator free?"、"Do I need my birth time?"
- 可见H2标题：完全没有这两句问题文字，只有泛化的"What is in your chart"等小标题

**为什么这比"没有FAQ schema"更糟**：Google结构化数据规范要求FAQPage标记的内容必须是用户在页面上能看到的文字。schema与可见内容不一致，轻则该标记被Google直接忽略（拿不到FAQ富媒体展示位，白做），重则16个页面是同一套模板批量生成、同一个bug，存在被判定"隐藏内容/结构化数据滥用"触发人工处罚的风险。**优先级应提到比"补内容深度"更高的位置。**

**重要前提**：这16个页面是 **2026-06-18 左右上线（5天内）**，GSC基本无有效排名数据，意味着：
- 现在是换slug的**零/低成本窗口期**——没有外链、没有内链锚文本积累、可能尚未完全索引
- 窗口期很短（通常2-4周），一旦开始被内容文章内链引用或外链SOP拿去做锚文本，换词成本会迅速上升

## 2.4 同义词替换决策流程（本周要跑的步骤）

**第一步：查GSC索引状态**
打开 GSC「网页索引」报告，确认16个URL是"已编入索引"还是"已发现，尚未编入索引"。
- 未索引 → 换词零代价，无需考虑重定向
- 已索引但零曝光 → 换词低代价
- 已有曝光/点击 → 换词前要拿数据验证收益是否明显超过损失（KDRoi差距>2倍才值得冒险）

**第二步：列出每个工具的同义候选词，查Ahrefs数据**

| 当前词 | 候选同义词 | 意图风险提示 |
|---|---|---|
| Birth Chart Calculator | Natal Chart Calculator | 圈内基本等价，值得查 |
| Rising Sign | Ascendant Calculator | Ascendant更"正确"但大众搜索量通常远低于Rising Sign，慎换 |
| Synastry Calculator | Compatibility Calculator | 后者量大但意图偏"娱乐测验"，跳出率风险高，慎换 |
| Rodden Rating | Birth Time Accuracy / Birth Time Rectification | Rectification是另一种服务意图（修正而非评估），不是纯同义词 |
| Current Planets | Planetary Positions Today / Transit Chart | 待查 |
| Moon Phase Calculator | Moon Phase Today / Lunar Phase Calculator | 待查 |
| Electional Astrology | Astrological Timing / Best Day Calculator | 待查 |
| Astrocartography Map | Relocation Astrology Map | 待查 |
| Celebrity Astro Twins | Celebrity Birth Chart / Famous People Zodiac Sign | 待查 |

其余（Big Three、Moon Sign、Ephemeris、Solar Return、Saturn Return、Composite Chart、Energy Timeline）暂无明显更优同义词，保留现状。

**第三步：每对候选词跑 KGR + KDRoi + Trends + 前10名构成检查**，按2.4流程公式计算，排除意图漂移的选项。

**第四步：决策分档**
- 数据明显更优 + 意图一致 + 当前页面零曝光 → 直接换
- 数据接近 或 意图有差异 → 保留现词，另开一个同义词的独立落地页（避免内部关键词蚕食）
- 数据不明确 → 不动

## 2.5 内容与内链优化（按优先级排序，已按2026-06-23实测数据调整）

**P0 — 修复FAQ schema与可见内容不匹配（15/16页面）**
以 saturn-return-calculator 为参照模板：把每个页面FAQPage schema里的问题文本，改成与页面可见H2标题逐字一致（或反过来，把H2标题改写成schema里的问题）。这是修bug，不是加功能，技术改动量小，但要先做——拖着不修，其余优化做得越多，潜在的结构化数据滥用风险面越大。

**P1 — 建工具间互链矩阵**
16个页面里15个零互链，是最大的内链权重浪费。需要设计一个"相关工具推荐"模块，按工具分类关系（Core Signs / Relationships / Timing 等）规定每个工具该推荐链向哪几个其他工具。

**P2 — 补面包屑导航（Schema.org BreadcrumbList）**
16/16缺失，模板级改动，一次性全站生效。

**P3 — 内容生产线反向内链**
每周25+篇文章系统性内链回对应工具页，目前基本浪费。

**P4 — 补最薄页面的内容深度**
优先 Solar Return Calculator（270-300字）、Composite Chart Calculator（290-310字），其次 Big Three / Rising Sign / Current Planets（320-350字）。Rodden Rating、Energy Timeline已经较厚，可以先不动。
