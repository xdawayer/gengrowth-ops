# 网页索引编制自动化需求文档

## 一、自动化目标

**核心问题：** 当前文章发布后依赖人工手动向 GSC 提交 URL，收录状态缺乏系统监控，出现问题时无人主动发现，文章长期未被 Google 收录而无人知情。

**自动化目标：**
- 文章发布后**自动提交收录请求**，不依赖人工记忆
- 建立**收录状态监控机制**，异常 14 天内自动发现并推送告警
- 建立**未收录原因诊断框架**，区分正常等待 vs 需要处理的问题，避免误判和过度焦虑
- 明确每类问题的标准处理动作，减少人工判断成本

---

## 二、当前人工流程

| 环节       | 当前操作                                      | 频率        | 估计耗时   |
| -------- | ----------------------------------------- | --------- | ------ |
| ~~URL 提交收录~~ | ~~进入 GSC → URL 检查工具 → 手动输入 URL → 点击"请求编入索引"~~ | ~~每篇文章发布后~~ | ~~5min/篇~~ | **✅ 已自动化** |
| 收录状态检查   | 进入 GSC → URL 检查工具 → 手动输入文章 URL → 查看当前收录状态 | 不定期（可能遗漏） | 5min/篇 |
| 未收录问题诊断  | 看 GSC 状态描述 → 自行判断原因 → 没有标准框架，容易误判         | 偶发        | 不定     |
| 修复后重新提交  | 人工再次进入 GSC 提交                             | 每次修复后     | 5min/次 |

**当前核心问题：**
- URL 提交收录已自动化 ✅；当前缺口在于**提交后的状态监控**：没有系统机制持续追踪哪些文章已收录、哪些超期未收录
- 没有监控机制，未收录文章可能沉默数周
- 无诊断框架，遇到"Crawled - currently not indexed"等状态时不知如何判断和处理

## 三、未收录原因分类框架（诊断核心）

> 这是本文档最重要的部分。区分**正常等待**和**需要处理**，是避免误判和乱操作的前提。

### 3.1 正常情况（不需要处理，等待即可）

| GSC 状态 / 场景 | 含义 | 建议操作 |
|---|---|---|
| **Discovered - currently not indexed** | Google 发现了页面但尚未爬取，排在爬取队列中。新站点和新文章极为常见，不代表技术问题 | 继续等待；D+21 仍未爬取则升级关注 |
| **新文章发布 D+14 内** | Google 爬取和收录本身有时间延迟，新站沙盒期延迟更长（有时 2–4 周）| D+14 再做第一次正式评估，不提前干预 |
| **Alternate page with proper canonical tag** | 页面有正确的 canonical 标签，Google 收录了 canonical 指向的主版本，这是预期行为 | 无需操作，属于正常的 URL 归一化处理 |
| **有意设置 noindex 的页面** | 如：隐私政策 / 服务条款 / 感谢页 / 登录页等不需要出现在搜索结果的页面 | 无需操作，确认是否有意设置即可 |
| **新站整体收录率偏低** | astrologywiki.com 于 2026-01-25 首次曝光，目前仍在沙盒期，整体收录率低于成熟站点是正常现象，不代表技术问题 | 持续发布高质量内容，外链权重积累后收录速度会改善 |
| **低流量长尾页面被降优先级** | Google 会主动降低对低价值页面的爬取频率，这是 Google 的资源分配策略 | 改善内容质量和内链密度，提高页面感知价值 |

> ⚠️ **审计时的常见误判：** 看到大量"Discovered - currently not indexed"就认为有技术问题 → 这在新站是完全正常的。真正需要关注的是"Crawled - currently not indexed"（已爬取但拒绝收录）。

---

### 3.2 需要处理的问题（分优先级）

| 优先级            | GSC 状态                                      | 含义                                  | 处理方向                                               |
| -------------- | ------------------------------------------- | ----------------------------------- | -------------------------------------------------- |
| 🔴 **P0 紧急**   | Not found (404)                             | 页面已消失，Google 爬取时返回 404              | 恢复页面 或 设置 301 重定向到新 URL                            |
| 🔴 **P0 紧急**   | Server error (5xx)                          | 服务器错误，Google 无法访问页面                 | 联系技术团队排查服务器                                        |
| 🟡 **P1 本周处理** | Crawled - currently not indexed             | 已爬取但主动拒绝收录：内容质量不足（薄内容 / 重复 / 信息价值低） | 内容扩充（≥1500词）/ 增加内链 / 提升 E-E-A-T 信号 / 检查是否与已有内容高度重复 |
| 🟡 **P1 本周处理** | Excluded by 'noindex' tag（非有意）              | 页面被错误加了 noindex 标签                  | 检查 CMS 模板或文章元数据，删除 noindex                         |
| 🟡 **P1 本周处理** | Blocked by robots.txt（非有意）                  | 页面被 robots.txt 错误屏蔽                 | 检查 robots.txt 配置，移除对应 Disallow 规则                  |
| 🟠 **P2 排期处理** | Duplicate, Google chose different canonical | Google 不认可你设置的 canonical，选择了另一个版本   | 检查页面是否存在实质重复内容，或 canonical 标签设置是否有误                |
| 🟠 **P2 排期处理** | Soft 404                                    | 页面存在但内容极少或不相关，Google 判定等同于 404      | 补充有实质内容，或删除页面并重定向                                  |
| 🟠 **P2 排期处理** | Page with redirect（链过长）                     | 重定向链太长（>3跳）或存在重定向循环                 | 简化重定向链，确保 301 直指最终 URL                             |
| ⬜ **P3 观察**    | Indexed, though blocked by robots.txt       | 收录了但 robots.txt 有屏蔽规则，存在状态不一致风险     | 核查 robots.txt 是否需要更新                               |

### 3.3 内容质量诊断清单（"Crawled - currently not indexed" 专项）

> 这是最常见的需要处理的状态，原因较多，需要逐项排查：

| 检查项 | 达标标准 | 诊断方法 |
|---|---|---|
| 内容字数 | ≥ 1,200 词（竞争力较强词 ≥ 1,800）| 字数统计工具 |
| 内容独特性 | 与站内其他页面重复率 < 30% | Copyscape / 人工比对 |
| 内链密度 | 至少 2–3 条来自其他已收录页面的内链 | 检查内链追踪表 |
| 页面可访问性 | 未登录状态可正常访问，加载时间 < 3s | 浏览器无痕模式测试 |
| 结构化数据 | 有 Article schema（或 FAQ / HowTo）| Google Rich Results Test |
| 标题与 meta | title 和 meta description 存在且不重复 | GSC / 浏览器检查 |
| E-E-A-T 信号 | 有作者署名 + 作者简介 + 发布日期 | 页面直接检查 |

## 五、各环节详细自动化需求

### 5.1 【⚠️ 部分完成，需补充】发布后自动提交 + 追踪表写入

> URL 提交 GSC 已自动化。**当前缺口：提交后没有写入追踪表**，导致 5.2 的监控流程缺少起点（不知道哪些 URL 已提交、何时提交）。

**需补充的自动化动作（在现有提交流程末尾追加）：**

在现有 Make 工作流的 GSC 提交步骤之后，新增一个 Google Sheets 写入节点：

| 字段 | 写入内容 |
|---|---|
| URL | 提交的完整页面 URL |
| 文章标题 | 从 CMS 元数据获取 |
| 发布日期 | CMS 发布时间戳 |
| 首次提交日期 | 触发时的当前时间 |
| 当前状态 | 固定写入："已提交，监控中" |

**工具方案：** 在现有 Make 流中追加 Google Sheets → Add Row 节点，无需新建工作流

**验收标准：**
- 每次 GSC 提交触发后，追踪表自动新增一行，URL 和时间戳准确
- 此追踪表作为 5.2 监控流程的数据起点

**对齐时核查：**
- 当前自动化是否已同步 ping sitemap？
- 是否已提交 IndexNow（Bing/Yandex）？

### 5.2 【P0】收录状态自动监控 + 告警

**触发条件：** 定时任务，每天 09:00 自动运行

**监控逻辑：**

| 检查时间点 | 触发条件 | 操作 |
|---|---|---|
| D+3 | 首次提交后 3 天 | 查询 GSC URL 状态，记录当前覆盖范围状态 |
| D+7 | 首次提交后 7 天 | 再次查询，若已收录 → 标记 ✅ 完成；若仍未收录 → 继续监控 |
| D+14 | 首次提交后 14 天 | 若仍未收录 → 触发诊断流程（见 5.3）+ 推送飞书告警 |
| D+30 | 若 14 天诊断后仍未收录 | 推送升级告警，标记为"需重点关注" |

**状态字段：**
```
已提交 → 监控中 → ✅ 已收录 / ⚠️ 超期未收录（触发诊断）/ 🔴 紧急问题（404/5xx）
```

**飞书告警格式：**
```
⚠️ 索引超期提醒
页面：[文章标题]
URL：[URL]
发布日期：[日期]
已过天数：D+14
当前 GSC 状态：Crawled - currently not indexed
建议操作：[自动诊断结果，见下方]
```

**工具方案：** Python Cron + GSC API → Google Sheets 更新 → Make → 飞书 Bot

**验收标准：**
- D+14 未收录 100% 触发告警，无遗漏
- 已收录页面自动停止监控，不产生噪音告警
- 告警消息包含 GSC 状态和具体建议操作

---

### 5.3 【P1】未收录原因自动分类 + 处理建议

**触发条件：** 由 5.2 的 D+14 未收录监控触发

**分类逻辑：**

拉取该 URL 的 GSC 覆盖范围详情（`urlInspection.index.inspect` 返回 `indexStatusResult.verdict` 和 `indexStatusResult.coverageState`），按以下规则路由：

```
coverageState = "Crawled - currently not indexed"
  → 诊断类别：内容质量问题
  → 自动检查：抓取页面字数 + 检查 meta robots 标签
  → 推送建议：内容扩充清单（见 3.3）

coverageState = "Discovered - currently not indexed"
  → 诊断类别：正常排队（新站常见）
  → 操作：继续监控至 D+21，不推送告警（避免误判噪音）

coverageState = "Excluded by 'noindex' tag"
  → 诊断类别：标签问题（需人工确认是否有意）
  → 推送告警：附 URL + 要求确认是否有意设置 noindex

coverageState = "Blocked by robots.txt"
  → 诊断类别：配置错误（大概率无意）
  → 推送 🔴 紧急告警：附 URL + robots.txt 当前内容

coverageState = "Not found (404)" 或 "Server error (5xx)"
  → 诊断类别：技术故障
  → 推送 🔴 紧急告警：立即处理

coverageState = "Duplicate, Google chose different canonical"
  → 诊断类别：重复内容 / canonical 问题
  → 推送建议：检查 canonical 标签和页面重复度

其他状态
  → 推送原始 GSC 状态 + 附分类框架链接，由人工判断
```

**输出格式（飞书消息）：**
```
🔍 索引诊断报告
页面：[文章标题]
URL：[URL]
GSC 状态：Crawled - currently not indexed
诊断结论：内容质量不足
建议操作：
  □ 检查字数（目标 ≥ 1,200 词）
  □ 确认至少 2 条来自已收录页面的内链
  □ 检查是否与站内其他页面内容高度重复
  □ 确认 author 字段和发布日期存在
处理完成后：在追踪表中标记"已修复"，系统将自动重新提交收录请求
```

**工具方案：** GSC API + Python 规则引擎 → 飞书 Bot

**验收标准：**
- 覆盖常见 GSC 状态类型（≥ 8 种），每种有对应建议
- "Discovered - currently not indexed"不触发噪音告警
- 诊断报告推送时间 ≤ 30 分钟（D+14 监控触发后）

### 5.4 【P1】修复后自动重新提交

**触发条件：** 人工在追踪表中将某 URL 状态更新为"已修复"

**处理逻辑：**
1. 追踪表触发 → Make webhook 检测到状态变更
2. 自动调用 GSC URL Inspection API 重新请求编入索引
3. 更新追踪表：重新提交日期 / 状态重置为"监控中"
4. 重新进入 5.2 监控流程（从 D+3 重新计时）

**工具方案：** Google Sheets Trigger → Make → GSC API

**验收标准：**
- 人工标记"已修复"后 ≤ 1 小时自动重新提交
- 追踪表状态和时间戳准确更新

---

### 5.5 【P2】月度索引状态自动报告

**触发条件：** 每月 1 日自动生成

**报告内容：**

| 指标 | 数据来源 | 说明 |
|---|---|---|
| 本月新发布文章数 | 追踪表 | — |
| 已收录 / 未收录比例 | GSC API | 按发布月份统计 |
| 平均收录天数 | 追踪表 | 发布日至首次确认收录日 |
| 当前各 GSC 状态分布 | GSC Coverage API | Indexed / Crawled-not-indexed / Discovered / Excluded 各有多少 |
| 本月修复处理数量 | 追踪表 | 标记"已修复"的 URL 数 |
| 需关注 URL 清单 | 追踪表 | 仍处于"⚠️ 超期未收录"状态的所有 URL |

输出：飞书文档 / Google Doc，@负责人审阅

**工具方案：** GSC API + Google Sheets API → Python 报告生成 → 飞书文档

---

### 5.6 【P3】Sitemap 自动更新与提交

**需求：**
- 确认 CMS/网站框架在新文章发布后自动更新 sitemap.xml（大多数现代框架默认支持，需核查）
- 若未自动更新：脚本在文章发布后触发 sitemap 重新生成
- 发布后自动 ping `https://www.google.com/ping?sitemap=[url]`

**验收标准：**
- 新文章 URL 在发布后 ≤ 1 小时出现在 sitemap.xml 中
- Sitemap ping 与 5.1 的 GSC 提交同步触发，无需单独操作
