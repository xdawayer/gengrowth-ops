# 外联 & 索引自动化需求 — 交叉审核反馈

> 审核方式：4 路并行 fan-out（架构对齐 / 技术核查 / 成本风险 / 逻辑一致性）+ Codex 独立第二意见，五位审核员结论高度一致。
> 各文档的逐条问题已追加在原文档末尾的「审核反馈」节；本文档承载**跨两份文档的战略层、优先级与对齐会决议项**。

---

## 一句话结论

需求拆解质量高（业务理解到位），但**实现层有两个会直接导致返工的硬伤，且整体方向与公司主力工作流 flow-mvp（本地）相反**——在搭一套 AI 碰不到的云端黑盒。建议：**需求照单全收，底座推翻重选**。索引文档回炉，外联文档大幅瘦身缓做。

---

## 🔴 三个硬伤（详见各文档末尾）

1. **索引文档的核心卖点"自动请求 Google 收录"技术上不存在**（GSC inspect API 只读 / Indexing API 仅限招聘·直播 / IndexNow 不通 Google / sitemap ping 端点 2023 已废弃返 404）。整条监控链路建在假起点上。→ 详见索引文档末尾反馈。
2. **新域名沙盒期跑自动冷邮件 = 拿主域信誉去赌**（缺独立发信域名 + 预热 2–4 周 + CAN-SPAM/GDPR 合规，"一周上线"不现实）。→ 详见外联文档末尾反馈。
3. **把"买 dofollow 链接"流程化、留痕化 = 积累 SEO 风险证据库**（应改用 rel=sponsored/nofollow）。→ 详见外联文档末尾反馈。

---

## 🧭 战略层：与 flow-mvp 的根本冲突（本文档重点）

两份文档默认的栈 = Make/Zapier + Instantly + Hunter + Firecrawl + Python Cron + **Google Sheets 当真相源** + 飞书。这是**第二套并行的自动化范式**，与公司"AI 通过 git + 文本文件本地干活"的战略相反。

问题不在用 SaaS，而在**把核心数据与核心逻辑放进 SaaS 界面里**：Claude Code 读不到、git 不能 diff、断了本地无法复现、接手只能看截图。

**逐环节判定后的结论**：约 17 个子环节里，**只有"邮件发送/送达"（Instantly）一项真正需要 SaaS**。其余全部可、且应改造成 flow-mvp 形态——本地 `.mjs` 脚本 + Claude Code subagent + git 里的 CSV/markdown，复用现有 `red-lines.mjs`、`content-draft-templates`。

| 环节 | 文档原方案 | 建议 |
|---|---|---|
| 候选站点预判 / 投稿要求摘要 / 表单字段提取 | Firecrawl + GPT-4o mini + Make | 改写：本地脚本 fan-out + Claude subagent + 写 CSV |
| 邮件序列发送 / 送达 / 回复检测 | Instantly.ai | **保留 SaaS**（唯一真正需要的） |
| 状态同步 / 链接验收 / 竞品反链拉取 | Make/Zapier/Python Cron | 改写：本地 `.mjs` + cron，结果写 git CSV |
| 月度报告 / 锚文字红线 / 原因分类 | Sheets 公式 / 飞书文档 / Python | 改写：Claude 生成 markdown 进 git；红线扩展 red-lines.mjs |
| **追踪表（真相源）** | Google Sheets | **统一回 git CSV**；Sheets 降级为只读视图 |

**收益**：月成本从 $122–167 降到约 **$37**（仅 Instantly）；逻辑回到 git，Claude Code 既是执行者也是维护者，人只做 review/PR。

## 📋 优先级重排：6-28 前真正该做的

| 排序 | 做什么 | 理由 | 成本/风险 |
|---|---|---|---|
| **#1 马上做** | 索引**收录监控 + 超期告警**（去掉"自动提交收录"假动作，只做 sitemap 校验 + GSC 只读监控 + 人工点按清单） | 新站沙盒期真痛点 | ≈$0，纯只读 |
| **#2 马上做** | 外联文档**第九章 SOP 人工优化**（付款前确认清单 / 去重 / 收录验证规范） | 零成本、当天落地、防真金白银损失 | $0 |
| **#3 做（降级版）** | 外联**轻量跟进提醒**（"D+N 没回复就提醒"，本地脚本，不碰 Instantly 全栈） | 解决唯一真痛点"没人跟进" | 近 $0 |
| 缓 | 候选站点预判、Hunter 查找、原因分类、月报 → 改造成 flow-mvp 形态后再上 | 好东西但不紧急 | — |
| **明确缓** | 整套外联自动冷邮件栈 | 体量没到盈亏平衡点 + 合规/声誉风险 | $122-167/月 + 隐性维护 |
| 砍 | 品牌提及提醒、锚文字实时统计、断链监控、HARO | 新站阶段纯空转 | — |

> **"月省 30–45 小时"存疑**：最大头 10–15h 是"人工写邮件"，文档自己说保留人工，不应算进自动化收益；且全文没有一个当前真实外联量的数字。

## ❓ 对齐会（6-24）建议决议的事项

1. **过去 60 天真实外联量是多少？联系几个站、成几个？**（外联自动化优先级的地基）
2. **冷邮件用哪个域名发？** 若用主域 astrologywiki.com → 叫停。
3. **索引文档"URL 提交已自动化 ✅"现在跑的到底是什么？** 确认没人误用 Indexing API 提交普通文章（若有，立即停用防处罚）。
4. **底座方向拍板**：采纳"需求保留 + 实现改造成 flow-mvp 形态（git CSV 真相源 + 本地脚本，仅 Instantly 留 SaaS）"？
5. **追踪表架构**：外联表与索引表是一张还是两张？真相源放 git 还是 Sheets？
6. **运维归属与密钥**：谁搭、谁维护、断了谁发现、API key 放哪。
