今天读什么｜别只给 Agent 接 API，要把浏览器变成“最后一公里执行层”

**1. 适合谁读 / 预计阅读时间**  
适合正在做 AI Builder、智能体工作流、线索采集、内容生产、竞品监控、增长自动化的人读。  
预计阅读时间：8—10 分钟。

**2. 为什么值得读**  
很多人做 Agent 时，默认思路是：能接 API 就接 API，接不上就先放着。  
但真实业务里，最麻烦的往往恰恰是那些**没有好 API、接口不全、登录复杂、页面经常改版**的系统：后台表单、供应商门户、招聘网站、活动报名页、竞品定价页、会员区内容页。

最近一批浏览器智能体实践给了一个更成熟的方向：**浏览器不是“不得已的土办法”，而是 Agent 落地真实世界的最后一公里执行层。**  
但这不代表什么都交给浏览器去点。真正可用的方法是分层：**能搜索就别进浏览器，能读取就别点按钮，能脚本化就别完全放飞，能校验结果就别只看执行过程。**

这对 GenGrowth 很重要。因为我们的很多增长动作天然跨页面、跨账号、跨站点：找线索、采集内容、看竞品、填表单、核对信息、跑运营后台。只靠 API，很多环节永远接不上；只靠人工，规模永远上不去。浏览器层，正好补上这块空白。

**3. 核心概念**  
第一，**浏览器是执行层，不是默认入口。**  
Firecrawl 这类工具的思路很清楚：先用搜索、抓取、抽取解决“读”的问题；只有在必须点击、翻页、登录、提交时，再进入浏览器交互。也就是说，浏览器该做的是最后一段，而不是整个流程都从浏览器开始。

第二，**浏览器智能体不是一个档位，而是一条光谱。**  
Browserbase 把它分成几层：  
- 固定流程里只让 AI 处理个别页面动作；  
- 大部分流程可控，只把复杂步骤交给 Agent；  
- 让 Agent 自己决定怎么走，但工具边界仍由你提供；  
- 完全目标驱动，直接让它去完成结果。  
关键不是“越自动越高级”，而是**风险高的地方多控制，变化大的地方多放权**。

第三，**抗页面改版能力，才是浏览器自动化真正的分水岭。**  
Skyvern 反复强调一个问题：传统 XPath/CSS 选择器方案，一旦网页结构改了就坏；视觉理解型方案更像人在看页面，靠文本、区域、上下文识别元素。  
简单说：以前自动化怕“这个按钮换位置了”，现在更应该追求“它知道那个按钮还是那个意思”。

第四，**浏览器任务不要只验证路径，要验证结果。**  
Browserbase 最近和 Microsoft 做的 Universal Verifier 很有启发：浏览器任务往往不只有一条正确路径，死盯“是不是按预定轨迹点的”会越来越不现实。更可靠的检查方式是：**最后是否真的完成了目标**。  
比如，不要只看“它有没有点提交按钮”，而要看“表单是否真的创建成功、结果页是否真的出现、数据是否真的写入”。

第五，**最稳的系统通常是混合式，不是全自动式。**  
先搜、再读、再抽取、最后才浏览器执行；固定步骤脚本化，不确定步骤交给 Agent；关键节点再加人工确认或结果校验。  
这比“一上来就让 Agent 自己开浏览器到处试”稳得多。

**4. 可复用方法**  
如果你想把浏览器智能体真正接进工作流，可以直接用这个五步法：

1）先区分“读”和“做”。  
读信息用搜索/抓取；做动作才进浏览器。

2）先挑没有 API、但流程重复的场景。  
比如采集竞品价格、登录后台下载报表、跨站点查公司信息。

3）把流程拆成“固定段”和“不确定段”。  
固定段写死，不确定段交给 Agent 推理。

4）给浏览器步骤补结果校验。  
不是看它点了什么，而是看目标是否真的达成。

5）从单站点试点，逐步扩到多站点。  
不要一开始就做万能网页工人，先做一个能稳定复跑的小闭环。

**5. GenGrowth 可以怎么用**  
第一，用在线索与客户研究。  
Agent 先搜公司，再进官网、招聘页、产品页、定价页补齐结构化信息，最后写回 CRM 或选题池。

第二，用在内容与情报采集。  
很多高价值页面有登录、翻页、动态加载，只靠静态抓取不够。可以把浏览器层作为“补采集”的最后一步。

第三，用在增长执行后台。  
比如活动平台、投放后台、报名系统、渠道门户，这些地方常常接口不开放，但又有重复动作，适合做半自动浏览器工作流。

第四，用在产品 QA。  
让浏览器 Agent 按真实用户路径走一遍注册、领取、提交、跳转流程，再由结果校验器判断是否真的完成。

**6. 今日行动**  
挑一个你们现在还在手工做、且“网页上能做但 API 接不上”的流程，把它写成四列：  
**目标结果 / 固定步骤 / 不确定步骤 / 成功校验方式**。  
如果这四列你写不清，说明还不适合自动化；如果写清了，这就是最值得先做的浏览器 Agent 试点。
