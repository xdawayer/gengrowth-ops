今天读什么｜别把所有工具都塞给 Agent，要学会做“按需供给”

**1. 适合谁读 / 预计阅读时间**  
适合正在做 AI Builder、智能体工作流、内容生产自动化、知识库检索、内部 Bot 编排的人读。  
预计阅读时间：8–10 分钟。

**2. 为什么值得读**  
很多团队做 Agent 时，第一反应是“多接几个工具，多给点资料，多加几层流程”。看起来更强，实际上常常更脆。  
最近几篇公开实践给了一个很清楚的提醒：**Agent 失败，很多时候不是因为工具太少，而是因为工具太多、上下文太肥、选择太乱。**

Anthropic 把这个问题总结成“上下文工程”——模型每一步真正能专注的注意力是有限的；LangChain 进一步把常见方法归纳成写入、选择、压缩、隔离四类；Vercel 则给了一个很硬的产品案例：他们把一个复杂数据 Agent 的大部分专用工具删掉后，反而把成功率从 80% 提到 100%，执行速度提升到 3.5 倍。  
这对 GenGrowth 很重要。因为我们未来的很多流程都会越来越像“会调用工具的系统工人”：查资料、读网页、整理结构、生成草稿、写回任务、同步知识。如果不控制工具和上下文，Agent 很快就会变成一个“什么都能碰、但什么都碰不好”的杂工。

**3. 核心概念**  
第一，**上下文不是越大越好，而是越准越好**。  
模型不是看到越多信息就越聪明。信息一多，注意力会被稀释，重要信号反而更难抓住。

第二，**工具越多，决策歧义越大**。  
如果两个工具功能重叠，连人都说不清“这时候该用哪个”，那 Agent 更容易乱选。很多异常不是模型笨，而是工具设计本来就模糊。

第三，**不要把所有工具一次性挂上去**。  
Anthropic 在文档里给了四个很实用的方向：按需搜索工具、把多步工具链折叠成一个脚本执行、缓存稳定工具定义、把过期的工具结果从上下文里裁掉。核心思想就一句话：**需要时再加载，不要默认全带。**

第四，**有时最好的 Agent 架构，是更少的架构**。  
Vercel 的经验很有代表性：与其造很多“帮模型思考”的专用工具，不如直接给模型一个更自然的工作环境，比如文件系统、清晰命名的数据目录、可读的说明文件。前提不是模型万能，而是你的底层资料本来就足够清楚。

第五，**隔离比堆叠更重要**。  
不是所有信息都要让主 Agent 一次看到。搜索结果、长网页、临时中间件、子任务探索，都可以先在子 Agent、运行环境或外部笔记里消化，最后只把真正有用的结论交回来。

**4. 可复用方法**  
如果你想给现有 Agent 做一次“瘦身”，可以直接用这个五步法：

1）先列出真实任务，而不是先列工具。  
比如“找资料、提炼要点、生成草稿、发到频道”，不要上来就挂十几个 MCP。

2）把重叠工具合并。  
能被一个脚本完成的，不要拆成五个来回调用。

3）按阶段暴露工具。  
检索阶段只给搜索和读取；写作阶段只给整理和输出；不要默认全开放。

4）定期压缩中间结果。  
长网页、长日志、长 tool result 不要一直留在主上下文里，提炼后就清掉。

5）把“清晰结构”当成基础设施。  
目录命名、文档索引、模板、字段规范，往往比再加一个智能工具更值钱。

**5. GenGrowth 可以怎么用**  
第一，用在 Hermes / 多 Bot 协作。  
不要让每个 bot 都默认拿到同一大堆工具。研究 bot 以搜索、抽取、总结为主；Ops bot 只拿执行层工具；公共频道 bot 只处理公开信息和低风险动作。

第二，用在研究报告流水线。  
把“搜集来源”“读原文”“提炼框架”“写中文学习文档”拆成不同阶段，每阶段只给必要能力，减少上下文污染和风格跑偏。

第三，用在知识库与内容生产。  
不要把整库文档一次性喂给 Agent。先给索引、目录、标签和少量高相关材料，再让它按需读取正文。

第四，用在增长自动化产品设计。  
以后无论做 SEO、线索整理、内容编排还是内部助手，都可以把“按需供给工具和上下文”作为默认架构原则，而不是把“全接入”当卖点。

**6. 今日行动**  
今天就可以做一件小事：  
挑一个你们已经在跑的 Agent 流程，做一次“工具减半实验”。

具体做法：  
- 列出它现在能调用的全部工具  
- 删掉其中功能重叠或低频的 30%–50%  
- 把一个多步链路改成单脚本执行  
- 只保留当前阶段真正需要的上下文  
- 对比一次运行的成功率、步骤数、耗时和输出质量

如果结果更稳，你们就找到了一条很关键的产品原则：**Agent 的能力，不在于背了多少工具，而在于每一步拿到的东西是否刚刚好。**
