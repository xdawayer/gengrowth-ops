今天读什么｜别让 Agent 每次从头学，开始建设“技能层”

1. 适合谁读 / 预计阅读时间

适合正在做 AI Builder、内容生产、研究整理、代码协作、知识库沉淀、增长自动化的人读。  
预计阅读时间：8—10 分钟。

2. 为什么值得读

很多团队现在已经不缺模型，也不缺工具，真正卡住落地的往往是另一件事：**同一类工作每次都要重新解释一遍。**

今天让 Agent 写报告，明天让它整理资料，后天让它改页面，看起来都能做；但一换会话或项目，Agent 又像“失忆”了一样，要重新交代口径、步骤、禁区和验收标准。

最近 Anthropic 和 Vercel 都在强调一个方向：不要只把经验留在聊天记录里，而要把重复有效的做法打包成 **Skill（技能）**。它不是一次性的提示词，而更像一份可复用的“岗位说明书 + SOP + 辅助脚本”。Agent 需要时再加载，不需要时不占上下文。

这对 GenGrowth 特别重要。因为我们的很多流程天然就是重复的：研究选题、抓来源、提炼结构、生成初稿、复核格式、沉淀知识、安排下一步。如果能沉淀成技能层，Agent 才会从“会聊天”变成“会按我们的方法交付”。

3. 核心概念

第一，**技能不是 Prompt 的别名，而是程序化经验包。**  
一个好技能里不只有一句指令，还会包含触发条件、步骤、输出格式、参考资料，甚至脚本。它解决的是“以后遇到这类任务，怎么稳定做”。

第二，**不是所有规则都该塞进常驻上下文。**  
团队通用原则、项目硬约束，适合放在 `CLAUDE.md` 或 `AGENTS.md` 这种每次都加载的文件里；只在特定场景才需要的流程，适合放进技能，按需调用。

第三，**技能本质上是 Agent 的程序性记忆。**  
长期事实记忆解决“记住什么”，技能层解决“具体怎么做”。前者像通讯录，后者像标准作业流程。

第四，**技能要能被触发，才算真的存在。**  
Vercel 的实测提醒了一个现实：有些技能本身没问题，但 Agent 不一定稳定触发它。所以高频通用信息要放到始终可见的入口，技能更适合承载专项流程。

第五，**技能不能替代硬护栏。**  
像“每次改完都要检查格式”“某目录禁止写入”这类刚性要求，更适合用 hook、脚本或测试强制执行。简单说：技能负责教方法，护栏负责保底线。

4. 可复用方法

如果你想给自己的 Agent 体系补一层技能系统，可以直接用这个四步法：

- **先找重复 3 次以上的任务。** 先挑最常返工、最容易口径漂移的一类活。  
- **按三层拆开。** 常驻规则放 `AGENTS.md/CLAUDE.md`，专项流程放 Skill，必须执行的检查放 hook 或脚本。  
- **每个 Skill 只写最小闭环。** 至少包含：用途、触发场景、步骤、输出格式、常见错误。  
- **拿真实任务回放测试。** 不只看“写得像不像”，还要看会不会被正确触发、能不能减少人工补充说明。

一个简单判断标准是：如果这个技能上线后，别人第一次接手同类任务时，不用再追着你问细节，它就开始有价值了。

5. GenGrowth 可以怎么用

第一，把研究报告流程做成 Skill。把“不重复选题、固定结构、10 分钟可读、来源归因”的要求一次写清，减少每天重新校口径。  
第二，把内容生产拆成技能链。比如“选题 Skill”“结构提炼 Skill”“发布版改写 Skill”“来源复核 Skill”，让不同 Agent 在固定边界里接力。  
第三，把内部协作模板技能化。任务卡撰写、周报整理、资料归档、Ops 草稿生成，都更依赖流程一致性，而不是灵感。

6. 今日行动

今天就做一件小事：挑一个你已经重复做过至少 3 次的流程，写出它的第一版 Skill 草稿。  
先写清 5 个点：用途、触发场景、步骤、输出格式、常见坑。  
然后拿明天的真实任务跑一次，看看它究竟减少了多少解释成本。
