今天读什么｜别再只改提示词，要给 Agent 设计一条“上下文流水线”

1. 适合谁读 / 预计阅读时间  
适合正在做 AI Builder、智能体工作流、知识库、内容生产、线索跟进自动化的人。尤其适合已经发现“模型本身挺聪明，但一接真实资料、工具、历史记录就开始跑偏”的团队。预计阅读时间：8—10 分钟。

2. 为什么值得读  
过去大家总把问题归咎于 prompt 不够好：多补几句、换个语气、再加几个例子，似乎就能让 Agent 更稳。可最近一批一线实践给出的信号越来越一致：很多生产问题根本不是“不会写提示词”，而是“没有设计上下文怎么进来、怎么退出、什么该保留、什么该丢掉”。

Anthropic 把这件事直接命名为 context engineering；Sourcegraph 进一步把它定义成“为每次推理精心设计模型真正看到的那一小块世界”；LangChain 也把常见做法总结成四个动作：写入、选择、压缩、隔离。换句话说，Prompt 只是门口那张说明书，真正决定 Agent 稳不稳的，是它背后那条上下文流水线。

一句话：下一阶段更有价值的 Agent，不是谁更会“说”，而是谁更会“喂”。

3. 核心概念  
第一，上下文不只是提示词。系统指令、检索结果、历史消息、工具定义、长期记忆、输出格式，都会一起占用模型的注意力预算。  

第二，更多上下文不等于更好。上下文一长，模型容易“看见但抓不住”，出现信息淹没、无关内容干扰、关键信息埋在中间等问题。  

第三，要把“预先全塞进去”改成“按需加载”。Anthropic 提到，很多有效 Agent 会先给模型轻量索引，再让它通过文件路径、查询或工具在运行时按需取数，而不是一上来把整份资料塞满窗口。  

第四，工具本身也是上下文负担。工具太多、描述太像、返回太肥，Agent 还没开始干活，注意力就先被耗掉。Anthropic 甚至展示过：把 MCP 工具改成在代码环境里按需调用，某些场景可把相关 token 消耗从 15 万降到约 2000。  

第五，长任务要靠压缩和外置记忆。旧消息要总结，原始工具结果要清掉，关键决策要写成 note 或进度文件；否则会话越跑越重，后面每一步都在背历史包袱。

4. 可复用方法  
如果你想马上给现有流程补一层“上下文工程”，可以直接用这个五步法：  

1）先盘点当前 Agent 每轮到底看了什么：提示词、检索块、工具说明、历史消息分别有多少。  
2）把上下文分成三类：必须常驻、按需读取、可总结归档。不要默认全部常驻。  
3）给检索加二次筛选：先粗搜，再重排，只让最相关的 3—5 份材料进入模型。  
4）给高噪音结果加隔离层：大表格、长网页、日志、转录文本先在代码/执行环境里清洗，再把摘要或差异返回模型。  
5）每跑到关键节点就留“外置记忆”：结论、未决问题、下一步，不要让模型只靠聊天记录记住一切。

5. GenGrowth 可以怎么用  
- 研究报告：不要把所有原文和链接一次性丢给模型，先做“候选来源池—相关性筛选—结构化笔记—成稿”四段式。  
- AI Builder 日报：把 builder 观点、产品更新、GenGrowth 机会点分层，先筛主题，再写正文，减少“信息很多但没有主线”。  
- 增长自动化：线索跟进时，把 CRM 字段、历史互动、行业背景、邮件草稿模板拆开供给，避免模型把旧沟通和新判断混成一团。  
- 知识库沉淀：不要只存全文，要额外沉淀“可复用摘要、标签、决策点、下一步动作”，让以后检索进来的不是一堆原料，而是可直接工作的上下文。

6. 今日行动  
今天就挑 1 条正在跑的 Agent 流程，做一次“上下文体检”：列出它每轮看到的 10 类信息，删掉 30% 最不必要的内容，再补上 1 个外置 note 文件和 1 个按需检索步骤。先别急着换模型，很多时候先瘦身上下文，效果就会明显变稳。
