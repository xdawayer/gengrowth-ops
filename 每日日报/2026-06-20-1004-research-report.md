今天读什么｜别把知识库只当搜索框，要给 Agent 一套“会读、会写、会忘”的记忆层

1. 适合谁读 / 预计阅读时间  
适合正在做 AI Builder、智能体工作流、内容生产、知识库沉淀、增长自动化的人。尤其适合已经给 Agent 接了文档、Slack、网页和表格，却发现它“这次记得、下次又忘”的团队。预计阅读时间：8—10 分钟。

2. 为什么值得读  
过去很多团队说“我们已经有知识库了”，其实指的是：文档能搜、FAQ 能查、向量库能召回。这样当然有用，但它更像给 Agent 配了一间图书馆，不等于给它长了记性。

最近这一层开始明显变化。Anthropic 在 6 月把 memory tool 和 context editing 一起推出，重点不是让模型多背一点，而是让它把重要信息写到上下文窗口之外，之后按需再读回来。LangChain 也把“写入、选择、压缩、隔离”列成上下文工程的四个核心动作，明确把 scratchpad、长期记忆、文件系统都当成正式架构。Red Hat 则进一步强调：真正可用的 Agent，不只是会检索企业资料，还要有会话记忆、长期记忆、写回机制，以及清理陈旧记忆的后台流程。

一句话：下一阶段更稳的 Agent，不是谁家知识库更大，而是谁能让 Agent 在合适的时候记住、在需要的时候取回、在过时的时候忘掉。

3. 核心概念  
第一，记忆不等于检索。检索解决“外部资料在哪里”，记忆解决“这件事我之前做过什么、用户偏好是什么、这条规则现在还算不算数”。  
第二，记忆至少分三层：当前任务里的临时笔记，跨会话保留的稳定事实，以及从多次经验里抽出来的流程规则。不要把它们全混成一团聊天记录。  
第三，真正关键的是写回。没有 write-back，Agent 只是每次重新搜索；有了写回，它才可能把调研结论、品牌偏好、项目约束、失败原因沉淀下来。  
第四，记忆必须按需读取，而不是每次全量塞回上下文；否则只会把旧噪音重新喂给模型。  
第五，记忆也会变坏。过时信息、错误总结、权限不当共享，都会把“会记事”变成“带着偏见反复犯错”。所以好的记忆层一定带来源、作用域、更新时间和淘汰机制。

4. 可复用方法  
如果你想给现有流程补一层“可写记忆”，可以直接用这五步：  
1）先区分什么该查库、什么该记住。产品文档适合检索，用户偏好、已确认规则、上次失败原因更适合记忆。  
2）给记忆分作用域：个人记忆、项目记忆、团队共享记忆，不要默认全员可见。  
3）给写回设触发点：任务完成、人工纠正、审批结论、复盘结束后，再沉淀 durable facts，而不是每轮对话都乱记。  
4）给每条记忆加最小元数据：来源、时间、适用范围、是否需要过期。  
5）定期压缩和清理：把高频重复信息提炼成规则，把过时项目和旧结论归档或删除。

5. GenGrowth 可以怎么用  
- 内容生产：把“品牌口径、栏目结构、已验证选题判断”写成长期记忆，减少每次重讲。  
- 研究助手：把一次调研里的关键结论、已排除方向、可信来源名单写回，避免隔天重查。  
- 增长自动化：把客户偏好、线索阶段、禁止触达对象、上次跟进结果沉淀成可控记忆，而不是散在聊天里。  
- 知识库沉淀：把 wiki / gbrain 从“只读资料池”升级成“检索 + 记忆写回”的双层系统：公开事实进知识库，任务经验进记忆层。

6. 今日行动  
今天就挑一个高频流程，补一张最小记忆表：  
- 哪些信息只在本次任务里临时保存？  
- 哪些信息要跨会话保留 7 天、30 天或长期？  
- 哪些信息必须人工确认后才能写回？  
- 哪些旧记忆如果继续存在，反而会害 Agent 做错？  
先把“该记什么、谁能看、何时失效”定清楚，再谈更大的知识库和更多自动化。
