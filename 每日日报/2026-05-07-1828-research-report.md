# 今天读什么｜Anthropic：别急着做 Agent，先把 Workflow 做对

> 同步来源：Slack `#gengrowth-research-reports`；发送时间：2026-05-07 18:28（UTC+8）。

标题：今天读什么｜Anthropic：别急着做 Agent，先把 Workflow 做对

试推重发说明：上一版偏“链接摘要”，读者需要点进原文才能获得主要内容；这版改成“可直接在 Slack 阅读”的中文学习文档。下面正文就是完整可读内容，链接只放在最后作为来源。

开场：

这篇 Anthropic 的《Building effective agents》值得读，是因为它没有追逐“Agent 很酷”的叙事，而是在回答一个更实际的问题：到底什么时候应该做 Agent，什么时候不该做？

它的核心提醒很朴素：很多真正好用的 AI 系统，不是靠复杂框架堆出来的，而是靠简单、可组合、可观察的模式一步步做出来的。对产品、运营、内容、销售、增长同学来说，这篇文章尤其有价值，因为它能帮我们少走很多“为了 Agent 而 Agent”的弯路。

一、这篇文章真正想解决什么问题

现在很多人一看到 Agent 火，就会自然地产生一个冲动：是不是所有流程都应该改造成 Agent？是不是产品里不加一个“智能体”，就显得不够 AI？

Anthropic 的经验刚好相反。

他们总结了大量团队构建 LLM agents 的实践后发现：真正成功、稳定、可维护的系统，通常不是最复杂的系统，而是更简单、更透明、更容易调试的系统。

换句话说，AI 产品的目标不是“看起来像 Agent”，而是“把任务可靠地完成”。

很多场景里，一个优化过的 LLM 调用，再配上检索、少量上下文示例、清晰的提示词，就已经够用了。你不一定需要让模型自己规划一切，也不一定需要把几十个工具都塞给它。

文章真正想解决的是一个工程和产品判断问题：

我们什么时候应该增加 agentic complexity，也就是让系统更像一个可以自主决策的 Agent？什么时候应该保持简单，用固定流程、明确步骤、可控工具，把结果做好？

Anthropic 的建议是：不要默认上 Agent。先从最简单有效的 LLM 方案开始，衡量效果，再决定是否增加复杂度。

这句话看起来保守，但其实非常务实。因为 Agentic systems 往往会用更高的延迟、更高的成本、更难的调试，去换取更好的灵活性或更强的任务表现。这个交换是否值得，必须看具体任务，而不是看概念热度。

二、先分清两个概念：Workflow 和 Agent

这篇文章里，一个很重要的区分是：Workflow 和 Agent 不是一回事。

Workflow，可以理解成“由代码预先定义好的流程”。

生活化一点说，workflow 像一份固定 SOP。比如你去办签证，材料检查、照片审核、缴费、预约、递交，每一步都很明确。你不希望办理人员每天临时发挥，你希望流程稳定、可预测、少出错。

在 AI 系统里，workflow 也是这样：开发者先设计好任务路径，模型在其中完成某些固定步骤。比如：

- 第一步：读取用户需求
- 第二步：提取关键信息
- 第三步：查找相关资料
- 第四步：生成初稿
- 第五步：检查是否符合格式
- 第六步：输出结果

模型可以参与每一步，但整体路线是代码规定好的。

Workflow 适合什么任务？

适合步骤明确、路径可预测、稳定性重要的任务。

例如：

- 把销售线索整理成表格
- 根据固定模板生成文章初稿
- 对客户反馈做分类
- 把会议纪要转成待办事项
- 按固定标准检查内容是否合规
- 从资料中提取字段并写入 CRM

这些任务不需要模型“自由探索世界”，而是需要它在清晰流程中稳定工作。

Agent，则是另一种模式。

Agent 可以理解成：你给一个聪明实习生一个目标和一组工具，让他自己决定步骤。

比如你对实习生说：“帮我研究一下某个行业最近有哪些潜在客户，并找出最值得联系的 20 家公司。”你不会完全规定他每一步怎么做。他可能先查行业报告，再看新闻，再搜公司官网，再判断公司规模，再整理名单。中间如果发现信息不足，他还会调整路径。

在 AI 系统里，Agent 就是由 LLM 动态决定流程和工具使用。它不是严格按预设路线走，而是根据当前情况判断下一步做什么。

Agent 适合什么任务？

适合开放式任务、步骤数量未知、需要灵活决策的任务。

例如：

- 研究一个新市场并形成判断
- 根据复杂目标自动搜集、筛选、整合资料
- 处理路径不确定的客户调研
- 多轮排查一个系统问题
- 在多个工具之间来回切换完成任务
- 根据中间结果不断调整策略

简单说：

Workflow 像固定 SOP，稳定、可控、便于调试。

Agent 像聪明实习生，灵活、主动、能处理未知，但也更难管理。

三、这篇文章的核心判断

1. 观点：不要默认上 Agent，先从最简单有效的方案开始。

解释：很多应用并不需要复杂 Agent。一个好的 LLM 调用，加上清楚的任务说明、必要的上下文、少量示例，往往已经能解决 70% 甚至 80% 的问题。复杂系统不是免费午餐，它会带来更多延迟、成本和调试难度。

对普通使用者/产品人的启发：做 AI 工具时，不要先问“能不能做成 Agent”，而要先问“最简单的可用方案是什么”。如果一个固定流程能解决问题，就不要急着引入自主决策。

2. 观点：Workflow 和 Agent 的选择，本质是“可预测性”和“开放性”的权衡。

解释：如果任务路径明确，workflow 通常更好；如果任务路径不确定，Agent 才更有价值。把明确任务做成 Agent，可能反而让系统变得不稳定；把开放任务硬塞进固定 workflow，也可能让系统不够灵活。

对普通使用者/产品人的启发：判断一个任务适合哪种 AI 形态，可以先问三个问题：步骤是否固定？结果标准是否明确？中途是否需要根据新信息改变方向？如果前两个答案是“是”，多半先用 workflow；如果第三个答案是“是”，才考虑 Agent。

3. 观点：成功的系统通常简单、可组合、透明，而不是复杂黑箱。

解释：Anthropic 特别强调，很多有效模式其实几行代码就能实现。框架可以帮你快速原型，比如调用模型、定义工具、解析工具调用、串联流程。但框架也可能遮蔽 prompt 和 response，让你看不清模型到底为什么失败。

对普通使用者/产品人的启发：如果你在做 AI 产品，透明性很重要。你要能看到模型输入了什么、输出了什么、调用了什么工具、在哪一步出错。否则系统看起来很高级，但一旦效果不稳定，就很难修。

4. 观点：Augmented LLM 是基础构件，不是越多工具越好。

解释：文章提到的 augmented LLM，可以理解成 LLM 加上 retrieval、tools、memory。也就是模型不仅能生成文本，还能查资料、用工具、记住某些上下文。但关键不是把工具堆满，而是给模型清晰、文档化、适合任务的工具接口。

对普通使用者/产品人的启发：工具不是越多越智能。真正重要的是：工具名字是否清楚？输入输出是否明确？失败时模型是否知道怎么办？边界是否写明？如果工具接口含糊，模型就会像拿到一堆说明书混乱的设备，越用越乱。

5. 观点：Prompt chaining 是非常实用的常见模式。

解释：Prompt chaining 指的是把一个大任务拆成固定步骤，每一步都让模型做更简单的事，中间加检查门槛。比如先让模型提取信息，再让它分类，再让它生成总结，最后让它自检格式。这样做的代价是延迟更高，但好处是准确率和可控性通常更好。

对普通使用者/产品人的启发：不要总想让模型“一口气完成所有事”。很多时候，把任务拆小，效果会明显提升。尤其是内容生产、线索整理、研究分析这类任务，拆步骤比堆一个超长 prompt 更可靠。

四、如果你要做一个 AI 工具，应该怎么落地

可以按下面 5 步走：

1. 单次 LLM 调用先跑通

先不要做复杂系统。把任务写成一个清楚的 prompt，让模型直接完成一次。

例如你要做“客户调研摘要”，第一版可以只输入客户官网介绍、新闻、销售备注，让模型输出一段结构化摘要。

目标不是一开始就完美，而是确认：模型是否理解任务？输出是否有基本价值？失败主要发生在哪里？

2. 加检索或上下文示例

如果模型缺信息，就加 retrieval；如果模型风格不稳定，就加少量高质量示例。

很多问题不是模型不够聪明，而是上下文不够。比如让模型写客户画像，如果没有行业、客户背景、历史互动，它只能泛泛而谈。加上检索和示例后，单次 LLM 调用可能就已经非常好用。

3. 拆 workflow

当任务开始变复杂，不要立刻变 Agent，先拆 workflow。

比如一个“生成行业研究简报”的流程可以拆成：

- 收集资料
- 提取事实
- 按主题归类
- 生成初稿
- 检查是否有无来源判断
- 改写成适合 Slack 阅读的版本

每一步都清楚，失败也容易定位。你会知道问题是资料不好、提取错了、分类不准，还是最后表达不行。

4. 给工具加清晰接口和边界

如果模型需要调用工具，比如搜索、查数据库、写入表格、发 Slack、建任务，一定要把工具设计清楚。

好工具接口应该让模型知道：

- 这个工具是做什么的
- 什么时候该用
- 输入需要哪些字段
- 输出代表什么
- 失败时怎么办
- 不应该用它做什么

这对 Agent 尤其重要。Agent 的能力很大程度取决于工具是否清楚。如果工具定义模糊，就像让实习生用一堆没标签的按钮做事，出错是必然的。

5. 只有任务开放且需要动态决策时，再升级为 Agent

最后再问：这个任务真的需要模型自己决定流程吗？

如果任务步骤固定，workflow 更稳。

如果任务中途需要不断判断，比如资料不够要继续搜，发现方向错了要改策略，需要在多个工具之间来回选择，那么可以考虑 Agent。

也就是说，Agent 应该是为了任务需要而出现，不是为了产品包装而出现。

五、对 GenGrowth / 彪哥的启发

1. 内容生产更适合先做 workflow，而不是全自动 Agent。

比如每日学习文档、行业观察、客户案例拆解，完全可以先做固定流程：选题、提取要点、生成中文稿、按 Slack 阅读体验改写、加来源、入库去重。这个流程明确，稳定性比“让 Agent 自由写一篇”更重要。

2. 线索研究可以分层：基础整理用 workflow，深度判断再用 Agent。

例如从公开资料中提取公司名称、行业、融资、职位、官网、联系方式，这些适合 workflow。  
但如果要判断“这家公司为什么现在可能需要 AI 增长方案”，可能就需要更开放的推理和多步搜索，这时可以引入 Agent。

3. 客户调研不要一开始追求全自动，先把 SOP 写清楚。

例如调研一个潜在客户，可以先固定为：公司概况、产品/服务、目标用户、近期动态、增长渠道、可能痛点、可切入话术。  
只要这个 SOP 稳定，AI 就可以持续辅助，而不是每次从零发挥。

4. 自动化流程要优先追求可观察。

对于发 Slack、写 wiki、入 gbrain、生成日报这类流程，最重要的是知道每一步是否成功、失败在哪里、是否重复入库。透明日志和去重机制，比“Agent 自己决定要不要发”更关键。

5. GenGrowth 的 AI 能力可以从“工具箱”而不是“大脑崇拜”开始建设。

不要急着打造一个无所不能的 Agent。更现实的路径是：先沉淀一批高质量 workflow，再把这些 workflow 变成可组合工具。等工具足够清楚、数据足够稳定，再让 Agent 在上层调度。

六、今天可以马上做的一个小练习

15 分钟练习：选一个你每周都会重复做的任务，判断它更适合 workflow 还是 Agent，并写出 SOP。

可以按这个模板来：

任务名称：  
例如：每周整理 10 个潜在客户并写初步触达建议。

第一步：判断任务类型

问自己三个问题：

1. 这个任务的步骤是否基本固定？  
2. 输出格式是否比较明确？  
3. 中途是否经常需要根据新发现改变方向？

如果 1 和 2 是“是”，大概率先做 workflow。  
如果 3 是“是”，并且变化很多，才考虑 Agent。

第二步：写出最小 SOP

例如：

1. 输入客户名单  
2. 抓取官网和公开资料  
3. 提取公司简介、行业、目标用户、近期动态  
4. 判断可能痛点  
5. 生成 3 条触达角度  
6. 输出为表格  
7. 人工检查后再发送

第三步：标出哪些步骤需要 LLM

不是每一步都要模型做。  
比如抓取资料可能是工具，去重可能是代码，判断痛点和生成话术才需要 LLM。

第四步：标出风险点

例如：

- 资料来源不可靠
- 模型编造客户痛点
- 触达话术太泛
- 输出格式不稳定

第五步：决定下一步

如果 SOP 很清楚，就先做 workflow。  
如果你发现每次都要根据不同客户大幅调整研究路径，再考虑升级成 Agent。

这个练习的价值在于：它会逼你先把任务讲清楚。很多 AI 自动化失败，不是模型能力不足，而是人类自己也没有定义清楚流程。

七、一句话总结

别急着做一个会“自由发挥”的 Agent，先做一个步骤清楚、结果稳定、出了问题能调试的 Workflow。
