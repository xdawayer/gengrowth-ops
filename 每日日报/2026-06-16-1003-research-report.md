今天读什么｜别让 Agent 把一切都背在聊天里，要把状态拆成三层

1. 适合谁读 / 预计阅读时间  
适合正在做 AI Builder、智能体工作流、内容生产自动化、研究助手、增长运营自动化的人读。尤其适合已经遇到这些问题的团队：Agent 一跑久就忘事、流程中断后要重来、文件产物散在聊天里、明明只是等审批却一直占着会话。预计阅读时间：8—10 分钟。

2. 为什么值得读  
很多团队现在已经不缺模型，也不缺工具，真正开始拖后腿的，是“状态怎么保存”。一旦任务从一句问答变成几十步工作，问题就会同时冒出来：上下文越聊越长、文件越堆越乱、运行一中断就丢进度、换一个 Agent 接手又要重新解释。

最近几家一线团队给出的方向很一致。OpenAI 在新版 Agents SDK 里，明确把状态拆开：运行状态、沙箱会话、工作区快照，不再默认“全塞进一段对话”。Anthropic 进一步强调，上下文是有限注意力，不是越长越好；长任务要靠压缩、笔记、交接，而不是硬扛。Vercel 则把 durable execution 做成工作流底座：步骤、暂停、重试、恢复都进入事件日志，而不是靠人猜“刚才跑到哪一步了”。

一句话：下一阶段更稳的 Agent，不是记性更好，而是更会把记忆从聊天里搬出来，变成可恢复、可交接、可审计的状态系统。

3. 核心概念  
第一，思考状态，不等于工作状态。模型当前要看的说明、最近几轮消息，属于“思考层”；已经生成的文件、表格、草稿、截图，属于“工作区层”；流程走到哪一步、是否待审批、能否重试，属于“运行层”。三层混在一起，系统就会越来越脆。

第二，长任务最怕的不是停，而是停了以后接不上。Anthropic 提到，长流程里与其让同一个 Agent 一直背着越来越长的历史，不如适时做上下文压缩，甚至直接重开一个干净上下文，再靠结构化交接文件把状态接上。

第三，文件和产物应当住在工作区，不该住在聊天气泡里。OpenAI 现在把 snapshot、session state、RunState 分开，本质上就是承认：可执行任务需要自己的桌面、抽屉和进度条，不能只靠聊天记录当办公室。

第四，真正该评估的，往往是最终状态，不是中间路径。一个 Agent 可能不是按你预想的顺序完成任务，但只要最后文件正确、审批完整、状态一致，它就是有效的。

4. 可复用方法  
如果你想给现有 Agent 补一层“状态架构”，可以直接用这五步：  
1）先把任务拆成三栏：模型当下要看什么、工作区要存什么、流程引擎要记什么。  
2）每完成一个关键动作，就强制留下产物或检查点，比如 `task.md`、`progress.md`、输出目录、审批结果，而不是只留一句“已完成”。  
3）凡是等待人、等待回调、等待时间到的步骤，都改成暂停—恢复，不要让会话空转。  
4）对长任务设“压缩点”或“交接点”：到一定长度后生成摘要、待办和风险，再用新上下文继续。  
5）验收时优先看最终状态：文件是否齐、记录是否对、下一步是否明确，而不只看对话过程像不像人。

5. GenGrowth 可以怎么用  
- 研究报告流：把“选题、资料、提纲、初稿、定稿”拆成独立状态，每步都有文件产物，第二天接着跑也不会断。  
- 内容生产流：让 Agent 生成渠道草稿、配图需求、发布时间建议，但把审批结果和最终发布版本单独存档。  
- 多 Bot 协作：Hermes、PM、Ops 之间不要靠转述聊天接力，而要靠任务卡、结构化摘要和明确状态字段交接。  
- AI Builder 场景：原型、代码、测试结果、评审意见分层保存，避免“页面能打开，但没人说得清现在到底算第几版”。

6. 今日行动  
今天就挑一条你们最常跑、最容易中断的流程，补三个东西：  
- 一个任务说明文件：目标、边界、验收标准。  
- 一个进度文件：当前状态、已完成、待处理、风险。  
- 一个产物目录：所有草稿、截图、表格、输出都放进去。  

如果这三样补上后，换一个 Agent 或隔一天还能继续推进，说明你的流程开始从“会聊天”升级成“会工作”了。
