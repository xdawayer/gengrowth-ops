今天读什么｜别只看 Agent 的结果，要看它“到底是怎么走到这里的”

**1. 适合谁读 / 预计阅读时间**  
适合正在做 AI Builder、Agent 工作流、内容生产自动化、知识库检索、多 Bot 协作、增长执行系统的人读。  
预计阅读时间：8–10 分钟。

**2. 为什么值得读**  
很多团队做 Agent，最先上线的是“能跑”，最后卡住的却是“出了问题不知道为什么”。  
传统软件出错，通常会报异常、打日志、给栈信息；但 Agent 更常见的失败方式不是报错，而是**表面顺利执行，最后给出一个看起来很像样、实际上跑偏的结果**。

最近公开资料在这件事上越来越一致。OpenAI 在新一代 Agents SDK 里，直接把 tracing 和 observability 做成官方能力；Anthropic 讲他们做多 Agent 研究系统时，也明确提到：用户会说“明明应该找到的信息却没找到”，如果没有完整执行轨迹，你根本不知道问题出在搜索词、来源选择、工具失败，还是上游 agent 交接错了上下文。  
这说明一件事：**Agent 进入生产后，最大的门槛之一，不再是“会不会调用工具”，而是“你能不能看清它每一步到底做了什么”。**

对 GenGrowth 特别重要。因为我们很多流程天然是多步的：找资料、筛来源、提炼框架、生成初稿、复核、同步知识库、安排下一步动作。只看最终输出，等于只看考试分数，不看答题过程；一旦结果不稳，根本没法迭代。

**3. 核心概念**  
第一，**监控不等于可观测**。  
监控回答的是“系统现在健不健康”，比如延迟、成功率、成本；可观测回答的是“这一次为什么会这样”。Agent 场景里，后者往往更关键。

第二，**Agent 需要的不是普通日志，而是执行轨迹**。  
一条完整轨迹里，至少要能看到：调用了哪个模型、用了哪个工具、传了什么参数、拿回了什么结果、在哪一步重试、哪一步卡住、最后是怎么决定输出的。

第三，**多 Agent 系统最容易坏在交接处**。  
公开资料反复提到一个新问题：handoff failure。也就是 A agent 给 B agent 的任务说明不完整、上下文错了，B 继续认真干活，最后把错误放大。  
所以真正要记录的，不只是“谁调用了谁”，而是**交接时到底传了什么**。

第四，**可观测性的最小单位不是整段对话，而是 step/span**。  
也就是把每一步作为结构化记录保存下来：输入、输出、耗时、错误类型、父子关系。这样你才能按工具名、错误类、任务类型去查，不用在海量日志里手动翻。

第五，**只看轨迹还不够，要把失败沉淀成评测**。  
LangChain 和 Braintrust 这类实践都在强调同一件事：线上轨迹发现问题后，最好一键转成回归测试样本。否则你今天修好了，明天换个提示词、换个工具版本，又会复发。

**4. 可复用方法**  
如果你想给现有 Agent 系统补上一层“飞行记录仪”，可以先做这个最小版四步法：

**第 1 步：先记工具调用，不求一步到位。**  
先把每次 tool call 记下来：工具名、输入摘要、输出摘要、耗时、是否报错。光这一步，就能抓住大量“看似正常、实际乱调工具”的问题。

**第 2 步：给交接加一张“任务封面”。**  
凡是主 Agent 把活交给子 Agent，都固定记录 3 个字段：目标、约束、期望产物。以后出了问题，先看交接封面是不是漏了关键信息。

**第 3 步：给失败分类型。**  
不要都算“回答不好”。至少分成：工具参数错、检索结果差、上下文丢失、重复循环、交接漏信息、审批/权限被拦。只有分类，才有改进方向。

**第 4 步：把真实失败转成小样本评测集。**  
每周挑 10 个真实翻车案例，变成固定测试题。后面改 prompt、改工具说明、改路由逻辑，都先跑这一组，避免“修一个，坏一片”。

**5. GenGrowth 可以怎么用**  
第一，用在**研究报告与内容生产链路**。  
比如“选题 → 搜资料 → 提炼结构 → 成文”这条链路里，最容易出问题的不是写作，而是前面的来源筛选和信息传递。把每一步轨迹存下来，就能知道文章跑偏到底是因为资料源差，还是摘要阶段漏了重点。

第二，用在**多 Bot 协作**。  
Hermes、PM、Ops、HR 这类角色型 Bot，如果后面越来越多走任务板和异步协作，最该记录的就是任务交接摘要、阻塞原因、回写结果，而不是只保留最后一句“已完成”。

第三，用在**增长自动化工作流**。  
像“抓公开信号 → 分类 → 生成建议 → 写入任务/知识库”这种流程，只要轨迹清楚，就能很快看出问题是出在分类规则、模型判断，还是下游写入动作。

第四，用在**隐私友好的系统调试**。  
Anthropic 的做法很值得学：不是一定要保存全部对话原文，很多时候记录高层决策结构、工具路径、交互模式，就已经足够定位问题，同时还能减少隐私风险。

**6. 今日行动**  
今天就做三件小事：  
1）给你最常用的一个 Agent 流程，加上“工具名 + 输入摘要 + 输出摘要 + 耗时”的记录；  
2）如果有多 Agent 交接，固定一份最小 handoff 模板：目标、约束、产物；  
3）把最近一次明显翻车案例，整理成一条回归测试样本。  

先别追求大而全。**能复盘一次真实失败的 Agent 系统，才开始具备长期进化能力。**
