今天读什么｜别把每个风险都做成审批弹窗，要让 Agent 学会“不确定时先问”

**1. 适合谁读 / 预计阅读时间**  
适合正在做 AI Builder、智能体工作流、内容生产、线索跟进、知识库整理、团队自动化的人读。尤其适合已经给 Agent 接了不少工具，也加了审批，但仍然遇到“它要么太莽、要么太烦、要么老停在奇怪地方”的团队。  
预计阅读时间：8—10 分钟。

**2. 为什么值得读**  
很多团队给 Agent 上线后的第一反应，是一路加审批：发消息要批、改记录要批、发布内容要批。这样当然更安全，但很快会出现两个问题：  
第一，人会被一堆低价值确认弹窗拖垮；第二，真正危险的地方，Agent 还是可能因为“理解错意图”而做错。

最近几条公开实践把更成熟的方向讲清楚了。Anthropic 在“Trustworthy agents”和“Measuring AI agent autonomy in practice”里都强调：真正稳的 Agent，不只是外面有审批，更重要的是它自己能识别不确定，知道什么时候该停、该问、该把决定交还给人。一个很有意思的数据是：在复杂任务里，Claude 主动停下来问澄清问题的频率，增长得比人类中断它还快；在最复杂任务上，它主动澄清的次数甚至明显高于人工打断。  
OpenAI 在 Deep Research 的多智能体示例里，也不是让研究代理拿到问题就直接开跑，而是先经过一个 triage，再决定要不要进入 clarifier（澄清提问），之后才把整理好的研究指令交给 research agent。  
Vercel 则把这件事落到了产品工作流里：遇到边界模糊、需要拍板的情况，不是让 Agent 硬猜，而是暂停、持久化状态、发到 Slack 给人判断，再从原地恢复。

一句话：**下一阶段更好用的 Agent，不只是“会执行”，而是“会在不确定时先问清楚”。**

**3. 核心概念**  
第一，**澄清不等于审批**。  
审批是“这件事风险高，所以必须过人”；澄清是“这件事信息不全或偏好不明，所以先别猜”。两者解决的是不同问题。

第二，**不确定性本身要成为产品信号**。  
不要把 Agent 设计成永远往前冲。它遇到缺少上下文、用户偏好不明确、规则冲突、结果不可逆时，应该能主动暴露“不确定”。

第三，**成熟用户的监督方式，不是步步点头，而是随时打断**。  
Anthropic 的观察很有启发：经验越多的用户，越倾向于给 Agent 更大自主权，但也越会在关键时刻介入。说明好的监督不是更密的按钮，而是更强的可见性和刹车能力。

第四，**澄清问题要短、准、少**。  
OpenAI 的做法是一次只问 2—3 个真正必要的问题，不重复问用户已经说过的事。问得太散，体验会崩；问得太晚，代价会变高。

第五，**暂停之后必须能恢复**。  
如果 Agent 一停就丢状态，那澄清和人工介入会把流程弄得更碎。Vercel 的 durable workflow 思路很关键：暂停、存状态、等人回复、再续跑，而不是整条流程重来。

**4. 可复用方法**  
如果你想给现有 Agent 补一层“先问清楚再执行”的能力，可以直接用这五步：

1）先把流程里的不确定点分三类：  
- **事实缺口**：缺字段、缺资料、缺权限；  
- **偏好缺口**：语气、优先级、取舍标准没说清；  
- **高风险动作**：外发、删除、付费、改生产。  

2）给三类缺口分别配动作：  
- 事实缺口：优先自己查；  
- 偏好缺口：先问用户；  
- 高风险动作：直接进审批或人工确认。  

3）把澄清问题压到最小。  
每次只问最影响结果的 1—3 个问题，最好让用户用选项或简短句就能答。

4）把“暂停—恢复”做成正式节点。  
别让 Agent 说一句“等你回复”就散了；要把当前进度、待确认项、下一步动作留住，回复后从原地继续。

5）记录每次停下来的原因。  
一周后回看，你会发现哪些问题该补字段、哪些该改默认值、哪些该前置规则，而不是一遍遍问同样的问题。

**5. GenGrowth 可以怎么用**  
- **线索跟进**：Agent 可以先自己补公司信息、行业背景；但如果“到底推咨询、课程还是工具服务”取决于主观判断，就先抛一个澄清问题，而不是直接写死话术。  
- **内容生产**：选题、资料收集、初稿整理可以自动跑；但涉及立场、品牌语气、是否对外发布时，先问“这篇偏方法论、案例拆解还是观点表达”。  
- **知识库沉淀**：Agent 可先自动分类、摘要、提标签；若一条内容横跨多个主题且会影响后续检索结构，就先让人拍板归档口径。  
- **多 Bot 协作**：Hermes/PM/Ops 之间传任务时，把“需要谁决策”单独标出来，避免低权限角色在意图不清时继续硬推进。

**6. 今日行动**  
今天就挑一条你们已经在跑的 Agent 流程，补上一个最小版本的“澄清层”：

- 先列出 5 个最常见的“它做错其实不是能力不够，而是没问清楚”的场景；  
- 从里面选 1 个高频场景，插入一个 `clarify` 节点，只允许问最多 3 个问题；  
- 连续观察 20 次运行，统计：它停下来的原因、减少了哪些返工、哪些问题其实该改成默认规则而不是继续问人。
