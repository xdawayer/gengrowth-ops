今天读什么｜别让 Agent 只交一段话，先给它一份“输出合同”

1. 适合谁读 / 预计阅读时间  
适合正在做 AI Builder、智能体工作流、内容生产、线索跟进、知识库沉淀的人读。尤其适合已经发现 AI“看起来会做事”，但一接到表格、审批、CRM、文档归档就开始不稳定的团队。预计阅读时间：8—10 分钟。

2. 为什么值得读  
很多 AI 项目卡住，不是因为模型不会写，而是因为它每次“交付的东西”都不一样。今天返回一段话，明天多两个字段，后天又漏掉关键信息。人读着像差不多，系统却接不住。

最近几条公开实践把这个问题讲得很清楚。OpenAI 把 Structured Outputs 做成正式能力，核心就是让模型按你定义的结构返回结果，不再靠“希望它听话”。Anthropic 也明确提醒：如果你需要稳定结构，应该优先用结构化输出，而不是不断补提示词；同时，工具和输出格式要尽量接近模型熟悉的自然形式，减少额外格式负担。Vercel 的 Lead Agent 模板则把这个思路落到真实业务：先用结构化结果给线索分类，再进入研究、邮件草稿和 Slack 审批。OpenAI 的 SchemaFlow 示例更进一步，把“需求解析—影响分析—计划—执行稿”拆成多段 typed contract，哪一段错了就查哪一段。

一句话：更稳的 AI 工作流，不是让模型多说一点，而是让它先按合同交付，再进入下一步。

3. 核心概念  
第一，输出合同就是“机器能验收的交付标准”。不是笼统地说“给我一版总结”，而是明确：必须有哪些字段、哪些是必填、哪些只能从枚举里选、哪些情况必须返回 unknown。

第二，合同的价值不只是规范格式，更是切断歧义。自由文本适合阅读，结构化结果适合流转。真正进系统的，常常不是“漂亮的一段话”，而是标题、状态、优先级、负责人、风险、下一步这些可消费字段。

第三，长流程不要只要一个大结果，要有分阶段合同。先抽取，再分类，再计划，再生成动作。这样哪一步出错都能定位，而不是把所有问题糊成一句“结果不太对”。

第四，合同要对模型友好。能用简单对象就别塞复杂转义；能把正文和元数据分开，就别要求它把整篇长文硬塞进 JSON 字符串。

第五，合同不是为了取代内容，而是为了托底内容。外层用结构保稳定，内层再生成自然语言，系统就既能自动流转，也保留可读性。

4. 可复用方法  
你可以直接套这个五步法：  
1）先找下游接收者：下一个环节到底是人、表格、审批卡片，还是另一个 Agent。  
2）只定义最小必要字段：例如 `结论 / 证据 / 风险 / 下一步`。  
3）把开放任务拆成阶段合同：`信息抽取`、`判断分类`、`行动计划` 分开返回。  
4）给关键字段加校验：必填、枚举、日期格式、URL、字数范围，都可以程序检查。  
5）为失败预留回路：校验不过就修复、降级到人工、或返回“信息不足”，不要硬往下流。

5. GenGrowth 可以怎么用  
- 研究报告：先让 Agent 输出“选题判断卡”，字段固定为值得读原因、核心概念、可复用方法、GenGrowth 落地点、参考来源；结构对了，再生成最终中文正文。  
- 增长线索：表单进来后，先结构化判断线索类型、行业、优先级、建议动作，再决定要不要进入深度研究和外联草稿。  
- 内容生产：把“一篇文章”拆成选题卡、大纲卡、渠道改写卡、发布检查卡，减少一次生成到底带来的返工。  
- 知识库沉淀：每次沉淀默认带 `summary / topics / entities / related / action`，以后检索和复用会更稳。

6. 今日行动  
今天就挑一个你最常重复的流程，补一张最小输出合同：  
- 做内容，先定义“大纲卡”字段；  
- 做线索，先定义“资格判断卡”字段；  
- 做研究，先定义“来源判断卡”字段。  
目标不是一次做全，而是先让下一个环节不再猜前一个环节到底交了什么。
