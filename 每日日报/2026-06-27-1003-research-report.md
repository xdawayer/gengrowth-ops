今天读什么｜别让 Agent 一直轮询，给它一套“事件唤醒”机制

**1. 适合谁读 / 预计阅读时间**  
适合正在做 AI Builder、Slack/飞书机器人、审批流、内容生产自动化、研究助手、增长自动化的人读。尤其适合已经把 Agent 接进真实流程，但还在用“每隔几秒查一次状态”“等人回复时一直挂着”的团队。  
预计阅读时间：10 分钟。

**2. 为什么值得读**  
很多团队做 Agent，第一阶段解决的是“它能不能做”；第二阶段真正拖后腿的，常常是“它等的时候怎么办”。  
比如：等人批准、等用户在线索卡里补信息、等 GitHub Actions 跑完、等 Slack 线程回复、等外部系统回调。很多系统的默认做法还是轮询：隔几秒查一次、超时再重试、失败再整段重跑。结果就是三种浪费一起出现：成本空转、状态易丢、流程难审计。

最近公开资料给了很一致的信号。OpenAI 已经把后台执行和 Webhook 作为正式能力组合起来：任务可以后台跑，完成后由事件主动通知；Vercel 进一步把“挂起—恢复”做成工作流原语，审批点击、线程回复、超时等待都能直接唤醒流程；Anthropic 也开始给 Managed Agents 提供 Webhook 事件和定时调度。  
一句话：**更成熟的 Agent，不是一直盯着世界问“好了没”，而是事情一发生，世界主动把它叫醒。**

**3. 核心概念**  
第一，**后台执行**和**事件唤醒**不是一回事。后台执行解决“不超时”，事件唤醒解决“何时继续”。只做前者，系统还是会不停查状态。  

第二，**轮询应该是兜底，不该是主路径**。真正适合事件驱动的场景，包括审批按钮、表单提交、消息回复、支付完成、任务结束、文件上传成功。能等事件，就别让 Agent 空转。  

第三，**等待点要变成显式节点**。不要让“等回复”藏在对话里，而要变成一个正式状态：等待谁、等什么事件、最长等多久、超时后怎么办。  

第四，**唤醒要带最小恢复信息**。最有用的不是把整段历史重新塞回模型，而是带上 `run_id / 当前步骤 / 事件类型 / 决策结果` 这类最小恢复包，让流程从原地继续。  

第五，**事件系统一定要有幂等、鉴权和超时**。OpenAI 的 Webhook 文档明确提醒要验签、用事件 ID 去重；Vercel 也建议给审批等待加超时分支。否则“会被叫醒”很容易变成“被重复叫醒”或“被错误的人叫醒”。

**4. 可复用方法**  
如果你想给现有 Agent 补一层最小“事件唤醒机制”，可以直接用这五步：

1）先把流程里所有“等待”列出来：等审批、等回复、等回调、等定时。  
2）给每个等待点指定事件源：按钮点击、消息回复、Webhook、定时器。  
3）为每次等待生成一个稳定关联键，比如任务 ID、线程 ID、审批单号，而不是随机散落在聊天里。  
4）给每个等待点补三件事：验签/鉴权、幂等去重、超时回退。  
5）让事件只唤醒对应步骤，不要每次都把整条链路从头再跑。

**5. GenGrowth 可以怎么用**  
- **内容生产**：日报初稿发到 Slack 后，不必每隔几分钟检查有没有反馈；有人在线程里回复，工作流再继续改稿。  
- **增长自动化**：线索进入 CRM、表单补全、预约确认这些动作，都可以变成下一步 Agent 的触发器，而不是靠定时扫表。  
- **研究整理**：深度研究任务可后台运行，完成后通过事件把结果推回频道或任务卡，不用人盯着等。  
- **多 Bot 协作**：高风险动作保留人工拍板，低风险步骤事件驱动恢复，能明显减少“卡在半路”和“重复催问”。

**6. 今日行动**  
今天就挑一条你们最常用、最容易“等人/等系统”的流程，做一个小改造：  
1）找出其中一个轮询点；  
2）把它改成“事件触发继续”；  
3）补上一个 24 小时超时和一个幂等键。  
只要先改掉一个点，你就会明显感受到：Agent 从“脚本”开始更像“同事”。
