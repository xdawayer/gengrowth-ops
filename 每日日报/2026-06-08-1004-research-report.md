今天读什么｜别让 Agent 一步步手点工具，给它一个会写小程序的执行层

1. 适合谁读 / 预计阅读时间

适合正在做 AI Builder、智能体工作流、内容生产自动化、知识整理、增长运营自动化的人读。尤其适合已经给 Agent 接了不少工具，却发现它一复杂就变慢、变贵、变脆的团队。  
预计阅读时间：8—10 分钟。

2. 为什么值得读

过去大家给 Agent 接工具，默认思路是：模型想一步、调一个工具、拿回结果，再想下一步。这个方式在 demo 阶段很好用，但一到真实流程就容易出问题：工具定义反复进上下文，中间 JSON 越堆越多，循环、重试、等待都要靠模型一轮轮“说出来”，成本和延迟一起上升。

最近几条公开信号很值得放在一起看。Anthropic 明确提出：直接工具调用会吞上下文，更高效的方式往往是让 Agent 写一小段代码去调工具。OpenAI 把原生 sandbox execution 放进 Agents SDK，说明“给 Agent 一个受控工作台”正在变成基础设施。LangChain 也把这种做法归到 context isolation：把大对象和重处理中间结果留在执行环境里，只把真正需要的结论返给模型。Vercel 的生产数据更直白：到 2026 年 4 月，带工具调用的请求占比升到 22.2%，而按 token 计，相关请求已经占到 58.9%。

一句话：**Agent 的瓶颈，越来越不是“会不会调工具”，而是“每一步都让模型亲自盯着调，值不值得”。**

3. 核心概念

第一，**工具调用不是执行架构。**  
真实系统至少要分三层：负责判断的推理层、负责跑代码和处理文件的执行层、负责权限和外部系统连接的集成层。

第二，**代码执行中间层，就是把啰嗦步骤下放。**  
如果任务里有循环、筛选、合并、清洗、重试、等待事件这些动作，让模型逐步口述很浪费；让它写成一小段程序，在 sandbox 里跑完，再把结果带回来，通常更快也更稳。

第三，**上下文隔离，比上下文加长更重要。**  
搜索结果、网页原文、表格中间态、日志片段，不一定都该塞回对话。更好的做法是：大对象留在环境里，模型只看摘要、差异、异常和最终产物。

第四，**执行环境不等于放权失控。**  
成熟做法不是让 sandbox 直接继承所有权限，而是把审批、审计、身份控制留在外层，让执行层只拿到当前任务所需的最小工具和最小凭证。

4. 可复用方法

如果你想把现有 Agent 从“工具连点器”升级成“会做事的执行者”，可以直接套这五步：

1）先找一个“中间结果很多”的流程，比如批量读网页、整理多份文档、比对 CSV、检查频道历史。  
2）把任务拆成两类：需要模型判断的，留给模型；只是计算、遍历、清洗、轮询的，下放到代码执行层。  
3）给执行层一个受控工作台：临时文件夹、白名单工具、可复用容器、超时和步数预算。  
4）要求它最终只回传高信号结果：摘要、异常、diff、候选结论，不回传整桶原始数据。  
5）对外发消息、写正式记录、改线上配置这类高风险动作，保留人工审批或至少保留干跑模式。

5. GenGrowth 可以怎么用

- **研究报告生产**：让 Agent 在执行层里批量抓来源、去重、抽段落、做初步归类，最后只把候选结构和关键信号交回写稿层。  
- **增长运营自动化**：处理渠道数据、线索表、发布清单、历史消息比对时，先在 sandbox 里清洗和聚合，再让模型判断下一步动作。  
- **AI Builder 工作流**：读组件目录、PR diff、字段 schema 这类重上下文任务，先让执行层整理成可审查工件，再交给 Builder 或审校 Agent 决策。

6. 今日行动

挑一个你现在已经接了 3 个以上工具的流程，不改业务目标，只改执行方式：  
先做一版“逐步工具调用”，再做一版“模型写小程序到 sandbox 执行”，对比三件事——总耗时、token 消耗、人工复核负担。  
如果第二版明显更稳，就说明你该优化的不是 prompt，而是执行层。
