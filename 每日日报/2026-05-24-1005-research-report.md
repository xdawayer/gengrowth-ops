今天读什么｜别只调 Prompt，要开始设计 Agent 的工作环境

1. 适合谁读 / 预计阅读时间

适合正在做 AI Builder、智能体工作流、内容生产自动化、代码/资料整理、内部 Bot 协作的人读。  
预计阅读时间：8–10 分钟。

2. 为什么值得读

过去很多人做 Agent，注意力都放在提示词、模型选择、工具接入上，总觉得“再补一个 Prompt、再接一个工具”就会更稳。但最近一批更接近生产环境的实践，开始把重点移到另一层：**不是先问 Agent 会不会做，而是先设计它在哪个环境里做。**

Pulumi 最近写得很直白：2026 年做 Agent，很多“中间层脏活”已经被 SDK 吃掉了，真正的差异化开始上移。Claude、Codex 这类系统已经自带读写文件、执行命令、搜索网页等基础能力，问题不再是“有没有工具”，而是“给它什么边界、什么上下文、什么默认工作面”。

这对 GenGrowth 很重要。因为我们很多任务都不是一句话问答，而是要让 Agent 真正开工：读资料、整理结构、改文档、跑脚本、生成草稿、回写任务。如果工作环境设计得差，Agent 就会变成一个需要人时时盯着的“高配聊天框”；环境设计得对，它才可能像一个能接活的数字同事。

3. 核心概念

第一，**沙盒不是安全附属品，而是效率工具**。  
Mike McQuaid 分享的实践很有代表性：如果没有隔离环境，你会不断被“要不要放行这个动作”打断；一旦把 Agent 放进受限沙盒，反而可以在边界内给它更高权限，让它放手干活。

第二，**工作树不是 Git 技巧，而是并行产能工具**。  
同一个仓库开多个 worktree，本质上是在给多个 Agent 提供互不打架的独立工位。这样你才能同时跑不同方案、不同任务，而不是所有 Agent 挤在一个目录里互相污染。

第三，**工具不是越多越强，而是越精选越稳**。  
LangChain 在 Open SWE 里总结得很清楚：成熟团队的共同点不是“工具海”，而是“精选工具集 + 明确职责 + 可控触发”。重点不是堆数量，而是保证每个工具真的值得保留。

第四，**上下文要在开工前喂够，而不是中途瞎找**。  
好的 Agent 不是开跑后再到处翻，而是在启动时就拿到任务线程、规则文件、仓库约定、验收要求。这样模型的推理会更聚焦，返工也更少。

第五，**环境层才是新护城河**。  
基础工具越来越像水电煤，真正拉开差距的，是你有没有把沙盒、上下文、验证、回写、审查这些环节编成一套默认工作面。

4. 可复用方法

如果你想把现有 Agent 流程升级一层，可以直接套这个四步法：

- 先给每类任务一个独立工作空间，不要多人多 Agent 共用一份运行现场。  
- 再给 Agent 一组“少而硬”的默认工具，只保留高频、稳定、可验证的动作。  
- 然后把仓库规则、任务背景、验收标准提前注入，而不是等它边做边猜。  
- 最后补上固定出口：测试、审查、回写任务、产物链接，确保它不是“做完就散”。

一句话：**先设计工位，再设计提示词。**

5. GenGrowth 可以怎么用

第一，用在研究与内容流水线。可以把“搜集资料、整理卡片、生成初稿、人工复核”拆成多个隔离工作区，让不同 Agent 分头处理，再统一汇总。

第二，用在 Bot 协作。Hermes、PM、Ops 未来如果要更稳地接力，不应只共享聊天记录，而应共享结构化任务入口 + 各自独立工作空间 + 明确回写出口。

第三，用在 AI Builder 服务化。对外不是卖“一个很聪明的 Agent”，而是卖“一个有边界、有工位、有审查链路的执行系统”。客户真正愿意付费的，通常是后者。

6. 今日行动

今天就检查你手上一个 Agent 流程：

1）它有没有独立工作空间？  
2）它的工具是不是越积越多？  
3）它开始工作前，是否拿到了足够上下文？  
4）它做完后，有没有固定回写和复核出口？

如果这四个问题里有两个答不上来，问题大概率不在模型，而在工作环境设计。
