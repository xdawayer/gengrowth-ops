# 今天读什么｜别只做知识库，要给 Agent 做“上下文供给系统”

> 同步来源：Slack `#gengrowth-research-reports`；发送时间：2026-05-13 10:01（UTC+8）。

今天读什么｜别只做知识库，要给 Agent 做“上下文供给系统”

1. 适合谁读 / 预计阅读时间

适合正在做内容生产、客户研究、销售线索整理、内部知识库、AI Agent 工作流、增长自动化的人读。  
预计阅读时间：8–10 分钟。

2. 为什么值得读

很多团队做 AI 自动化时，第一反应是：“我们需要一个知识库。”于是把文档、销售记录、客户反馈、项目资料、竞品信息都丢进去，再接一个聊天机器人。

但真实使用后会发现：资料放进去了，不代表 Agent 就能用好。它可能找不到关键文件、引用过期资料、混淆客户背景、把不同项目的信息拼错，最后人还是不敢让它执行任务。

这说明问题不只是“有没有知识库”，而是有没有一套稳定的“上下文供给系统”。

知识库像仓库，负责存东西；上下文供给系统像调度中心，负责在正确的时间，把正确的信息，以正确的结构，交给正确的 Agent。最近 Garry Tan 提到 GBrain 把 memory、code、search 统一到一个 graph 和 query interface 下，本质上也是这个方向：不要让信息分散在文件夹、聊天记录、搜索框里，而是把它变成 Agent 能持续调用的工作底座。

3. 核心概念

第一，知识库不是上下文。

知识库解决“资料存在哪里”；上下文解决“这次任务应该用哪些资料、按什么优先级使用、哪些资料不能用”。  
比如写一篇客户案例，Agent 需要的不是全部客户资料，而是：客户行业、痛点、购买前状态、使用后结果、可公开引用的数据、不能外传的信息边界。

第二，搜索不是理解。

普通搜索通常返回一堆相似文档，但 Agent 真正需要的是结构化关系：这个客户属于哪个行业？对应哪个产品方案？过去沟通过哪些需求？有哪些可复用素材？哪些结论已经过期？  
这就是为什么 graph / entity / metadata 变重要。它们不是为了看起来高级，而是为了减少 Agent 把信息拼错。

第三，上下文要有“新鲜度”和“可信度”。

同一个客户，三个月前的需求和今天的需求可能已经不同；同一个产品，旧版功能说明可能已经失效。  
如果 Agent 不知道资料更新时间、来源、适用范围，就容易把旧信息当成当前事实。

第四，权限边界也是上下文的一部分。

不是所有资料都应该被所有 Agent 读取。内容 Agent 可以看公开案例和脱敏摘要，但不应该读取合同细节；Ops Agent 可以整理 inbox 草稿，但不应该继承 CEO/PM 的私密决策上下文。  
所以好的上下文系统不仅要“找得到”，还要“只给该给的”。

4. 可复用方法

可以用一个简单框架来设计上下文供给系统：任务 → 资料 → 结构 → 校验 → 回写。

第一步，先按任务建上下文，而不是按文件夹建知识库。

不要问“我们有哪些资料”，而要问“Agent 要完成哪些任务”。  
例如：

- 写 SEO 文章，需要关键词、目标读者、产品定位、竞品页面、历史爆文。
- 做客户研究，需要客户画像、访谈记录、痛点标签、购买阶段、可跟进动作。
- 生成销售草稿，需要客户背景、上一轮沟通、当前意图、禁用话术、下一步目标。

第二步，给资料加最小元数据。

不需要一开始做复杂系统，先给每条资料补 5 个字段就够用：

- 来源：来自哪里
- 对象：关联哪个客户/项目/产品/主题
- 时间：什么时候产生或更新
- 权限：谁可以看、谁不能看
- 用途：适合用于内容、销售、研究、客服还是内部决策

第三步，让 Agent 先“取证”，再“生成”。

一个可控流程应该是：

1. 先检索相关资料  
2. 摘出证据和来源  
3. 判断是否足够完成任务  
4. 不足时标记缺口  
5. 足够时再生成结果  

不要让 Agent 一上来就写正文。对增长团队来说，“依据是什么”比“写得像不像”更重要。

第四步，设置低置信度处理。

当资料冲突、过期或缺失时，Agent 不应该硬编。可以要求它输出：

- 我找到了哪些资料
- 哪些信息互相冲突
- 哪些结论不能确认
- 需要人工补充什么
- 可以先做哪些低风险动作

第五步，把产出回写成新上下文。

每次 Agent 完成任务后，不要只留下最终文案，还要沉淀中间资产：选题、标签、客户痛点、可复用句式、失败原因、人工修改点。  
这些才是下一次自动化变聪明的燃料。

5. GenGrowth 可以怎么用

对 GenGrowth 来说，这个主题可以直接落在四个场景。

第一，内容生产。

把客户资料、SEO 关键词、历史文章、竞品页面、产品定位统一成“内容上下文包”。每次写文章前，Agent 先生成内容简报：目标读者是谁、该解决什么问题、可引用哪些事实、哪些表达不能用。这样内容质量会比单纯 prompt 写作稳定很多。

第二，客户研究。

把销售沟通、客户网站、社媒信息、行业标签、过往需求整理成客户卡片。Agent 不只是总结客户，而是输出“下一步增长机会”：适合做 SEO、GEO、社媒测品、自动化工作流，还是先做定位梳理。

第三，内部 Bot 协作。

CEO / PM / Ops / Hermes 等不同 Bot 不应该共享私密上下文，但可以共享任务化摘要。上下文供给系统可以帮助判断：这个任务需要谁推进、能传递哪些最小必要信息、哪些内容必须留在原角色边界内。

第四，研究报告和情报沉淀。

每天的 AI Builder 信息很多，如果只存链接，复用价值很低。更好的方式是沉淀为主题节点：Agent QA、上下文工程、结构化输出、企业 Agent 落地、知识图谱、增长自动化。以后写文章或设计服务包时，Agent 可以按主题调用，而不是重新翻历史消息。

6. 今日行动

今天可以做一个很小的试点，不要先做大系统。

选一个高频任务：比如“写一篇 GenGrowth 客户案例文章”或“整理一个潜在客户研究卡片”。

然后建立一个最小上下文模板：

- 任务目标是什么
- 需要哪些资料
- 资料来源在哪里
- 哪些信息可以公开使用
- 哪些信息需要人工确认
- 最终产出要回写哪些标签

接着让 Agent 按“先取证、再生成”的流程跑一次。重点观察三件事：

1. 它找资料是否准确  
2. 它有没有引用过期或不该用的信息  
3. 它生成的内容是否比普通 prompt 更可控  

如果这一步跑通，再考虑把模板扩展到 SEO、客户研究、销售跟进、日报生成等更多流程。
