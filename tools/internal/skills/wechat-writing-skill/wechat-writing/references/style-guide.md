# 写作风格完整指南

> 本指南有两种声音。**下面"一/二/三"主要写声音 1（踩坑活人感）**，范文是占星网站那篇。
> **声音 2（创始人框架）见下面"零"节**。动笔前先按 SKILL.md Step 2 定好走哪种声音。

## 零、创始人框架声音（声音 2）

Lynne 视角，从实践提炼框架和趋势判断。和踩坑声音的区别：不靠"踩坑情绪叙事"，靠"判断 + 论证"。

**关键特征**：
- **反共识切入**：开头就拆一个多数人默认正确的假设（如"流程崩 = 模型不够聪明"）。
- **递归论证**：每节回答上节引出的反问，链条是"拆旧 → 建新 → 防反驳 → 具体下注 → 方法论/边界"。
- **数据/原理锚定**：关键判断有市场数据、第一性原理推导或真实来源支撑，不纯断言。
- **知性坦诚**：承认判断可能被证伪（"这套做法也不是定论，还在跑数据验证"）。
- **结构惯例**：常用"写在前面的话"开场（含读者导航：这篇讲什么、需要几分钟）+ "写在最后"金句收束；判断句单独加粗成段。

**共享规则**：句子短、段落小（≤5 行）、口语但克制、手机端友好、禁区词同下（§1.6）。

**标杆范文（不内嵌，直接读原文对齐）**：
- `内容资产/微信公众号/离开年营收数亿的公司后，我All in了一个"价值被严重低估"的市场.md`（数据表现最好）
- `内容资产/微信公众号/模型不是门槛，承接系统才是——小团队 AI 主工作流实战拆解.md`（prompt→context→harness 框架范例）

## 一、详细风格规则（声音 1：踩坑活人感）

### 1.1 语气校准

**核心原则：像跟朋友吐槽一样写，不是写报告。**

- 口语化表达：可以用"卧槽""真的会笑出声""无语"这类词
- 自嘲语句：比如"笑着笑着就沉默了""心疼吗？心疼。但不删更心疼"
- 情绪要真实：有崩溃、有无语、有小得意，不要全程正能量
- 内心独白：用问句展示思考过程，比如"那个时刻我问自己：还救吗？"
- 类比要接地气：比如"就像那种考试交卷前说'这次稳了'的同学，结果成绩出来挂科了"

**情绪节奏（一篇文章内要有起伏）：**
- 崩溃/无语 → 小得意/成就感 → 自嘲/反思 → 收获/总结
- 不要通篇一个情绪，要有高低起伏，像聊天一样自然流动

### 1.2 句子和段落

- **句子短**，能用句号就别用逗号
- 一个段落说一件事，别堆在一起
- 重点结论单独成段，**加粗**或空行强调
- 可以用"教训：""解法：""转折点："这种直给的小标题
- 对话体穿插，增加画面感：比如直接写老板说了什么、我当时怎么想的

### 1.3 开头设计

**前3句话决定读者走不走。**

好的开头模式：
- **先说结论**：「先说结论：我做出来了。但过程嘛……」
- **抛出冲突**：「故事要从那个让我心态崩掉的下午说起。」
- **一个细节切入**：「我用 AI 生成了一个占卜网站，花了4美金。然后我开始焦虑。」
- **反差/意外**：「我以为最难的是写代码。结果最难的是不让 AI 乱改我的代码。」

绝对不要这样开头：
- "在这个 XXX 的时代……"
- "随着 AI 的发展……"
- "相信大家都有过这样的经历……"
- 长段背景介绍

### 1.4 结尾设计

**结尾要给读者一个明确的"收获感"。**

好的结尾模式：
- **收获清单**：用加粗的数字列表总结关键教训
- **一句话金句收尾**：「这个认知本身就很值。」
- **留一个钩子**：预告下一篇的方向，比如「等我踩完增长的坑再来写」
- **三连肯定句**：简短有力，如「能做出来吗？能。会踩坑吗？会。值吗？值。」

结尾可以自然引出增长话题，但不要硬广。用老板的一句话、一个观察、一个未完待续的问题来过渡。不需要硬 CTA。

### 1.5 细节要求

- **多用具体数字**：4美金、每天$40+、改了三次、删了2000多行代码
- **工具名要真实**：Claude Code Opus、Codex、AI Studio、Gemini——不要用"某 AI 工具"替代
- **报错代码和真实场景可以写进去**：400、429、500 这种报错数字本身就有画面感
- **保留"不完美"的部分**：「还在优化中」「还没完全解决」
- **引用真实对话**：老板说的话、自己的内心独白、和 AI 的交互
- **用引号标注 AI 的输出**：比如 Claude 说"完美"——这种细节很有传播力

### 1.6 绝对禁区

**用词禁区：**
- "在这个 XXX 的时代"开头
- "首先、其次、最后、综上所述"
- "干货满满""建议收藏""强烈推荐"
- "赋能、抓手、闭环、颗粒度、生态、矩阵、打法"
- "相信大家""众所周知""不言而喻"
- "深度好文""万字长文""一文读懂"

**结构禁区：**
- 排比句抒情
- 每段都工整地三点并列
- 用一二三四五做大纲式罗列（教训总结除外）
- 每个标题后面都跟一段解释性文字

**内容禁区：**
- 假装一切顺利，美化过程
- 假装自己很懂技术（我就是不懂，这是卖点不是缺点）
- 给出未经验证的建议，说得像"权威指南"一样
- 贬低其他工具或方法（说自己的选择就好）
- 把偶然的成功包装成必然的方法论

### 1.7 微信公众号排版

- 正文字号 15px，行间距 1.75 倍，字间距 1px
- 段落之间空一行
- 重要结论用 **加粗**
- 分隔线（---）用于分隔大的章节
- 小标题用 `## 坑1：xxx` 的格式，简洁直接
- 每隔3-5个段落考虑插入配图
- 关键转折点前后各空一行

## 二、完整范文

以下是 GenGrowth成长笔记 的第一篇公众号文章全文。这篇文章代表了目标风格的标准。写新文章时，语气、节奏、结构都要以此为锚点。

---

## 范文：一个不会写代码的产品经理，用7天做出占星网站

先说结论：我做出来了。

但过程嘛……就是那种"边骂边做，骂完还得继续做"的状态。

---

### 起点：4美金的冲击

故事要从那个让我心态崩掉的下午说起。

我用AI生成了一个占卜网站，一句话需求，几分钟出活，花了4美金。

当时我的第一反应是：卧槽，这也太简单了吧。

然后第二反应是：那老板花这么高工资雇我，图啥？

这个心理落差，笑着笑着就沉默了。

但一个demo和一个真正能用的产品，中间隔着一百个坑。后面的7天，我全用来验证这句话了。

---

### 坑1：工具选错，等于白干

一开始我用中转站的Claude Code Sonnet，便宜。

代价是：降智的时候写出来的代码纯纯灾难，改一个bug冒三个，改着改着就成屎山了。

后来试了AI Studio做前端。Gemini有个毛病——你说你的，它写它的。你让它改A，它顺手把BCD都改了，还觉得自己特别聪明。

最崩溃的是网络不稳定那几天，400、429、500和"continue"轮着报错。我盯着屏幕，真的会笑出声。不是好笑，是无语。

**转折点：上了满血Claude Code Opus + Codex。**

Codex虽然慢得抠脚，但它能兜底修bug，这个很重要。

**教训：一定要用最好的工具。** 不是装逼，是因为用烂工具你根本不知道AI的能力边界在哪，很容易半路放弃。我要是一开始就用满血版，至少能省两天。

---

### 坑2：架构没想清楚，推倒重来

我最开始的思路：Claude Code + 多Agent（PM、Coder、Tester、UI），一把梭。

想得挺美：PM写需求，Coder写代码，Tester测试，UI出界面，全自动流水线。

结果呢？代码量越滚越大，屎山越堆越高，UI丑得我自己都不想看。

换思路：AI Studio生成前端，Claude补后端。

还是屎山。而且这次改不动了——改一处，崩三处，经典屎山特征。

那个时刻我问自己：还救吗？

盯着满屏的报错，我做了一个艰难的决定：不救了，重写。

大概删了2000多行代码。心疼吗？心疼。但不删更心疼。

**教训：前期花1小时想清楚架构，比后期花3天填坑值。** 我现在的做法是先用AI Studio生成结构图，确认没问题再动手写。别一上来就让AI梭哈，它梭得很快，你填坑很慢。

---

### 坑3：生成太慢，好几次以为卡死了

网站能跑了，但AI内容生成延迟久到离谱。

我的优化路径：
- 最开始每个场景各自prompt → 乱
- 改成总prompt + 分场景 → 好一点
- 上RAG + 并行生成 → 更慢了
- 保留总分结构，同时细化角色，场景，语气和文化背景 → 还不错
- 持续优化中……

**教训：prompt架构要提前设计。** 别想着先能跑再优化，后面改成本很高。

---

### 坑4：UI反复改，没有尽头

AI生成的第一版UI，怎么说呢……能看，但就是那种"一眼AI味"的感觉。

配色像PPT模板，布局像后台管理系统，字体大小忽大忽小。

我让它改，它改完我更崩溃——改了颜色，把我调好的间距也改了。改了按钮，把我的圆角也改了。

来来回回改了大概十几版。

后来我学乖了：**先保证不丑，再追求好看。**

流程是这样：AI Studio出第一版 → 截图标注我要改哪里 → 一次只改一个地方 → 用frontend-design之类的skill微调细节。

没有一步到位的方法，就是磨。但至少找到了"不让它乱改"的方法：指令要具体，改动要小步。

---

### 坑5：功能越加越多，把自己埋了

本命盘、今日运势、合盘、CBT日记、Wiki、占星工具……

产品经理老毛病犯了——逛占星社区看到什么功能就想加。

"这个功能用户肯定喜欢！"

"这个竞品有，我们也得有！"

"加这个应该不难吧？"

加着加着，功能列表从5个变成15个。然后发现：每加一个功能，前面做好的东西就有可能崩。

有一次加了个小功能，结果首页的运势模块直接白屏了。排查了两小时，发现是状态管理冲突。

**教训：先做核心功能，跑通了再说。** 现在我在砍功能，把15个砍回7个。早知道一开始就克制点，能省多少事。

---

### 坑6：Claude的"完美"循环

这个必须单独说。

Claude改bug的时候会进入一种自嗨模式：

"我找到问题了。" → 改完还是报错

"这次真的找到了，完美。" → 还是报错

无限循环，无限自信。

我盯着它输出"完美"两个字，真的无语到笑。

就像那种考试交卷前说"这次稳了"的同学，结果成绩出来挂科了。一模一样的自信。

**解法：** 用Codex兜底。Claude搞不定的丢给Codex，慢是慢，但它不会跟你说"完美"——它只会默默把问题解决掉。

---

### 关于成本

后来token烧得太猛，每天$40+，Codex额度也秒没。

有一天我看了下账单，一周烧了快$300。

我跟老板说了这个数字，以为会被说"你怎么这么费钱"。

结果她说：不心疼，报销，开最高级的账号。不要因为工具阻碍产出。

**这句话我记住了：工具成本是最不值得省的成本。**

后来老板拿自己的星盘测试网站，一边看一边逐条比对，连连说"真准！真准！"

然后跟我说："我对你很有信心。"

那一刻还是挺有成就感的。毕竟折腾了这么久，总算有个正反馈了。

（但她还是天天问我"网站好了没"，她自己的SEO还没开始搞呢）

---

### 7天下来，我学到的几件事

**1. 工具选最好的。** 省钱省在工具上，最后会用时间和心态买单。

**2. 架构先想清楚。** 用AI Studio画结构图，确认了再写。别一上来就让AI梭哈。

**3. 屎山不要救。** 改不动就重写，沉没成本是沉没成本。

**4. prompt要当架构设计。** 不是随便写几句话的事，提前规划好场景和结构。

**5. 功能要克制。** 先跑通核心，别什么都想要。

**6. Codex是兜底神器。** Claude自嗨的时候，丢给Codex收尾。

---

### 最后

现在回想，我转AI coding有点晚了。

如果更早用满血工具，少在烂工具上耗时间，可能5天就搞定了。

但说实话，做出来只是第一步。

老板跟我说过一句话："AI让做产品变简单了，但让产品被人发现变得更难了。"

这7天我验证了前半句。后半句嘛……等我踩完增长的坑再来写。

---

这篇文章就当给想试AI coding的产品经理一个参考：

**能做出来吗？能。**

**会踩坑吗？会。**

**值吗？值。**

起码你会知道，原来不会写代码，也能做出一个真正能用的东西。

这个认知本身就很值。

## 风格提取要点

从这篇范文中提取的关键风格特征，写新文章时对照：

| 特征 | 范文中的体现 |
|------|------------|
| 开头直给 | "先说结论：我做出来了" — 不铺垫，直接说 |
| 口语化 | "卧槽""纯纯灾难""慢得抠脚" — 像在跟朋友说话 |
| 自问自答 | "还救吗？" "心疼吗？心疼。但不删更心疼" — 模拟内心活动 |
| 具体数字 | "4美金""2000多行""$40+/天""$300" — 不含糊 |
| 工具真名 | "Claude Code Sonnet""AI Studio""Gemini""Codex" — 不说"某工具" |
| 接地气类比 | "就像考试交卷前说'这次稳了'的同学" — 人人都懂 |
| 真实对话 | 老板说"不心疼，报销" — 直接引用 |
| 教训格式 | "**教训：xxx**" — 加粗，简短，可操作 |
| 保留不完美 | "持续优化中……" — 不假装全搞定了 |
| 情绪起伏 | 崩溃→无语→成就感→自嘲 — 不是一个调 |
| 结尾金句 | "这个认知本身就很值" — 一句话收住 |
| 留钩子 | "等我踩完增长的坑再来写" — 预告下一篇 |
