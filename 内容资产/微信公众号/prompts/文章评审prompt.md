# 任务：评审微信公众号文章

## 你的角色

你是一位资深内容编辑，负责评审微信公众号"GenGrowth成长笔记"的文章。
目标受众：个人开发者/一人公司。
你需要给出诚实、具体、有建设性的评审意见。

## 作者Profile：创始人（Lynne）

创始人文章的核心是**从实践中提炼框架和趋势判断**。不要求每段都有个人踩坑故事，但要求论述有深度、逻辑自洽、能做到"升华而不鸡汤"。读者期待的是：一个在前线做事的创始人，怎么理解正在发生的变化。

## 评审维度与评分标准

### 1. 核心观点（权重25%）

| 分数 | 标准 |
|------|------|
| 5 | 核心观点是**反共识**的——拆解了一个多数人认为正确的假设，并给出了替代框架。能用一句话概括 |
| 4 | 核心观点清晰且有独特性，但"反共识"的力度不够，更像"大家隐约知道但没说清楚的事" |
| 3 | 能看出主题，但观点停在行业共识层面，没有提供新的认知框架 |
| 2 | 多个观点并列，没有主次 |
| 1 | 读完不知道作者的判断是什么 |

### 2. 论据与深度（权重25%）

| 分数 | 标准 |
|------|------|
| 5 | 关键论点有市场数据或第一性原理推导支撑，升华落在趋势判断上，知性坦诚（承认可证伪性） |
| 4 | 框架清晰，大部分论点有数据/推导支撑，个别升华稍显悬空 |
| 3 | 有框架但论据偏软（多为比喻和概念，少有数据和推导） |
| 2 | 框架松散，论点多为断言 |
| 1 | 通篇断言，无论据 |

检查方法：对每个关键论点问"凭什么这么说？"——合格的答案是市场数据、第一性原理推导或真实用户反馈。不合格的答案是"因为作者这么觉得"。

### 3. 逻辑链（权重25%）

| 分数 | 标准 |
|------|------|
| 5 | **递归论证**：每节回答上一节引发的自然反问，形成"拆旧→建新→防反驳→具体下注→方法论"的链条 |
| 4 | 整体有递进关系，但1-2处过渡依赖显性连接词而非论证驱动 |
| 3 | 基本有序，但部分章节可以任意调换顺序（并列而非递归） |
| 2 | 各节独立存在，像要点罗列 |
| 1 | 逻辑混乱，前后矛盾 |

检查方法：读完每节后问"所以呢？"或"凭什么？"——如果下一节恰好回答了这个问题，说明递归成立。尝试交换任意两节顺序，如果不影响理解，说明是并列而非递归。

### 4. 可读性（权重15%）

| 分数 | 标准 |
|------|------|
| 5 | 段落短（≤5行），节奏有变化，金句自然嵌入，手机端阅读流畅 |
| 4 | 整体流畅，个别段落过长 |
| 3 | 可读但有些拖沓 |
| 2 | 段落普遍过长 |
| 1 | 不适合公众号 |

### 5. 标题与开头（权重10%）

| 分数 | 标准 |
|------|------|
| 5 | 标题包含反共识判断或价值承诺，开头在前3段制造认知冲突，并提供读者导航 |
| 4 | 标题有判断力，开头能制造好奇但缺少读者导航 |
| 3 | 标题平平，开头有铺垫但进入问题较慢 |
| 2 | 标题无判断力，开头在做元叙述（"这篇文章我想讲..."） |
| 1 | 标题和开头都让人想关闭 |

## 风格基线参考

以下是该公众号数据表现最好的创始人文章《离开年营收数亿的公司后，我All in了一个"价值被严重低估"的市场》的开头和结构，作为风格基线：

**开头：**
> 写在前面的话：
> 这篇文章不是离职声明，也不是创业宣言。
> 它是我花了两三个月时间，拆解自己的人生算法后，得出的一个反共识结论。
> 如果你只想知道我接下来要做什么，可以直接拉到最后。
> 如果你想理解我看到的那个"被定价错误的市场"，请给我12分钟。

**结构模式（递归论证）：**
- 第一节"大多数人在优化错误的变量"→ 拆解"营收=价值"的假设
- 第二节"真正的机会在被误解的稀缺性"→ 回应"那你看好什么？"
- 第三节"为什么大公司无法填补这个缺口"→ 回应"为什么没人做？"
- 第四节"我在下注什么"→ 回应"那你做什么？"
- 第五节"为什么选择透明构建"→ 回应"怎么做？"

**关键特征**：反共识切入、递归论证、市场数据锚定（"15亿独立工作者"）、决策理论语言（optionality/凸性机会）、知性坦诚（"这些判断可能是错的"）。

# 我们怎么用 Obsidian、GitHub、终端和 skills，搭企业级主工作流

**写在前面的话：**

上一篇文章里，我写了为什么 OpenClaw 很好，但我们最后没有把它放进主工作流。

这一篇，我想把另一半讲清楚：

**我们现在到底怎么搭一套更适合 OPC 和小团队的企业级主工作流。**

如果把这套系统说得更直白一点，它其实就是用 `Obsidian + GitHub + 大模型 + 终端 + skills`，去搭一套更像"企业级 OpenClaw"的东西。

但这里的重点，不是做一个更强的个人助手。

而是让一条正式工作，能从对话进入一个**可沉淀、可版本化、可评审、可执行**的系统。

这篇文章不是工具推荐，也不是手把手教程。

我更想讲的是：**为什么我们现在越来越倾向于这样设计一套企业级主工作流。**

这里的"企业级"，不是指大公司。

它真正的意思是：**一套正式工作流，不能只靠人的记忆和临场判断，而要能被系统接住。**

## 一、这不是工具堆砌，而是在给正式工作搭承接系统

说得直白一点，我们这套系统就是用 `Obsidian + GitHub + 终端 + 大模型 + skills`，去搭一套更像"企业级 OpenClaw"的东西。

看到这个组合，第一反应可能是：是不是又在堆工具？

如果目标只是个人提效，确实没必要。但如果你想要的不是"今天更顺手"，而是"这条工作链路下个月还能稳定跑"，问题就变了。

这套东西合在一起，是一条正式工作的承接系统。它至少要解决三件事：

- 让 AI 能看到真实业务上下文，而不是每次从零开始
- 让正式输出有版本、有归属，而不是散落在聊天记录里
- 让一条工作在多人协同、持续迭代时，仍然能被追踪、被修改、被复盘

### 三个视角：从 prompt 到 context 到 harness

在讲具体怎么搭之前，先把一个框架讲清楚。这三个词，刚好对应了"怎么用好大模型"这件事的三个视角：

**prompt engineering（提示词工程）**：一次任务怎么问得更准。通过定义问题、语言约束和任务边界，把输出空间收窄。

**context engineering（上下文工程）**：怎么让回答更准。核心是模型在动手之前，到底能看到什么文档、历史决策和业务上下文。

**harness engineering（驾驭工程）**：在进入长链路、多人协同、正式交付之后，模型外面的环境、约束、反馈回路怎么设计，才能让结果稳定。

如果只做聊天助手，停在第一层就够了。

但如果你想搭主工作流，迟早会发现：

**对话越短，prompt 越重要；工作越长，harness 越重要。**

后面每一层的选择，都可以放进这个框架里理解。以下不再重复注释这三个概念。

真正让我越来越在意的，其实已经不是"怎么把提示词写得更漂亮"，而是"怎么给 AI 搭一个能承接正式工作的系统外壳"。

如果要给大模型在这套架构里下一个更准确的位置，我会把它理解成这套系统的能力内核，而不是另一层新的治理结构。

Obsidian 提供可读、可关联的上下文组织方式，GitHub 提供贯穿草稿到定稿的版本基线、归属和变更追踪，终端提供执行入口，skills 提供流程边界，而大模型负责在这些边界内做理解、生成、改写和衍生。

它不是系统本身，但它是这套系统真正开始工作的引擎。

因为企业级工作流真正难的，从来不是"让 AI 先做起来"，而是"让它在该收手的地方收手"。

## 二、第一层是 Obsidian：让上下文有位置，也有边界

我们把知识库从飞书迁到 Obsidian，不是因为飞书不好用。恰恰相反，飞书对人很友好。

但从 AI 工作流的角度看，知识库最重要的要求不是"人看着舒服"，而是：**它能不能被稳定读取、关联、引用和持续更新。**

一条正式工作，最怕的不是 AI 写得慢，而是上下文没有固定位置：商业判断在聊天里，产品约束在群消息里，历史决策在脑子里，最新文档在某个不知道谁改过的在线页面里。

这种状态下，AI 每次都在黑暗中重新摸索。

所以我们的第一步，通常不是"先开一个对话"，而是先确认：这件事的背景文档在哪里，和哪些已有文档相关，它属于什么类型。

在这层上，Obsidian 真正有价值的不是界面，而是：文档天然在本地，目录和引用关系清楚，Markdown 对 AI 足够友好，更容易成为长期可读的知识层，而不是临时协作文档。

### 上下文不只要"有位置"，还要"有分层"

但光有知识库还不够。真跑起来以后会发现，所有内容都是 Markdown，放哪都差不多，直到它们开始互相污染。

我们现在会明确区分四类文档：

- 正式文档：商业计划、项目规划、PRD、SPEC、评审结论
- 过程记录：日报、阶段性复盘、临时推演
- 外部输入：调研资料、市场线索
- 对外资产：公众号文章、已发布内容

为什么这么麻烦？因为一旦这些内容混在一起，AI 就很容易把"正在思考的东西"和"已经定稿的东西"混为一谈。对主工作流来说，真正稀缺的不是文档数量，而是文档边界。

对人来说，知识库解决的是"找得到"。对 AI 来说，知识库解决的是"看得见"。而文档分层解决的是"分得清"。

在主工作流里，**看不见等于不存在，分不清等于全是噪声。**

## 三、第二层是 GitHub：版本不是工程师的专利

很多人把 GitHub 理解成工程团队的工具。但我们现在越来越把它当成正式工作从草稿阶段就开始使用的版本层。

一条正式工作一旦开始往下走，最怕的不是写得不好，而是：**没人知道哪一版才算数。**

比如一份 PRD。如果它只停在聊天记录里，或者某个被来回覆盖的在线文档里，那后面所有衍生出来的东西都会开始漂：SPEC 基于哪版 PRD 写的，评审意见挂在哪版上，后续执行到底在追哪版结论。

版本不是技术细节，而是业务秩序。

在这套主工作流里，GitHub 更像正式工作的版本化协作层。它解决的不是"能不能同步"，而是：哪版在流转、谁改了什么、为什么改、后续衍生内容应该基于什么继续往下走。

小团队工作流里最容易被低估的，恰恰是这一层。因为小团队没有大公司的流程兜底，一旦版本漂了，几乎没人有空把它完整捋回来。

## 四、正式生成发生在终端：GUI 是前台，终端是生产车间

如果只是闲聊、问答、摘要，图形界面的助手非常好用。但正式生成和持续改写，我们更习惯在终端里做。

原因不神秘。终端把几件原本断开的事放进了同一条链路：读取本地文件、引用已有文档、生成或修改内容、直接写回系统、继续衍生下一个文档。

这减少了一种很常见的损耗：**在聊天框里生成一版，再手动复制、粘贴、搬运、整理、归档。**

每多一次搬运，主工作流就多一次断裂的机会。

GUI 助手是前台，终端是生产车间。prompt 在这里当然还重要，但它的位置变了。它负责驱动一次具体生成，不再独自承担整个工作流。

## 五、为什么要有 skills：把随机性收进流程

skills 不是"给 AI 加技能包"。更准确地说，它的作用是：**把高频任务，从"每次重新提示"变成"有边界的流程调用"。**

真正重要的不是 AI 这次能不能灵光一现，而是：同类任务下次还能不能稳定跑、关键步骤是不是被固定下来了、输出格式和归档位置是不是可控、出错以后能不能把错误沉淀进流程，而不是只停在这次抱怨里。

比如一份公司调研报告怎么起手、一份产品分析怎么组织结构、一篇文章怎么从草稿进入正式稿、一份 PRD 怎么衍生到 SPEC。如果这些事每次都靠人临时想提示词，AI 的上限很难稳定。

一旦它们被固化进 skills，变化就出现了：人的负担变小，AI 的随机性变小，工作链路更容易被复用。

以前我们总想把 AI 调得更聪明。现在我更倾向于先把环境调得更稳定。

## 六、一条 PRD 链路，看这套系统怎么运转

把前面四层落到一个最具体的例子上。选 PRD，因为它天然处在正式工作链路的起点，又会继续往下衍生出 SPEC、评审和执行。

在我们这套系统里，一份 PRD 不会这样产生：

"在一个对话框里聊完，得到一版文档，然后发给团队。"

它更像是这样一条链路：

1. 在终端里把需求讨论清楚，同时引用已有的项目规划、业务背景和相关文档
2. 第一版草稿直接落到本地知识库，成为一个有明确位置和分层归属的文档对象
3. GitHub 把它纳入版本链路，而不是靠人回忆"刚才那版是不是最新的"
4. 评审围绕这份正式 PRD 继续修改、确认和收敛，而不是对一段聊天记录拍脑袋
5. 只有进入确认链路后，它才继续往下衍生 SPEC 和执行文档，而不是每一步重新开一个孤立对话

这条链路不酷，但它解决的都是实际问题：前面改了一处，后面会不会还在看旧版本；这次讨论的结论，下次还能不能被引用；评审是在对正式文档做评审，还是在对聊天记录做猜测。

同时，权限逻辑也跟着这条链路变化。摘要、检索、草稿生成这类动作，默认可以放得更开；但只要开始碰正式文档、版本确认、评审结论和对外口径，权限就必须收紧，进入版本链路和人工判断。

一份 PRD 之所以不能只停在对话框里，不是因为聊天不好用，而是因为：**对话适合生成，系统才适合承接。**

## 七、这也在反过来影响我对 GenGrowth 的理解

搭这套系统的过程，也在改变我理解 GenGrowth 的方式。

GenGrowth 当然可以有助手能力。但如果只停在助手层，它解决的更多还是个人效率问题。我现在更在意的是另一件事：**怎么把 AI 的能力真正接进业务系统，让它能被管理、被追踪、被复用。**

这也是为什么 harness engineering 这个词对我来说越来越重要。它提醒我，真正难的不是让模型这一次更聪明，而是让它在正式业务里持续地、不失控地工作。

## 写在最后

回头看这套系统，Obsidian、GitHub、终端、大模型、skills，没有特别新的概念，也没有花哨的界面。

但如果把每一层的选择放在一起看，它们指向的其实是同一件事：

**小团队的 AI 竞争力，不在于谁用的模型更强，而在于谁先把"对话"变成了"系统"。**

Obsidian 是让上下文从对话进入系统。GitHub 是让草稿、修改和归属进入系统。终端是让生成从聊天框进入系统。skills 是让流程从临场发挥进入系统。

每一层做的事情都很朴素，但它们共同完成了一次转换：从"AI 帮我做了一件事"，到"AI 的输出能被系统接住"。

当然，这套东西也有它的边界。当流程本身的维护成本开始高于它节省的协作成本时，就该停了。系统是用来承接工作的，不是用来承接系统本身的。对小团队来说，永远要警惕为了"规范"而制造出比问题更大的负担。

所以最后的判断其实很简单：

对个人助手来说，好用很重要。

对主工作流来说，能被管理更重要。

而对 OPC 和小团队来说，这可能比再多接几个 AI 工具，都更值得先做。
