# 你的 Agent 不是不够聪明，是交接太像聊天

**写在前面的话：**

我们做 Agent 工作流，踩过一个反复出现的坑：上一步看起来完成得很漂亮，下一步却接不住。

一开始我以为是模型不够聪明。换更强的模型、把提示词写得更细，该崩的地方照样崩。

后来才想明白：问题不在模型，在**交接**。

这篇想讲清楚一件事：一条正式工作流真正决定稳不稳的，往往不是模型有多聪明，而是**两步之间怎么交接**。以及，我们怎么把"对话式交接"换成"契约式交接"。

如果你也在做多步 Agent、内容流水线，或者多个 Bot 协作，这篇会给你一张能直接抄的交接单。需要大概十分钟。

## 一、流程崩了，我们第一反应总是怪模型

先说一个很常见的错觉。

Demo 里，一个 Agent 写出一段漂亮的总结，大家会觉得：通了，这流程能跑。

可一旦把它接到下一个 Agent、接到数据库、接到任务系统或者发布链路，问题就冒出来了：字段对不上、状态不明确、上下文越堆越肥、失败之后没法恢复。

这个时候，团队的第一反应几乎都是同一个：是不是模型不够强？换个更聪明的。

我们也这么干过。结果是，模型确实更聪明了，但流程该断的地方还是断。

道理其实不难想：**更强的模型像一个更聪明的同事，它能少犯单步的理解错误，却消除不了交接环节的不确定性。** 如果交出来的东西里没有状态、没有版本、没有风险、没有下一步，再聪明的下游也只能靠猜。

因为真正出问题的不是"这一步做得好不好"，而是"这一步的结果，下一步能不能直接用"。

**一个 Agent 把话说得漂亮，和它把结果交得清楚，是两件事。** Demo 看的是前者，工作流吃的是后者。

## 二、交接像聊天，错在哪

那为什么"交得清楚"这么难？

因为我们默认让 Agent 之间用**聊天的方式**接力——上一个吐出一大段自由文本，下一个再把这段文本重新理解一遍。

问题就藏在"重新理解"这四个字里。

自由文本对人是友好的，人扫一眼就知道重点在哪。但对下游的 Agent 来说，每接收一段自由文本，它都要重新做一次解析和判断。**每多一次重新理解，就多一次偏差的机会。**

对话越长，这个偏差累积得越厉害。你会发现链路跑到后面，已经没人说得清当前到底是基于哪一版结果在往下走了。

这不是模型笨。这是因为我们在用一种"适合人读"的格式，去做一件"需要机器稳定消费"的事。

AWS 在它那份多 Agent 架构建议里，把结构化输出直接放在 agent 与 agent 的边界上——它的说法是，这道边界的目的不是让输出更规整，而是**阻止不确定性向下游扩散**。我很认同。结构化输出不是为了好看，而是在两个环节之间立一道边界，让"解释"变少，让"消费"变多。

换句话说：**对话适合生成，系统才适合承接。** 在主工作流里，交接处需要的不是更漂亮的话，是更稳的边界。

## 三、"那不就是加个格式吗"

讲到这儿，通常会有人反问一句：那不就是让输出带个固定格式吗，至于单独写一篇？

如果只是套个模板，确实不至于。但"契约式交接"真正难的，是三件不那么直觉的事。

**第一，交接的内容要"最小充分"，不是越多越好。**

很多人以为，把上一步所有的思考都倒给下一步，下游就不会出错。恰恰相反。传太少，下游得重复推理；传太多，噪音会把真正有用的信息淹掉。好的交接，只传下一步必需的结果、依据、状态和待办，其余的不塞。

**第二，大产物不要在上下文里搬来搬去，要按引用传。**

Anthropic 那篇讲长任务 harness 设计的工程文章反复强调一点：别让所有历史一直堆在同一个上下文里，而是靠结构化产物和外部 artifact 接力。落到做法上——每一步只加载当前需要的规则和资料；大报告、长表格、代码、长摘要，落到文件或外部系统里，交接的时候只传一个引用和摘要。

这其实和我之前写过的一个判断是一回事：**在主工作流里，看不见等于不存在，分不清等于全是噪声。** 上下文不是越满越好，是越准越好。

**第三，工具的返回值本身，也是契约的一部分。**

一个工具如果只会返回"成功"或"失败"两个字，Agent 拿到"失败"之后是懵的——它不知道下一步该补什么，只能盲目重试。更实用的设计是：返回值分简洁版和详细版，报错时直接说清楚该改哪个参数。这样 Agent 才接得住一个失败，而不是卡在那里。

你看，这三件事没有一件是"加个格式"能解决的。它们是在回答同一个问题：**怎么让一次交接，是在传递一个可以被直接执行的结果，而不是传递一段需要重新理解的话。**

## 四、我们怎么落：一张六字段的交接单

说了这么多判断，给点能直接用的。

我们现在的做法，是在每一个交接处，固定输出六个字段。不管上一步是哪个 Agent、用的哪个模型，交出来的东西长一个样：

- `status`：这一步是成功、失败，还是需要人工确认
- `task`：这一步到底完成了什么
- `result`：结构化的结果本身
- `evidence`：结果的来源或依据
- `risks`：还有哪些不确定项、哪些缺口
- `next_action`：下一个节点该做什么

定这张单子的时候，关键不是问"上一步想说什么"，而是问：**下一个节点，必须拿到哪些字段才能继续？** 这两个问法，会得出完全不同的交接内容。

给一个能直接抄的样例。我们研究报告流水线里，"搜资料"这一步交给"提纲"这一步时，交出来的是这样一张单子：

```yaml
status: needs_review              # 还有源没核验，需人工确认
task: 完成"AI 工作流稳定性"选题的资料初筛
result:
  usable_sources: 8
  rejected_sources: 5
  core_claim: "工作流稳定性正从'优化提示词'转向'设计交接契约'"
evidence:
  - "AWS 多 Agent 架构：结构化输出 = 边界"
  - "Anthropic harness 设计：上下文分段 + 产物外置"
risks:
  - "其中一篇报告原文尚未核验，先标 needs_review"
next_action: "进入提纲生成；但风险项核验前，不要进入成稿"
```

对比一下原来怎么传：上一步丢给下一步的，是一段"我筛了一批资料，大概有这么几个方向，质量还行"的总结。

**崩在哪？** 下游拿到这段话，不知道哪些源已核验、哪些被否、当前到底基于哪一版结论往下写。万一中间换了模型重跑一遍，更没人说得清现在追的是哪一版。

**改成这张单子之后**，下游不用重新读全文：照着 `result` 写提纲、看到 `status: needs_review` 就知道要等核验、`next_action` 直接告诉它"能做提纲、别成稿"。失败也能恢复——`risks` 写明了卡在哪，补完那一项接着跑就行，不用从头再来。

选题线索系统、多 Bot 协作也是同一套：只传结果、依据、状态、下一步，不传那种"需要重新理解"的散文。

## 五、什么时候该停：契约不是越重越好

最后得说一个反方向的话，不然容易把人带沟里。

契约式交接不等于全自动，也不等于流程越严越好。

有两种情况必须明确停下来，转人工：一是高风险动作，比如写入、发送、删除、审批；二是同一步连续失败。这种时候让系统硬撑到底，比断在那儿更危险。**多 Agent 不代表无人值守，它只是让该交给人的地方，交得更清楚。**

还有一层边界更要紧。当你维护这套交接规则的成本，开始高于它帮你省下的协作成本时，就该收手了。

我一直提醒自己一句话：**系统是用来承接工作的，不是用来承接系统本身的。** 给一条一次性的小任务也套六字段交接单，那是自找麻烦。契约是给"会反复跑、会换人换模型、会往下衍生"的正式流程准备的。

当然，这套做法也不是定论。我们也还在调——哪些字段是真有用的、哪些场景其实不需要这么重，都还在跑数据验证。这篇给的是我们现在的判断，不是标准答案。

## 写在最后

回过头看，这件事其实很朴素。

我们花了不少时间想"怎么让模型更聪明"，后来发现，让一条流程稳定下来的，往往不是模型那一端，而是模型和模型之间、这一步和下一步之间，**交接的方式**。

对话适合生成，系统才适合承接。

把 Agent 之间的"对话"换成"契约"，不是为了规范而规范，是因为只有这样，一条正式工作流才能在换模型、换人、持续迭代的情况下，还跑得稳。

如果你也在被"上一步好像成了、下一步却崩了"困扰，今天可以只做一件小事：挑一条你正在跑的流程，给其中两三个步骤之间，补一张哪怕只有 `status / result / risks / next_action` 四个字段的交接单。

先让它接得住，再谈让它更聪明。
