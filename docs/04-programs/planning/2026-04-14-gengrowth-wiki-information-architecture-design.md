# GenGrowth Wiki 信息架构与迁移设计

> 日期：2026-04-14
> 目标：为 `gengrowth-wiki` 建立一套长期稳定、可扩展到多角色与多 Agent 的目录边界，并据此执行分批迁移。

## 一、设计目标

本次调整要解决的不是“文件放乱了”这个表象，而是三个更底层的问题：

1. 目录按部门名命名，导致同一类对象分散在多处。
2. 正式规则、业务实例、草稿输入混放，Agent 后续很难稳定判断边界。
3. 一些目录已经接入自动化或外部工具，不能按普通文档目录处理。

本设计的目标是：

1. 让每个文件只有一个主归属。
2. 让未来的 `HR agent`、`Legal agent`、`Ops agent` 都能按对象目录而不是按人脑记忆工作。
3. 降低二次迁移概率，避免今天迁完、下周再拆。

## 二、核心分类原则

### 2.1 四个判断问题

每个文件先回答四个问题：

1. 它是不是当前正式基线？
2. 它是不是某一次真实业务产生的实例？
3. 它是不是草稿、过程材料或外部输入？
4. 它是不是自动化入口、工具代码或独立系统？

### 2.2 四类主归属

| 类别 | 定义 | 主去向 |
|---|---|---|
| 正式基线 | 当前有效、长期引用、会被当作规则/模板/制度使用 | `docs/` |
| 业务实例 | 某一次业务实际产生的最终件、记录件、凭证件 | `实例库/` |
| 过程材料 | 草稿、讨论稿、待发布内容、外部参考输入 | `工作台/`、`参考资料/` |
| 系统/工具 | 自动保存入口、工具源码、扩展、独立仓库 | 原位冻结或并入 `tools/` |

### 2.3 三条硬规则

1. “后续可能会查阅”不等于“应进入 `docs/`”。
2. “已签/已发/已审批的具体文件”默认不进入 `docs/`，而进入 `实例库/`。
3. `docs/90-archive/` 只归档“旧版正式文档”，不承接业务实例。

## 三、目标顶层结构

本次迁移后的目标顶层结构如下：

```text
/
├── AGENTS.md
├── CLAUDE.md
├── REFERENCE.md
├── Clippings/               ← 系统入口，冻结
├── docs/                    ← 当前正式知识库
├── 实例库/                  ← 真实业务实例与最终件
├── 工作台/                  ← 草稿、待整理过程材料
├── 参考资料/                ← 外部输入与参考材料
├── 内容资产/              ← 已发布/待发布内容成品与素材
├── task-collab/             ← 任务协同区
├── tools/                   ← 工具、扩展、vendored repo、配套说明
├── 每日日报/                ← 日报
└── wzb-obsidian/            ← 独立知识系统，冻结
```

## 五、`docs/` 的目标内部结构

### 5.1 顶层模块不变

`docs/01-company` 到 `docs/06-shared` 的大模块仍然成立，不做推倒重来。

### 5.2 本次重点新增或调整的结构

```text
docs/05-governance/
├── people-ops/
│   ├── recruiting/
│   │   ├── playbooks/
│   │   ├── templates/
│   │   └── role-specs/
│   ├── onboarding/
│   │   ├── playbooks/
│   │   └── templates/
│   └── policies/
├── contracts/
│   ├── employment/
│   │   ├── templates/
│   │   └── playbooks/
│   └── commercial/
│       ├── templates/
│       └── playbooks/
├── finance-payments/
│   ├── playbooks/
│   └── templates/
└── corporate/
    ├── playbooks/
    └── policies/
```

### 5.3 `01-company` 与 `05-governance` 的边界

| 主题 | 主归属 |
|---|---|
| 商业计划书、公司战略、组织设计、价值观、文化原则 | `docs/01-company/` |
| 员工手册、入职制度、招聘 SOP、账号规则、合同模板、支付规则 | `docs/05-governance/` |

关键判断：

1. 回答“公司是谁、往哪走”的，进 `01-company/`
2. 回答“如何治理、如何约束、如何协作”的，进 `05-governance/`

## 六、`实例库/` 的定位

`实例库/` 不是旧文件仓，也不是废弃文档仓。它只存放真实业务实例。

建议结构：

```text
实例库/
├── contracts/
│   ├── employment/
│   └── commercial/
├── people-ops/
│   ├── resumes/
│   ├── candidate-evaluations/
│   ├── offers/
│   └── onboarding/
├── finance-payments/
└── corporate/
    ├── entity-documents/
    └── capital/
```

典型文件：

1. 已签劳动合同
2. 已签服务合同
3. 已发 Offer PDF
4. 候选人简历
5. 付款回单、发票扫描件
6. 营业执照、变更登记通知书、增资协议

## 七、`工作台/` 的定位

`工作台/` 是人和 Agent 的加工区，不是正式文档区。

建议结构：

```text
工作台/
├── inbox/
├── content/
│   └── wechat/
├── people-ops/
├── legal-finance/
└── references/
```

适合进入 `工作台/` 的内容：

1. 正在改的 HR/法务草稿
2. 待发布文章草稿
3. 评审 prompt
4. 临时会议整理
5. 尚未沉淀为正式规则的操作材料

## 八、特殊目录处理规则

### 8.1 `Clippings/`

`Clippings/` 当前是 Obsidian Web Clipper 的默认落盘入口。

本轮规则：

1. 不改名
2. 不迁移
3. 不纳入第一批整治目录

后续如果要调整，必须先改插件配置，再同步修改流程文档和自动化脚本。

### 8.2 `tools/`

`tools/` 定义为：

1. 内部工具
2. 外部 vendored 工具
3. 浏览器扩展
4. 技能包与脚本
5. 与工具强耦合的 README/安装说明

但不承接：

1. 泛方法论分析文档
2. 通用工作流心得
3. 与某个工具并不强绑定的研究文档

### 8.3 `docs/90-archive/`

`docs/90-archive/` 只做一件事：归档旧版正式文档。

它与 `实例库/` 的区别是：

1. `90-archive`：旧版规则、旧版方案、旧版正式文档
2. `实例库`：真实业务产生的最终件

## 九、第一批迁移范围

第一批只做高收益、高确定性的区域：

1. `法务财务/`
2. `人事行政/`
3. `docs/01-company/hr/`
4. `公司公共文档/`
5. `内容资产草稿/`

原因：

1. 这些目录交叉最严重
2. 与正式文档体系关联最强
3. 最容易影响后续 Agent 边界

本轮先不动：

1. `Clippings/`
2. `wzb-obsidian/`
3. `tools/` 以外的开发/实验目录

## 十、推荐执行顺序

### 方案选择

推荐采用：

1. 先全局定结构
2. 再做第一批高确定性迁移
3. 最后处理工具区、工作台区和零散文件

不推荐：

1. 先迁 `docs/01-company/hr/`
2. 再看根目录其他目录

原因是这样会把局部最优做成全局返工。

## 十一、迁移完成后的理想工作流

### 正式规则类文档

`工作台/` -> `docs/` -> `docs/90-archive/`

### 实际业务文件

`工作台/` 或模板填充 -> `实例库/`

### 外部输入

`Clippings/` 或 `参考资料/` -> 消化后进入 `docs/` / `工作台/` / `内容资产/`

## 十二、结论

本次迁移不应理解为“把文件搬整齐”，而应理解为：

1. 给正式知识库建立边界
2. 给业务实例建立稳定容器
3. 给 Agent 留出长期可扩展的工作地图

目录边界一旦定稳，后续新增 HR、法务、营销、产品 Agent 时，都会明显更稳。
