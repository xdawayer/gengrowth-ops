## 3. 模块二：社媒营销

| 核心目标     | 执行重点                     | 关键产出                            |
| -------- | ------------------------ | ------------------------------- |
| 完成核心平台建设 | 先把关键平台资料、权限和角色边界打通       | 核心平台账号矩阵与基础资料                   |
| 建立创始人主叙事 | 以公开实验为主线，由创始人首发，官方号承接和沉淀 | 首批创始人内容与官方承接内容                  |
| 建立轻量内容协同 | 用“母稿 + 平台适配器”替代多套重型流程    | 内容协同方案 `v0.1` 与首批平台适配模板 / skill |

### 3.1 平台与要完成的事项

| 平台 | 角色 | 要完成的事项 |
| --- | --- | --- |
| `X` | 创始人主战场 | 发布实验进展、核心判断、线程内容；作为首批英文公开内容主阵地 |
| `LinkedIn` | 创始人信任背书 | 发布长帖、阶段判断、对外叙事；承接更完整的 founder 视角 |
| `Reddit` | 问题场景验证 | 发布问题帖、参与评论互动、收集用户原话和真实需求 |
| `Indie Hackers` | Build in Public 日志 | 发布周报式实验日志、里程碑与复盘 |
| `微信公众号` | 中文深度沉淀 | 发布长文复盘、方法论和阶段总结 |
| `官方 X / LinkedIn` | 品牌承接 | 转发创始人内容、沉淀案例、导流官网或 waitlist |

### 3.2 内容协同方案

不建议一开始为每个平台各做一套完整重型 skill。更合适的方案是：

`1 份母稿 / 实验简报 + 多个平台适配器`

原因：
1. 维护成本更低。
2. 更适合创始人快速发布。
3. 更容易保证不同平台内容既复用又不只是翻译。

执行计划：
1. 保留现有微信公众号 skill，继续作为长文深度稿能力。
2. 新增 `X / LinkedIn / Reddit / Indie Hackers` 的轻量平台适配模板或 skill。
3. 统一输入为“实验简报 / 内容母稿”，统一字段包括：核心观点、证据、截图、CTA、平台禁忌。
4. 输出时按平台规范重写，而不是直译：
   `X` 重短句和 hook，`LinkedIn` 重判断和背景，`Reddit` 重问题语境和评论空间，`Indie Hackers` 重日志和数据。
5. 建立发布记录与复用关系，确保一份母稿可以拆成多平台版本并可追溯。
