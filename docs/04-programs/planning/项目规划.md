### 4.1 执行目标与口径

本轮执行围绕三个目标推进：
1. 目标A（官网预览）：完成基于 `SPEC v0.6 + PRD v1.3` 的可点击预览，覆盖首页、Features、Blog、Templates、About、Contact、404/500、法务入口。
2. 目标B（MVP 上线）：完成 `PRD v1.3` 的首版最小闭环，上线后具备 Waitlist 转化、Blog 内容引擎、归因、法务、OAuth、发布审计能力。
3. 目标C（增长验证）：以 `astrologywiki.com` 跑通“社媒测品 -> SEO 固盘 -> 归因复盘 -> 公开总结”闭环，并沉淀为可复用的 GenGrowth Playbook。

执行口径：
1. 自然流量 = `Organic Search + Social Referral`，不含付费投放。
2. `5000` 自然流量继续保留为**冲刺目标**，但不再作为唯一刚性承诺。
3. 本轮保底目标：`2026-05-31` 前完成闭环并累计达到 `2000-3000` 自然会话；若社媒探测与收录同步超预期，则冲刺 `5000`。
4. “可发布/无 P0”口径：无 `P0` 缺陷，`P1` 有替代路径且记录修复计划，关键链路可回滚、可监控、可审计。

### 4.2 文档基线与执行原则

1. 页面结构、文案、SEO/GEO 细节以 `SPEC.md` 为准。
2. 产品范围、FR、数据模型、NFR、验收以 `PRD v1.3` 为准。
3. 评审状态以“状态化评审单”为准；`R20-R23` 这类执行型建议在本模块下沉为节奏、排期和资源约束。
4. 首版实现遵循 `PRD v1.3` 的 MVP 收敛原则：逻辑职责完整保留，运行时编排收敛为 `3-4` 条主链路。
5. 高变动能力优先做成可插拔 Skill，不把它们硬编码进单一固定 Agent。

### 4.5 账号与治理约束（摘要）

本计划只保留项目执行必须知道的治理约束，不再重复账号分层、授权、安全与周节奏细节。  
详细口径统一引用：
1. `docs/03-marketing/02-social-media/social-media-operations.md`
2. `docs/05-governance/account-access/2026-03-12-platform-account-registration-playbook.md`

项目级统一口径：
1. 对外公开账号默认采用“双账号优先”：创始人账号负责实验叙事，官方账号负责案例沉淀与品牌承接。
2. 新 C 端实验产品在未达阈值前，不新增公开账号矩阵，只保留站点、落地页或社区测试入口。
3. 仅当产品连续 `3-4` 周出现稳定主题、独立受众、明确转化且有明确 Owner 时，才升级为独立产品账号。
4. 客户账号不纳入 `GenGrowth` 自有账号池，统一按“客户持有资产、GenGrowth 获取可撤销授权”处理。
5. 数据壁垒来自匿名化、聚合化学习资产，而不是长期占有客户原始数据。

### 4.6 客户协同与授权（摘要）

本计划只记录客户协同的项目级约束；授权方式、审批等级和交接台账以治理文档为准。

1. 客户保留账号产权、主体资质与高风险审批权。
2. `GenGrowth` 负责策略、内容草稿、实验设计、归因分析与复盘建议。
3. 长期协作优先平台原生角色授权或 OAuth，禁止长期共享密码。
4. 所有授权都必须可审计、可撤销、可交接。
5. 合同、权限与撤权基线见 `docs/05-governance/account-access/2026-03-12-platform-account-registration-playbook.md` 与 `docs/05-governance/对外合作/2026-03-04-client-data-authorization-terms.md`。

### 4.7 数据授权与学习闭环（摘要）

本计划只保留数据治理的最小执行口径，不重复数据分层、审计和保留机制的完整说明。

1. 原始数据默认归客户；`GenGrowth` 只在交付所需范围内处理。
2. 跨项目可复用的学习资产，只保留匿名化、聚合化、不可逆的衍生结果。
3. 采集遵循最小必要原则，原始数据与衍生数据必须隔离。
4. OAuth、审计日志、撤权、保留期限和案例授权按治理文档执行。
5. 统一事件字典、UTM 规范、内容标签和实验编号，必须在 MVP 前落地。

### 4.8 工具与技术路线

主工具 / 主模型：`OpenAI Codex（GPT-5）`  
主技术栈：`Next.js + Supabase + Vercel`

选择理由：
1. 预览快：分支级 Preview URL，适合先做官网预览。
2. 上线快：一体化前后端与托管，适合 MVP 从 0 到发布。
3. 可治理：适合补齐 OAuth、审计日志、同意管理、Waitlist、Blog 与归因链路。
4. 与 `PRD v1.3` 的 Skill-First、模型路由、版本治理方向兼容。

### 4.9 当前执行说明（非周计划）

本计划不再维护按周、按日期或 `Day 0-14` 的执行计划，避免历史排期继续干扰当前判断。

当前只保留以下长期有效的执行说明：
1. 对外账号默认只经营“创始人账号 + 官方账号”两套公开资产。
2. C 端产品的真实获客继续在对应渠道执行；X 上的实验解释默认由创始人账号首发，官方账号做沉淀。
3. 只有当某个产品连续 `3-4` 周形成稳定主题、独立受众、明确转化且有明确 Owner 时，才升级为独立产品账号。
4. 注册、养号、恢复信息、设备隔离等细节统一以 `docs/05-governance/account-access/2026-03-12-platform-account-registration-playbook.md` 为准。
5. 发帖节奏、审批、复盘、交接统一以 `docs/03-marketing/02-social-media/social-media-operations.md` 为准。

### 4.10 当前跟踪维度

1. 创始人账号是否稳定承担实验叙事与阶段判断输出。
2. 官方账号是否完成案例沉淀、产品更新与官网导流。
3. 是否有产品达到独立升级阈值，或需要明确暂停。
4. 是否完成必要的授权留痕、恢复信息维护与风险兜底。
5. 是否产出足够清晰的实验复盘，能支持后续案例沉淀或产品决策。
