## 1. 定位收敛：从"对外 SaaS 平台"到"内部增长操作系统"

### 1.1 三个收窄（要求 1 / 2 / 3）

**收窄一 — 定位**：GenGrowth 不是对外卖的产品，是**团队自用的内部增长操作系统**。允许流程中"不好自动化、标准难定"的环节用人工替代，以换取一周可落地。

**收窄二 — 服务对象**：MVP 只服务一类产品，第一个实验产品是 **astrologywiki.com**。适用边界直接沿用关键词研究 SOP 的适用场景：

| 维度   | 适用范围                                         |
| ---- | -------------------------------------------- |
| 业务类型 | 内容站 / 工具站 / 早期 SaaS（信息产品 / 自助型工具）            |
| 站点阶段 | 新站到中等权重（DR 0–40），60–90 天看初步效果                |
| 目标地区 | 英文谷歌生态（Ahrefs / DataForSEO / GSC 工具链）        |
| 获客方式 | SEO 为主 + 社媒"双轨"协同（Track A 内容 + Track B 社交信号） |

不在此范围的产品（电商 / 本地 SEO / 中文站 / 老站刷新 / 付费投放主导），MVP 阶段不接。

**收窄三 — 最简原则**：对增长作用不明显又耗时的，不做；不能闭环、标准难定的，不做；第一阶段不做 agent、不做自动编排器。

### 1.2 北极星目标与起点

- **北极星**：astrologywiki.com 日 PV 5000（美国地区为主）。这是 **6–12 个月**的目标，不是单个 60 天周期能到的（占星属高 DR 品类，SEO 起效 3–6 个月）。
- **MVP 首个 60 天周期的目标**：跑通"关键词→集群→内容→收录→排名→测量"全闭环，并让领先指标达标（见第 8 节）。
- **起点**：astrologywiki 的关键词研究与 1+N 集群化已做过第一版（`GenGrowth内容生产执行表_1+N聚类版`，2026-05-14）。GSC / GA4 已接入，测量闭环第一天即成立。

### 1.3 MVP 明确不做（边界注明）

| 不做的事 | 原因 | 何时再考虑 |
|---|---|---|
| Agent / 自动执行 / 周循环编排器 | 不能闭环、风险不可控 | 流程经 2–3 个产品验证后 |
| 框架 ch0 的 JSON Schema 注册中心 / 多租户 / region resolver / LLM 防火墙 / RBAC / 状态机运行时 | 内部单产品场景用不到 | 转为对外多租户产品时 |
| 框架 ch9 的 sunset / 资产墓地 / 归档状态机 | 新站没有可下线的对象 | 站点有 100+ 成熟页面后 |
| 框架 ch7 的 CRO 实验体系（落地页 / CTA / 定价实验） | astrologywiki 的"转化"= PV，无复杂转化漏斗 | 产品引入付费/注册转化目标后 |
| 关键词自动语义聚类 | 标准难定，人工 + 脚本辅助即可 | 聚类规则被 2–3 个产品验证后 |
| Evidence Graph / Claim 归一化 / VOC 摄取 | 重，对单产品 SEO 增长边际价值低 | 不在 MVP 路线内 |

## 4. 三处口径冲突的统一方案

### 4.1 意图体系统一（解 5.1 冲突 2）

**根因**：关键词 SOP 的"意图"和 v0.18 的"意图"根本不是同一个东西。

| 体系 | 实际在判断什么 | 用途 | 归属 |
|---|---|---|---|
| 关键词 SOP 4 类<br>Commercial / Transactional / Problem-aware / Informational | **用户意图** | 决定分桶、商业价值、排期 | 关键词主表 M 列（不变） |
| v0.18 6 类<br>Info / Compare / Tutorial / Utility / Experience / BOFU | **SERP 形态**（Top10 长什么样） | 决定用哪个内容模板 | 内容资产表 G 列，**改名"SERP 形态"** |

**裁决**：两套都保留，各管一段。意图管"值不值得做"，SERP 形态管"怎么做"。给一张交叉参考表（立项时用）：

| 用户意图（关键词层） | 常见 SERP 形态 | 内容模板 Template |
|---|---|---|
| Informational | Info / Tutorial | Definition / Tutorial |
| Commercial | Compare | Comparison |
| Transactional | BOFU | Programmatic |
| Problem-aware | Experience / Tutorial | Case Study / Tutorial |
| 任意意图 + SERP 是计算器/工具 | Utility | Programmatic（工具页） |

> v0.18 文档需把"Search Intent 判定"一节标题改为"SERP 形态判定"，并删掉它自己那套与关键词主表无关的意图词典。

### 4.2 桶 × 页面类型 → Tier 映射矩阵（解 5.1 冲突 3）

**根因**：四桶是关键词层（决定排期优先级），Tier 是文章层（决定单篇投入成本），两个正交的轴，原来没连起来。

**裁决**：用一张矩阵把它们焊死。立项时查表填内容资产表 F 列。

| 桶 ＼ 页面类型 | Pillar 页 | Cluster 页 |
|---|---|---|
| 快速胜利 | **T1** | T2 |
| 长尾（矩阵执行） | T2 | **T3** |
| 趋势词 | T2（SERP 弱→升 T1） | T2 |
| 战略词 | **T1**（全周期 ≤ 5 篇） | —（战略词只做 pillar） |

Tier 的含义沿用 v0.18（决定单篇成本）：

- **T1 重装**：必须人工搜证 Reddit，≥ 2 个 Logic 零件，QA 严审。
- **T2 标准**：可用 AI 专家模拟法搜证，标准 QA。
- **T3 占位**：不搜证，AI 直接组装，检查开头即发布，≤ 10 分钟/篇。

### 4.3 关键词 SOP ↔ 产品定位卡 的接口（解你的已知问题 1）

**根因**：关键词 SOP 与"产品与 ICP 情报"脱节——种子词维度、战略词、TOPIC_KEYWORDS 数组都是凭空配的。

**裁决**：S1 产品定位卡产出 4 个字段，直接喂进 S3 关键词研究的具体输入点：

| 产品定位卡输出 | 喂给 keyword SOP 的哪一步 |
|---|---|
| 种子词维度（3–5 个并列维度） | 来源 3「种子词多维度拓展」的维度输入 |
| 战略词候选（品牌/品类门面词） | 四桶里战略词桶的人工指定来源 |
| TOPIC_KEYWORDS 数组 | 趋势词闸门 G1「话题相关」的配置数组（关键词主表 K 列公式依赖它） |
| 核心命名实体清单 | S5 的 Entity 主权候选 + GEO 命名实体优化 |

> 这一步把"产品理解"真正接进关键词流水线，而不是让 SEO 同事自己猜维度。

## 5. 九步全流程需求

> 每步统一格式：**目标 / 输入 / 输出 / 载体 / 验收标准 / 引用文档**。

### S1 产品定位卡（原 ch1）

- **目标**：用 1 页定位卡替代框架 ch1 的证据图 + ICP 卡包，给后续流程一个明确起点。
- **输入**：产品 URL、亲自使用产品 5 分钟、GenGrowth 已有产品档案草稿。
- **输出**：一页《产品定位卡》，固定 8 个字段：

  1. 产品类别 + 相邻类别
  2. 目标用户（一句话，最可能带来流量/留存的人群）
  3. 核心 JTBD（用户来这个站要完成的任务，2–3 条）
  4. 差异化（为什么用户选它不选竞品，1 条）
  5. **种子词维度**（3–5 个并列维度，喂 S3）
  6. **战略词候选**（≤ 5 个门面词，喂 S3）
  7. **TOPIC_KEYWORDS 数组**（话题相关性闸门用，喂 S3）
  8. **核心命名实体清单**（喂 S5 Entity 主权 + GEO）
- **载体**：人工 SOP，创始人参与（属于框架明确"不产品化"的业务判断项）。
- **验收标准**：8 个字段全部填写；字段 5–8 可直接粘进关键词研究流程，无需 SEO 同事二次猜测。
- **引用**：执行步骤见 `day0-diagnosis-sop.md` Step 0。

### S2 竞品与市场情报（原 ch2）

- **目标**：找到可对标的竞品和可切入的 SERP 薄弱点。不做框架 ch2 的产品层市场全景重活。
- **输入**：产品定位卡、2–3 个竞品 URL。
- **输出**：竞品基线快照（DR / 引用域名 / 估算流量 / 对标用途）+ SERP 薄弱点判断。
- **载体**：人工 + Ahrefs。
- **验收标准**：每个竞品标注"关键词对标（DR 差距 ≤ 30）/ 话题发现（DR 差距 > 30）"；若全部竞品 DR 差距 > 30，已按 SOP 零节走"整个品类 DR 都高"的备用路径。
- **引用**：`day0-diagnosis-sop.md` Step 0 / Step 2.3 / Step 3；竞品选择策略见 `keyword-research-sop.md` 零节。

### S3 关键词研究与主题集群化（原 ch3）— 核心步骤

#### S3a 关键词研究（六源四桶）

- **目标**：产出已分桶的关键词主表。
- **输出**：关键词主表（300–500 词，A–X 列 + Y 列 cluster_id 待填）。
- **载体**：**系统**——Sheets 模板（`keyword-sheet-setup.gs`）+ DataForSEO 批量拉数脚本（自动填 C–I 列：搜索量/KD/CPC/SERP DR）。六源采集、社区挖掘、趋势判断仍人工。
- **验收标准**：四桶分桶完成；快速胜利桶 ≥ 10 词（不足触发红灯，见 Day 0 SOP Step 5）。

#### S3b 主题集群化（新增 —— 解你的已知问题 2）

> **这是补掉"一词一篇导致内容散乱"的关键步骤。** 它坐在"关键词分桶"和"单篇内容产出"之间。

- **目标**：把分桶后的散词，聚成"1 主词 + N 关联词"的集群，让搜索引擎、AI、用户都能看懂站点在做什么。
- **集群化规则（5 步，可执行）**：

  1. **定主题域**：从产品定位卡的 TOPIC_KEYWORDS + 关键词主表高频词根，划 5–15 个主题域。每个主题域 = 1 个集群。
  2. **每个集群定 pillar**：选该域内搜索量最大、意图最泛的词做 pillar 主词。
  3. **归集**：关键词主表里属于该域的词，全部回填 cluster_id（Y 列）。
  4. **集群内切文章（1+N）**：把集群内的词再按"同一篇能否自然覆盖"分组，每组 = 1 篇文章。**合并/拆分判断标准**：
     - **合并为一篇**：关联词是主词的同义/复数/疑问变体，意图相同，抽 2–3 个关联词查 Google SERP，与主词 Top10 重叠 ≥ 4/10。
     - **拆成集群内多篇**：意图不同（计算器 vs 对比 vs 定义）、页面类型不同、SERP 重叠 < 4/10。
  5. **Entity 主权**：集群定 1 个 pillar 级主权实体，每篇 cluster 文定 1 个细分实体，集群内不互相内耗。
- **输出**：集群表填好；关键词主表 Y 列 cluster_id 回填完成。
- **载体**：**人工 + 脚本辅助**。脚本只做"按词根/语义相似度给归并建议"；最终归集、切文章人工定。自动语义聚类 defer（标准未验证）。
- **验收标准**：≥ 80% 的快速胜利词与长尾词已归入某个集群；每个集群有明确 pillar 和主权实体；无两个集群争夺同一主权实体。
- **范例**：`GenGrowth内容生产执行表_1+N聚类版`（astrologywiki 已做过第一版，沿用其格式）。

## 6. 关键词为核心 → 落地为"集群主轴"（回答你的第 6 点）

**结论**：内容生产、分发、外联**应该**围绕关键词关联展开，不应相互割裂。但绑定的单元是**主题集群（1+N）**，不是单个关键词。

数据流——`cluster_id` 是贯穿全流程的那根脊柱：

```
关键词主表（需求库）
   │  S3b 集群化，回填 cluster_id
   ▼
集群表（内容/排名/分发的单元）
   │
   ├─▶ S5 内容生产：集群蓝图 → pillar + N 篇 cluster 文（每篇内容资产表带 cluster_id）
   │
   ├─▶ S5/S9 内链：cluster 文 ⇄ pillar 在同 cluster_id 内互链
   │
   ├─▶ S6 外链：外链目标页 = 集群 pillar；Social 信号回流标 cluster_id
   │
   └─▶ S7/S9 测量与刷新：排名、流量、刷新队列都按 cluster_id 聚合
```

谁也不孤立：一个集群 = 一批词 + 一组互链的文章 + 一个外链目标 + 一组可聚合的指标。这就是"以关键词为核心做拓展和关联"的具体落法。
