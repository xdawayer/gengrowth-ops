# GenGrowth整体营销框架

文档版本：`v1.2`  
更新日期：`2026-04-08`  
状态：`与PRD双轨模型对齐（AI跑实验，人发总结）`

## 一、营销目标

1、建立公司的品牌信任度，获取行业口碑、投资人关注度；

2、为GenGrowth积累个人开发者的用户，形成公司的长期资产；

3、为C端的星图产品积累C端用户，带来C端产品的收入増长

4、通过C端的星图产品的营销实验，为GenGrowth沉淀营销増长的能力以及产品框架

> **附：营销背景prompt**
>
> 公司现在有两个产品：
>
> 产品1：针对个人开发者做的营销増长工具GenGrowth，主要由创始人负责，从头开始学习，无代码开发，并利用产品经理研发的C端产品做"自增长实验"逐步产品化；
>
> 产品2：针对C端用户开发的星座产品，主要由产品经理负责。
>
> 以表格方式输出整体的内容营销框架，明确创始人账号、GenGrowth 官方账号及内部执行角色的分工、目标、社媒营销平台优先级、账号定位、账号名称、发布内容框架，并说明复用性。

## 二、核心营销思路

**GenGrowth整体营销，围绕 "一个核心（Transparency as Marketing，透明度即营销），两套外部账号体系（创始人、官网），一个内部执行角色（PM/增长）"，多平台适配（X、Linkedin、微信公众号、Indiehackers、Reddit、ProductHunt等），打造"以内容建立信任、为产品冷启动蓄力"的跨平台内容营销**

> 在产品未上线（Pre-Product）阶段，**信任 > 品牌**。人们更容易关注一个"正在解决难题的活人"，而不是一个"还没产品的公司"。

**核心定位 (The Hook)：Transparency as Marketing（透明度即营销）——前期我们最大的资产：构建产品的过程**

**补充定位：Transparency as Attribution（归因即透明）——前期最有说服力的内容，不是“我们做了什么”，而是“系统为什么这样决策，以及结果如何”。**

对外内容要从“结果展示”升级为“决策过程展示”：

- 展示假设来源：为什么先做社媒验证，再做SEO固盘
- 展示归因证据：哪条内容、哪个渠道、哪类意图带来有效转化
- 展示系统切换：当SEO见效慢时，系统如何自动升高社媒策略权重并复盘

GenGrowth不仅仅是一个工具公司，更是一个 "独立开发者成功系统"的构建者和先驱实践者。内容旨在分享过程、教育市场、建立权威并聚集社区。

> **对美国市场：** 强调 **"Build in Public" (公开构建)**。西方开发者社区非常吃这一套，他们喜欢看着一个产品从0到1长出来。
>
> **对国内市场：** 强调 **"出海实战/降维打击"**。展示你作为成熟CEO如何用专业商业逻辑去降维做独立开发工具，满足国内开发者对"出海掘金"的渴望。

## 三、内容策略：两套外部账号体系 + 一个内部执行角色

所有公开内容统一围绕两套账号体系展开：**创始人账号负责实验叙事与方法论，官方账号负责案例沉淀与品牌承接**。  
PM / 增长执行仍然是关键角色，但默认不再单独维护一套 X 公开人格，而是把实验执行、截图、归因与复盘素材持续输送给创始人账号和官方账号。

**协同复用性：** 保持 **"创始人发起 -> PM/增长执行验证（产出实验数据）-> 创始人公开解释 -> 官方沉淀案例"** 的高效流转闭环。

### 3.1 双账号结构下的实验公开归属

1. C 端产品的**真实获客动作**继续发生在对应产品渠道，例如 Reddit、社区、App Store、SEO 页面。
2. C 端实验的**X 上公开叙事**默认放在创始人账号，包括假设、动作、失败复盘、关键数据截图与决策变化。
3. 官方号不承担高频实验日报，只负责精选二次分发：案例、方法论、产品更新、Blog、Waitlist CTA。
4. 只有当某个 C 端产品连续 `3-4` 周形成独立受众和稳定转化时，才考虑升级为独立账号资产。

### 3.2 人机分工与发布权限（与PRD对齐，优先级高于上表）

1. Agent 负责“生产与测试”：生成测试内容、变体、发布建议与归因打点；在已授权账号和安全闸门下执行测试贴。
2. PM / 增长负责“执行与解释”：基于 Agent 输出的 `MeasurementReview` 和失败复盘，整理结构化素材与阶段总结，不强制要求持有独立 X 账号。
3. 创始人负责“战略提炼与公开发布”：将 PM / 增长的实验结果上升为方法论与对外叙事，服务募资与品牌信任。
4. 官网负责“长期沉淀”：把“假设 -> 实验 -> 归因 -> 决策”链路整理为可检索案例与销售物料。

### 3.3 自增长实验的新定义（Transparency as Attribution）

自增长实验不再等同于“晒结果”，而是标准化输出以下四段：

1. Hypothesis：本轮假设与目标指标（为什么做）。
2. Action：Agent 执行了哪些动作（做了什么）。
3. Attribution：渠道/内容/页面/转化的归因结果（发生了什么）。
4. Decision：下一轮是扩量、迭代还是暂停（接下来做什么）。

内容运营必须**高复用、低摩擦**

内容复用逻辑：一篇关于"独立开发者获客成本"的深度分析，可以拆解为：创始人视角的行业观察（LinkedIn长文）、产品经理视角的实战数据图表（Twitter线程）、一个具体的踩坑故事（公众号/博客），以及一个总结观点的金句短视频（TikTok/视频号）。

## 四、关键成功要素总结

1. **星座App必须是"真做"：** PM必须不仅为了测试工具，更要为了星座App的收入负责。**只有星座App真的赚到钱了，GenGrowth的故事才性感。"自增长实验"的成败数据必须真实透明，哪怕失败，真诚的复盘比成功的炫耀更具信任价值。**

2. **创始人必须"真学"：** 即使是 No-code，创始人展示的学习曲线越真实，越能激励目标用户（同样想做产品的超级个体）。

3. **协作必须"有戏"，但不强依赖多账号：** 不再要求创始人和 PM 在 X 上长期维持双簧账号。更有效的做法是让创始人持续公开"我让团队试了什么、为什么失败/成功、下一步怎么改"，把内部协作转成对外可消费的内容。

4. **资产沉淀**：所有活动的最终目标，都必须指向为**官网邮件列表**增加一个高质量的订阅者。

5. **归因透明必须可复用**：每次公开复盘必须包含“决策依据 + 归因截图 + 下一步动作”，允许公开失败，不允许跳过证据链。

6. **长期壁垒不是占有原始数据，而是沉淀学习资产**：`GenGrowth` 真正应持续积累的是匿名化的实验结论、主题命中规律、CTA 与页面转化经验、可复用 schema、模板和优先级规则，而不是把客户或样本产品的原始数据当成资产本身。

当前有效的执行说明只有以下几条：
1. PM / 增长不承担“额外维护独立 X 账号”的职责，实验素材优先服务创始人账号与官方账号。
2. 创始人账号负责“方法论 + 核心实验叙事”，不承担流水账式日报，只发值得传播和解释的实验节点。
3. 官方账号负责案例、Blog、产品更新与官网承接，不做高频实验日报。
4. C 端实验的真实获客继续在 Reddit、社区、SEO 页面、App Store 等对应渠道执行。
5. 所有对外“成功案例”必须包含证据链，至少能说明假设、动作、归因与下一步决策。
6. 只有在产品形成独立受众、稳定转化并有明确 Owner 后，才升级为独立产品账号。
