# 增长团队角色与分工（单一事实源 / SSOT）

> **用途**：增长团队"谁负责 S1–S9 哪一段"的**单一事实源**（SSOT = Single Source of Truth，软件/数据领域通用叫法：某件事只在一处写权威版，其他地方引用而不复制）。各人岗位职责文档、绩效文档、各 SOP 在涉及分工时**一律引用本文，不复制**，避免双源漂移。
> **状态**：draft，三角色到齐并跑过一个周期后转 stable。
> **关联**：`2026-05-15-gengrowth-internal-growth-mvp-prd-v0.8`（S1–S9 流程定义）、`GenGrowth整体营销框架`（元层/对象层）、`social-media-operations`（账号分层）、`SEO 运营绩效评估体系`（绩效框架）。

## 一、三个执行角色（一句话）

| 角色 | 在岗 | 一句话职责 | 核心对象 |
|---|---|---|---|
| **SEO 运营**（马博洋） | 1 人（在岗） | 需求侧到内容侧：定位卡 → 关键词 → 集群 → 内容生产（S1–S5），并**全权负责外链（标的 + 联系 + 内容 + 把控）** | 见下方统一对象 |
| **AI 内容运营**（彭满） | 在岗 | 把内容通过自有社媒/社区扩散成真实增长渠道（S6 自有线），跑 Social Probe，沉淀社媒 SOP | 见下方统一对象 |
| **KOL 运营** | **待补** | 对接外部 KOL/KOC，由对方之口发布；管关系、谈判、确保发布与追踪（S6 借势线） | 见下方统一对象 |

> **统一核心对象（三角色相同）**：早期以 **astrologywiki C 端**为主 → 后续扩展到**合作发行的 C 端产品** + **GenGrowth 官网（B 端）**。三人都围绕这组对象工作，不各自割裂。

> 说明：**创始人**不在上表的"执行角色"内。创始人负责 S1 业务判断的参与、对外审批，以及**元层 build in public**（创始人账号 + 官方账号、透明度即营销）——把三人在对象层产出的真实实验上升为对外叙事。元层不归本表三角色维护。

## 三、S6 三条子渠道细分（三角色唯一重叠区，重点）

S6「分发与外联」长出三条并列子渠道，按"自有 / 服务 SEO / 借势"区分归属：

| 子渠道                              | 性质         | 策略归属                                          | 执行归属                                                       |
| -------------------------------- | ---------- | --------------------------------------------- | ---------------------------------------------------------- |
| **自有社媒分发 + Social Probe / 社区观察** | 我们自己的账号发声  | 彭满                                            | 彭满                                                         |
| **外链**                           | 内容为 SEO 服务 | **SEO 马博洋**（标的页、锚文本、质量门槛）                     | **SEO 马博洋**（联系、PR、换链内容均自己把控）；彭满在社区里发现的外链机会作为 lead 转交，不接手执行 |
| **KOL / KOC 外部合作发布**             | 借第三方红人之口   | **KOL 运营**（红人寻源、谈判、合作策略）；带链时标的/锚文本由 **SEO** 定 | **KOL 运营**（管关系、确保发布、追踪）；**彭满**供品牌信息/卖点/素材包/红线              |

**三个防撞车原则**：

1. **自有 vs 借势**：彭满管"我们自己发"，KOL 运营管"别人帮我们发"，不重叠。
2. **外链整条归 SEO**：外链需要端到端 SEO 判断（标的+锚文本+联系+内容+把控），现由马博洋全权负责；彭满/KOL 不动主表，只在各自渠道发现外链机会时回流给马博洋。
3. **内容供给**：KOL 内容由红人按自己口吻创作，内容运营只供品牌卖点/素材/红线，不代写。

**四个交界接口（谁交什么给谁，全部挂 cluster_id）**：

1. **SEO → 内容（内容供给）**：SEO 每产出一篇集群内容，内容运营负责分发放大到自有社媒；分发目标页 = 该集群 pillar。
2. **内容 → SEO（信号回流）**：内容运营把高共鸣话题 / 高互动选题整理成关键词候选回流 S3，SEO 决定是否纳入关键词主表。
3. **内容 → KOL（素材供给）**：内容运营供品牌信息 / 卖点 / 素材包 / 红线；红人按自己口吻创作，内容运营不代写。
4. **三方共享测量**：三人共用一张内容资产表，按 cluster_id 聚合排名 / 流量 / 社媒回流 / 合作来源，S7 复盘一起做。

> 一句话边界：**SEO 决定"做什么内容、外链指哪"，内容运营决定"自有内容怎么扩散、社媒侧该反过来做什么"，KOL 运营管"借谁的口发"，测量闭环三人共享。** 内容生产、关键词主表只归 SEO 写；内容 / KOL 通过"信号回流"影响，不直接动主表。

## 四、绩效归属对应（避免协作产物扯皮）

S6 是多人协作产物，效果好坏的账分开记：

| 产物 | 计入谁的绩效 | 理由 |

| 外链**产出量/质量** + 对**排名的贡献**（DR、目标词排名） | SEO 马博洋（目标达成 + 日常工作） | 外链整条（联系/内容/把控）现由马博洋负责，端到端归他 |

| KOL 带链对排名的贡献 | SEO（目标达成） | 同外链策略层 |

## 六、可衡量指标、数据来源与三人拆分

> **三人指标能分开算的前提 = 两把钥匙**：每个分发动作都带 `cluster_id`（挂哪个集群）+ `UTM`（哪条渠道、谁发的）。GA4 按 UTM 自动拆渠道，共享 Sheet 按 cluster_id 聚合，互不串。

### 6.1 指标 × 数据来源对照

| 指标                        | 归属     | 数据来源                                     | 怎么取 / 拆分键                                                                         |
| ------------------------- | ------ | ---------------------------------------- | --------------------------------------------------------------------------------- |
| **自然搜索 PV**、收录率、关键词排名     | 马博洋    | GSC + GA4                                | 现成后台；GA4 channel=Organic Search                                                   |
| DR、引用域名、外链产出量/质量          | 马博洋    | Ahrefs + **外链台账（待建）**                    | Ahrefs 自动 + 手动台账记每条链接相关性/质量                                                       |
| **社媒可归因引流**（可拆到平台 + 帖子 + 落地文章） | 彭满     | GA4                                      | 链接带 4 段 UTM：`source`=平台、`medium=social`、`campaign=cluster_id`、`content=action_id（帖子）`；落地文章看 GA4 着陆页（见 6.2） |
| 社媒资产（粉丝 / 曝光 / 互动率）       | 彭满     | 各平台原生分析（X / Reddit / TikTok / Pinterest） | 平台后台，每周手动汇总进周报                                                                    |
| Social Probe 信号回流命中       | 彭满     | **共享 Sheet 回流清单列**                       | 手动记"提了几个 → 马博洋采纳几个"，周例会核                                                          |
| KOL 曝光                    | KOL 运营 | 红人提供截图 + 平台                              | 手动进 KOL 台账                                                                        |
| **KOL 引流 / 转化**           | KOL 运营 | GA4                                      | KOL 链接带专属 `utm_medium=kol`、`utm_campaign=kol-{红人}-{cluster_id}` → 关联 CTA / 工具使用事件 |
| KOL 合作数量 / 质量             | KOL 运营 | **KOL 合作台账（待建）**                         | 手动记合作数 + 质量评分（受众匹配/交付质量）                                                          |

### 6.2 三人怎么同步、不打架

1. **唯一数据脊柱 = 那张共享 Google Sheet**（选题登记 / 内容资产表）。三人所有动作挂 cluster_id，写进同一张表，不各存一份。
2. **一个 GA4 属性 + 一套 UTM 命名约定（必须先定死，4 个参数缺一不可）**：
   - `utm_source` = **具体平台**（reddit / pinterest / x / tiktok；KOL 用 `kol-{红人}`）→ 拆到"哪个平台 / 哪个红人"
   - `utm_medium` = **渠道类型**（social / backlink / kol / newsletter）→ 三人各自前缀，天然把流量拆开
   - `utm_campaign` = **cluster_id**（如 `CL-AW-05`）→ 拆到"哪个集群"
   - `utm_content` = **`action_id`（这条分发动作/帖子的 id，取自 6-ID 体系）** → 拆到"哪条帖子 / 哪条评论带来的"
   > **注意**：utm_content 填的是**帖子 id，不是文章 id**。"流量落到哪篇文章"由 GA4 的**着陆页（Landing Page）自动记录**——着陆页 URL 就对应那篇文章的 `page_id`，不用再塞进 UTM。
   > 这样 4 段 + 着陆页一起，GA4 能回答"**Reddit 上的哪条帖子（action_id），把人导到了北交点 pillar（page_id），带了多少流量**"。
3. **每周一 S8 例会 + 周报脚本 = 唯一对齐点**：脚本从 GA4 / GSC / Ahrefs + Sheet 拉数，生成一页"分渠道 / 分人"看板。平台原生数据（粉丝/曝光）和手动台账（外链/KOL/信号回流）由各人每周自报进周报，wzb / Lynne 抽查。
