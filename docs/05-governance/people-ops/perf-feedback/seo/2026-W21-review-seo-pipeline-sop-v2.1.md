# SEO 复评 — 内容生产流水线 SOP v2.1

## 总体结论

v2.1 的修订方向是对的：它确实抓住了上一轮评审里最核心的两个问题，即同类相食和机械 Part 拆分，并开始把单人审核产能、负向词、心理安全、T3 信息增益纳入 SOP。

但当前版本还不能直接标为 `final`。原因不是方向错，而是执行层仍有几处会导致落地失败的结构性问题：

1. 列号与字段定义严重冲突，操作员按文档执行会填错表。
2. “彻底删除多余行”会破坏关键词层数据、指标追踪和 6-ID 外键关系。
3. `us_share`、CTA、Day 14/30/60 复盘仍未接回 v0.7 的闭环。
4. T3 审核时长被压到 `< 3 min`，与 v0.7 的 10-20 分钟审核模型冲突。
5. AIO/GEO 的“留悬念”写法有过度模板化风险，可能损害用户体验和 AI 引用质量。

建议状态：`status: review` 或 `status: revise-before-final`。修完本文 P0/P1 后可进入执行。

## 二、仍存在的结构性问题

### P0-1｜列号体系冲突，按文档执行会填错表

v2.1 同时出现多套列号：

- 第一部分写 `E列 (Intent)`，但同段又写 `Associated Keywords (E列)`。
- AI 组装参数写 `Target Keyword: {D列词} | Associated Keywords: {E列全量长尾词}`。
- 同一参数块又写 `Intent: {H列}`、`Entity: {H列术语}`、`Tier: {I列}`、`Friction: {I列痛点}`。
- PRD 附录 C 与 `keyword-sheet-setup.gs` v3.0 的实际 schema 是：`Target Keyword=A`、`Associated Keywords=B`、`Intent=E`、`Tier=F`、`Entity=H`、`Friction=I`、`page_role=R`。

影响：这是直接落地阻断。操作员无法知道该按 SOP、PRD 还是 Google Sheet 实际列填写；AI prompt 也会把 Intent、Entity、Tier、Friction 混淆。

建议：v2.1 不再写字母列号，统一写字段名；如必须写列号，以 `.gs` v3.0 生成表为唯一来源，并在 SOP 开头声明“列结构以选题登记表 v2.1 为准”。

### P0-2｜“彻底删除多余行”会破坏关键词层数据

v2.1 要求把 1+N 折叠后的多余关键词行“彻底删除”。这会把关键词层的搜索量、KD、来源、后续 GSC 查询词、合并依据一起删掉。

这与 v0.7 的三层数据模型冲突：关键词主表是一行一关键词，选题登记表是一行一页面。正确做法不是删除关键词，而是在页面层合并：

- 关键词主表保留所有原始关键词。
- 选题登记表只建一个 Page Asset。
- 被合并词进入 `Associated Keywords` 或新增 `merged_keywords` 字段。
- 原关键词可标记 `merged_into_page_id`，而不是删除。

影响：删除行会让后续无法解释“为什么这些词被合并到这页”，也无法从 GSC 查询词回写到页面和集群。

### P1-1｜`us_share` 地区闸门没有进入 SOP

v0.7 明确：北极星目标是美国为主的日 PV 5000，主题集群表必须有 `us_share` 三档标签。v2.1 没有要求建卡前核查 `us_share`，也没有规定低美国占比集群不占 P0 产能。

影响：Vedic / Nakshatra 仍可能被当作普通量产集群推进，造成总 PV 增长但美国 PV 目标不达标。

建议：第二部分 The 1-Person Filter 加一条前置检查：`us_share=低` 的集群不得排 P0；`us_share=中` 需抽查 2-3 个头词国家分布；页面卡必须挂载到已有 `cluster_id`，不能绕过主题集群表。

### P1-2｜发布后复盘仍缺失

v2.1 仍是五步生产流程，止于内链布线。它没有承接 v0.7 §7.9 的 Day 14/30/60 规则，也没有使用结果复盘表。

影响：内容发布后没有收录、排名、点击、PV、美国 PV 的闭环判断，后续无法决定刷新、合并、暂停或扩产。这是 W20 已指出、W21 再次指出的问题，当前仍未修。

建议：新增 STEP 6：发布后复盘与资产处置。最小字段包括 `page_id`、`cluster_id`、URL、Day 14 收录/impressions、Day 30 Top50 词数/clicks、Day 60 PV/us_pv、决策。

### P1-3｜CTA 和转化追踪缺位

v2.1 的生产流程没有要求填写 `CTA` / `cta_id`，也没有页面角色到 CTA 的映射。v0.7 要求每个 Page Asset 至少有一个 CTA，并通过 CTA Map 承接 GA4 事件。

影响：内容可以生产，但不能验证是否把流量导向工具页或 newsletter。对增长系统来说，这是“有流量入口、无转化闭环”。

建议：STEP 1 建卡时加入 `cta_id` 或 CTA Map 选择；STEP 4 生成时把 CTA 作为强制输入；QA 加“CTA 是否存在且与 page_role 匹配”。

### P1-4｜T3 审核 `< 3 min` 与 v0.7 产能模型冲突

v0.7 对 T3 的人工要求是 10-20 分钟，已经是轻审核。v2.1 将审核限时压到 `< 3 min`，同时要求检查 T3 占比、生成表格 FAQ、心理安全、AI 幻觉词、视觉结构、TL;DR、内链。这些动作无法在 3 分钟内稳定完成。

影响：SOP 会诱导执行者跳过真实审核，最终仍回到“AI 批量薄内容”的风险。

建议：把 T3 改为“生成后机械 QA 3 分钟 + 发布前抽样/批量复核，总人工 10-15 分钟”。如果坚持 3 分钟，必须明确只适用于低风险、非心理安全、已套模板、已有 Pillar 的 Support 页。

### P1-5｜AIO/GEO 策略过度模板化

`## TL;DR`、FAQ、表格是合理的 AEO 基础结构，但 v2.1 强制“对 How 留悬念”的写法风险较高。AI 概览更偏好完整、清晰、可引用的答案；刻意留悬念可能让页面显得像点击诱导，也不符合“开头直接回答搜索意图”的 v0.7 验收标准。

建议：改成“首段完整回答 What；How 给出步骤摘要，并在正文展开细节”。不建议把 `This requires 3 crucial steps...` 作为全站强制句式。

### P2-1｜负向词处理位置合理，但“删除整行”措辞过重

负向词系统报警是合理的，精准实体词组也比单字黑名单更好。但 SOP 写“直接删除整行”，容易让执行者在选题登记表或关键词主表里做破坏性操作。

建议：改为“不得进入生产队列；在关键词主表标记 `❌跳过`；如误命中白名单，人工复核后恢复”。只有确认是错误导入的临时行，才允许删除。

### P2-2｜Standalone 规则有进步，但需要独立页判定清单

v2.1 把 Standalone 限制在 10% 以内，方向正确。但“无法归入任何已知集群”仍偏主观。

建议加 3 个判定条件：无同主题伙伴词、SERP 意图独立、无法自然链接到现有 Pillar。否则进入 Backlog 等待合并，不直接建独立页。

## 三、可落地修订清单

| 优先级 | 修订项 | 最小改法 |
|---|---|---|
| P0 | 统一字段与列号 | 删除 SOP 内所有不一致列号，只保留字段名；附一张与 `.gs` v3.0 一致的字段表。 |
| P0 | 改 1+N 合并动作 | “删除多余行”改为“关键词主表保留，页面层折叠；被合并词标记 `merged_into_page_id` 或进入 Associated Keywords”。 |
| P1 | 加 `us_share` 前置闸门 | 建卡前必须确认 `cluster_id` 与 `us_share`；低美国占比不排 P0。 |
| P1 | 加 STEP 6 复盘 | 直接引用 v0.7 §7.9 Day 14/30/60，并写入结果复盘表。 |
| P1 | 加 CTA Map | 每个页面卡必须有 CTA 或 `cta_id`，QA 检查 CTA 与 page_role 是否匹配。 |

| P1 | 修 AIO/GEO 文案 | 保留 TL;DR/FAQ/表格，取消强制悬念句式，改为完整摘要 + 正文展开。 |

## 四、最终判断

v2.1 不是失败稿，而是“方向正确但执行契约未闭合”的版本。它已经解决了 v2.0 最大的 SEO 架构问题：同类相食和机械拆分。但如果直接执行，会在 Google Sheet 列号、数据删除、地区目标、CTA 和复盘闭环上出错。

建议不要重写整份 SOP，只做一次小修订到 v2.2：

1. 先修 P0 字段/数据模型。
2. 再补 `us_share`、CTA、STEP 6。
3. 最后调整 T3 审核与 AIO 句式。

完成后，v2.2 可以作为 astrologywiki 的正式内容生产 SOP 使用。
