# SEO 运营输出质量评审标尺（Rubric）v0.1

> **用途**：作为 `seo-quality-audit` subagent 和人工抽审的统一标尺，回答"SEO 运营产出的内容好不好"。
> **定位**：绩效评估体系 § 维度 2（日常工作）下"内容质量"子项的可执行落地依据。
> **状态**：draft，第一次完整跑过一个评审周期后转 stable。

---

## 一、评审视角与基调

**视角**：以 SEO 专家 + GenGrowth 战略 + 营销规范 + 岗位 SOP 综合判断，**不是单纯的格式检查**。

**基调（agent 与人工抽审均适用）**：

- **专业、客观、严厉**
- **关键问题前置**，不重要细节适当忽略
- **禁止软话**：不允许"建议考虑 / 可以补充 / 或许更好"这类措辞；改用"缺失 / 不达标 / 必须补充"
- **禁止空评**：不允许"整体良好 / 质量不错"这种笼统结论；必须指出具体哪句、哪段、哪个数据
- **主动找问题**：未发现问题 ≠ 没问题；必须主动扫描潜在风险

## 三、AI 立场与演进路径

### 3.1 立场

**绝对允许且鼓励**用 AI 生成内容——AI 是必备工具，不是问题来源。

**拒绝的是"AI 直出未加工"**。识别信号：

- 结论通用化、未结合本公司业务 / 客户 / 数据
- 大段空泛排比、关键数据无具体来源
- 缺少 SEO 专业判断（关键词意图、CPC=0 词处理、长尾分桶逻辑）
- 英中翻译腔（"在……的情况下"、"对于……而言"、"值得注意的是"密集出现）
- 用词风格与团队 SOP 明显不一致

**判断准则**：评的是**结果质量**，不是是否用了 AI。

- 用 AI 提速 + 人工加工到位 = 合格
- 不用 AI 但内容空泛 = 不合格
- 用 AI 直接搬运未加工 = P0 否决

### 3.2 演进路径（前期 → 长期）

| 阶段 | 上游（SOP） | 下游（审计） |
|------|------------|--------------|
| 前期（v0.1–v0.3） | 持续打磨 SOP | 双保险：每周自动 + 人工抽审 |
| 中期（v0.4+，上游 SOP 稳定后） | 主保险 | 月度抽审 + 异常触发 |
| 长期（SOP 完全内化后） | 单保险 | 仅"随机抽审 + 异常触发"作底线 |

**红线约束**：P0 维度（数据真实 / 战略偏离 / AI 搬运）**永远保留低频抽审**——SOP 是死的、业务是活的，下游审计是发现"SOP 盲区"的唯一信号，不可完全去除。

## 四、评审范围

### 4.1 默认评审的 3 类主要文档

| 类型 | 位置 | 评审单位 |
|------|------|----------|
| 增长实验记录 | `docs/repo/gengrowth-ops/inbox/...`（基于 `seed-client-growth-experiment-template.md`） | 按文档全文 |
| 关键词主表批次 | SEO 运营每次评审前导出 CSV 至 `docs/repo/gengrowth-ops/inbox/keyword-master/{YYYY-WW}.csv` | 按本批次新筛选行 |
| SEO 内容创作 SOP | `docs/repo/gengrowth-ops/inbox/...` 或迁入 SOP 区 | 按文档全文 |

### 4.2 默认不自动评审

- 大量产生的 SEO 文章和过程文档——会噪音淹没信号
- **触发例外**：评审主要文档时，若需追溯到具体文章作证，可**手动触发**单篇文章评审

### 4.3 评审前的确认流程（强制）

Agent 启动评审前必须：

1. 列出"建议评审清单"（哪些文档、哪个时间窗口、对应类型）
2. 等待 Lynne 拍板确认
3. 确认后才开审

> 这一步是为了避免之前"审错对象"的事故——agent 不自行决定审什么。

## 五、评分维度（P0 / P1 / P2 三档）

输出顺序**强制 P0 → P1 → P2**。每个 P0 问题独立段落 + 加粗展示；P1 列必改清单；P2 可合并陈述。

### P0 — 致命问题（一票否决候选）

#### P0.1 数据真实性

| 等级 | 锚点 |
|------|------|
| 优秀 | 所有数据均标注来源（GSC/GA4/Ahrefs/工具截图链接），与原始来源一致 |
| 合格 | 关键数据有来源，次要数据可推断 |
| 待改进 | 出现以下任一即触发：数据无来源；与 GSC/GA4 实测不符；被四舍五入到失去意义（"约 1 万 PV"）；用"业内平均水平"等通用数据替代实测 |

#### P0.2 战略匹配度

| 等级 | 锚点 |
|------|------|
| 优秀 | 明确服务订阅付费业务 + 美巴 Google 生态 + Q2 PV 增长目标 |
| 合格 | 方向正确，部分动作有偏移但可纠 |
| 待改进 | 出现以下任一即触发：选客户/选关键词跑去做百度系；主战场不是美国/巴西；YMYL/本地服务作主选项；关键词主表中订阅付费意图词 < 30%；整体动作偏离 Q2 PV 5k–10k 目标 |

#### P0.3 AI 搬运检测

| 等级 | 锚点 |
|------|------|
| 优秀 | 能看出明显的人工加工痕迹（业务术语、本公司数据、SEO 专业判断） |
| 合格 | AI 生成的骨架 + 充分的人工填充 |
| 待改进 | 出现以下任一即触发：整段通用化结论未结合具体客户/业务；大段排比无具体数据；缺少 SEO 专业判断；密集翻译腔；用词风格与团队 SOP 明显不一致 |

---

### P1 — 结构问题（关键，必改）

#### P1.1 框架完整性

| 等级 | 锚点 |
|------|------|
| 优秀 | 完全按对应 SOP 跑完所有关键步骤 |
| 合格 | 主线步骤齐全，1–2 个次要步骤可补 |
| 待改进 | 缺关键章节、SOP 步骤被跳过、Section I 或核心字段空白 |

#### P1.2 逻辑严密性

| 等级 | 锚点 |
|------|------|
| 优秀 | 数据 → 分析 → 结论 → 下一步动作 链路清晰 |
| 合格 | 链路成立，偶有跳跃但可补 |
| 待改进 | 结论与数据脱节；推理跳跃；"快速胜利词"等判断无依据；归因偷懒（用"算法变化"等不可证的归因兜底） |

---

### P2 — 执行问题（影响落地，需改）

#### P2.1 可行性

- 动作能否落地、资源是否匹配、时序是否合理
- 待改进信号：动作无人负责、资源缺位、时序冲突、依赖未达成的前置条件

#### P2.2 规范合规性

- 模板字段、命名、归档位置
- **由 doc-audit subagent 兜底**，质量评审只简评，不重复细节

## 六、评级映射

| 评级 | 触发条件 |
|------|----------|
| 待改进 | 任一 P0 问题成立 |
| 合格 | 无 P0；存在 P1，必改后通过 |
| 良好 | 无 P0；无显著 P1；仅有 P2 |
| 优秀 | 无 P0；无 P1；P2 极少；且有亮点 |

**配套动作**：

- `待改进` → 触发 1on1 当面谈
- `优秀` → 在月度回顾中点名表扬，作为团队标杆

```markdown

### P0 致命问题（{N} 项）

**P0.1 数据真实性 — {评级}**

- 问题：{具体哪句 / 哪段 / 哪个数据}

---

### P1 结构问题（{N} 项）

- P1.1 框架完整性：{具体问题 + 原文位置}

---

### P2 执行问题（{N} 项）

{可合并陈述，列表呈现}

---

### 改进优先级（按必须性排序）

1. {最高优先级动作，对应某个 P0/P1}
2. {次高优先级动作}
3. ...
```

## 八、触发与频率

| 触发方式 | 谁来发起 | 频率 |
|----------|----------|------|
| 手动触发（常规） | Lynne 在 Claude Code 中说"评估 SEO 输出" | 每周 1 次（前期） |
| 异常触发 | 发现 P0 信号、上游 SOP 重大更新、季度评审前 | 不定期 |
| **不做 cron 自动调度** | —— | —— |

**不进 cron 的理由**：

1. 审计前必须由人确认审什么（防止"审错对象"复发）
2. Agent 在 Claude Code 会话内运行，不放进 launchd
3. 前期由 Lynne 控制节奏，便于观察 ROI 后再决定是否半自动化

## 十、配套机制

### 10.1 关键词主表 CSV 导出约定

- SEO 运营**每次评审前**：把本批次新筛选的关键词导出为 CSV
- 命名：`{YYYY-WW}-keyword-batch.csv`
- 位置：`docs/repo/gengrowth-ops/inbox/keyword-master/`
- 字段必含：keyword / search_volume / cpc / intent / bucket / strategic_fit_note

### 10.2 与 doc-audit 的分工

| Agent | 关注 | 输出 |
|-------|------|------|
| `doc-audit` | 错放 / 重叠 / 命名 / 层级 / README 盲区 | 治理类清单 |
| `seo-quality-audit` | 内容好坏（本 rubric） | 评级 + P0/P1/P2 报告 |

两者**不重叠**——seo-quality-audit 不重复 doc-audit 已经覆盖的规范检查（P2.2 仅简评）。
